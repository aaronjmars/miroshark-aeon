## Self-Improve — blocked-features registry (aeon PR #50)
- **Skill**: self-improve (even-day schedule, `0 13 */2 * *` — last ran 2026-05-30)
- **Trigger**: `repo-actions` has suggested **Operator Profile** 13 times across 2026-05-08 → 2026-06-01 (confirmed via grep over `articles/repo-actions-*.md`). Today's `feature` skill pivoted off it after grep confirmed `SimulationState` has no `operator` / `created_by` field — `platform_stats.py:42-49` upstream documents `project_id` as the closest stable identifier. Without a memory primitive, tomorrow's `repo-actions` run on Jun-3 would suggest it again on the next eligible day (the 7-day exclusion window keeps dropping the same blocked idea back into the eligible pool).
- **Fix**: created `memory/topics/blocked-features.md` as a registry for verified-blocked ideas with documented schema (signature keywords / category / reason / verifying log / suggestion history / unblock condition). Updated `skills/repo-actions/SKILL.md` step 4 to read the registry, do case-insensitive signature-keyword match against each candidate idea, append a one-line `Excluded (blocked): <title>` note to the article's Selection Rationale section, and run a 30-second re-verification on each match so blocks lift automatically when upstream changes land. Bootstrapped with one entry: Operator Profile.
- **Files changed (2 files):**
  - `memory/topics/blocked-features.md` — new file (registry + 1 entry).
  - `skills/repo-actions/SKILL.md` — step 4 appended (read registry → match → exclude → note).
- **Design decisions:**
  - **Memory file, not skill prompt.** The block list grows over time; embedding it in the prompt would couple the data to the skill version and make adds require a skill edit.
  - **Signature keywords, not idea-id matching.** Idea titles drift between runs ("Operator Profile" / "Operator Profile Page" / "Operator Identity Layer"). Substring match on a small keyword set is robust to renaming.
  - **Auto-unblock on each match.** Each entry includes an "Unblock when" condition; the skill re-verifies on the cloned repo before excluding. If the constraint has lifted, the entry is removed in the same run and the idea returns to the eligible pool.
  - **One-line Selection Rationale note.** Operator sees what was filtered without having to crawl the registry file.
- **Validation**: SKILL.md is markdown read by Claude at run time — no script to test. Next `repo-actions` run (Jun-3 14:00 UTC) validates end-to-end.
- **Impact**: frees one idea slot per `repo-actions` run for net-new suggestions; prevents wasted feature-build cycles on perpetually-blocked ideas; creates a memory layer for verified upstream constraints that compounds across runs.
