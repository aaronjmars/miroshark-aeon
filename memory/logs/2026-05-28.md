  - `backend/app/services/webhook_service.py` (+237 LoC) — added token constants + category frozensets, `_resolve_event_filter`, `_payload_direction`, `_payload_confidence_pct`, `_payload_quality_key`, `payload_passes_event_filter(payload, events) -> (bool, trace)`; wired filter into `fire_webhook_for_simulation` between dedup and dispatch.
  - `backend/tests/test_unit_webhook_events.py` (new, 25 offline tests) — parser normalisation, OR-within-category for all three categories, cross-category AND, confidence floors (>=75 / 50–75), quality bucket inclusion, unknown-token handling, failed-sim bypass, end-to-end `fire_webhook_for_simulation` behaviour with filter set/unset.
  - `.env.example` — documented WEBHOOK_EVENTS block alongside WEBHOOK_SECRET (token reference + 3 example settings).
  - `docs/WEBHOOKS.md` — new "Filtering events" section (token table, AND/OR logic, examples); Configuration section references it.
  - `docs/FEATURES.md` — new "Webhook Event Filtering" entry with full semantics + impl notes.
- **Design decisions**:
  - **OR within category, AND across categories** — intuitive for the 10 integrators scenario; `bullish,bearish` = "any directional" not "both at once".
  - **Failed sims bypass the filter** — the alert an operator most needs to see should never be the one a filter swallows.
  - **Unknown tokens silently ignored** — a typo shouldn't disable the webhook (recorded in trace dict for the log).
  - **Manual retries bypass the filter** — same pattern as the existing dedup bypass; operator-driven re-send is explicit.
  - **Late-bound env read** — same `os.environ` pattern as `WEBHOOK_URL` / `WEBHOOK_SECRET`; no restart needed to flip rules.
  - **Suppressed deliveries log info-level but don't write to webhook-log.jsonl** — log is for *attempted* deliveries; filter skip is a different concern.
  - **Direction derivation reuses Discord embed colour rule** (`>=` plurality) — filter matches what the visual surfaces show.
- **Validation**: pytest is blocked in sandbox (`python` not allowlisted — known limitation, logged in past skill notes). CI authoritative. Static review confirmed import paths + logic.

## Self-Improve — token-report Grok query spam filter
- **Skill:** self-improve (even-day schedule, `0 13 */2 * *`)
- **Trigger:** today's token-report log: `Social: ... All 5 Grok results are spam/bot accounts, 0 engagement. No organic signal. Pattern continues from prior days.` Same zero-engagement contract-drop spam pattern that drove aeon PR #47 (disable fetch-tweets + tweet-allocator) on 2026-05-27. token-report still consumes `.xai-cache/token-report-social.json` and pays for the XAI call even when every returned tweet is spam.
- **Why fetch-tweets was disabled but token-report wasn't:** fetch-tweets fed tweet-allocator (which then paid wallets, so spam = real money loss). token-report only uses the social signal as one section of a larger price report — spam degrades the section but doesn't burn money. Filtering at the Grok-query level reclaims that section's value without disabling the call.
- **Fix:** rewrote the token-report Grok prompt in `scripts/prefetch-xai.sh` (line 187) to apply three explicit pre-filters before Grok picks results — (1) drop zero-likes AND zero-retweets tweets (the dominant bot pattern in low-cap cashtag streams); (2) drop contract-drop / "vote for" / "fam drop" / "exclusive drop" templates and known clone domains (arbihunter.live, toknsite.live, toknsite.club, coinmarkettcap.fun, *.live/*.club farms); (3) drop duplicate-template spam (same wording across multiple handles). Tells Grok to return zero results rather than fall back to spam if nothing passes.
- **Files changed (1 file, +1 prompt line replaced):**
  - `scripts/prefetch-xai.sh` (~line 187, in the `token-report)` case) — prompt rewrite with the three explicit filter rules; preserves the symbol-confirmation sidecar pattern and the from/to date range.
- **Design decisions:**
  - **Query-level filter, not post-process filter** — Grok has the full tweet metadata at picking time (likes, RT count, full text, posting cluster). A post-process filter (à la `filter-xai-tweets.py`) sees only what Grok rendered into `output_text` and can only re-validate the cashtag substring. Pushing the rule into the prompt lets Grok consider more candidates and pick better ones.
  - **Stay within the existing skill contract** — `token-report/SKILL.md` Path A still parses the same JSON shape; no skill-side change needed.
  - **Zero is acceptable** — explicit instruction to return zero results rather than fall back to spam. The token-report skill's Path B already handles the empty case cleanly ("X/Grok data wasn't available for this run").
  - **Domain list is illustrative, not exhaustive** — Grok extrapolates from the named domains to other clone-farm patterns. Coding a hard regex of spam domains would rot weekly.
- **Validation:** `scripts/prefetch-xai.sh` is a bash script invoking curl + jq; no python tests to run, and the runtime path only fires inside the workflow (XAI_API_KEY + token-report skill name + memory/MEMORY.md tracked token row). Tomorrow's 06:00 UTC `token-report` run validates end-to-end; if the cache returns zero results after filtering, Path B kicks in.
- **Impact:** higher signal-to-spam ratio in the daily token-report Social Pulse section; on days where only spam exists, the section cleanly degrades to "X/Grok data unavailable" instead of citing scam contract drops as if they were sentiment. Saves nothing on XAI billing (one call/day either way) but reclaims report quality.
