## Agent Self-Improvement — fetch-tweets ID-based dedup
- **Trigger:** Today's `fetch-tweets` log contains 24 unique tweet URLs — 13 named (`x.com/<handle>/status/<id>`) + 11 annotation-only (`x.com/i/status/<id>`). Across the persistent seen-file (`memory/fetch-tweets-seen.txt`, 115 URLs), 54 (47%) are in the `i/status` form. Grok's `output_text` and its `content.annotations[]` can return the same tweet under either URL shape across runs; the current URL-based dedup in `fetch-tweets` step 5 would mis-treat those as distinct tweets and re-notify. No active breakage yet (all 115 seen URLs have unique tweet IDs), but the logic is fragile given the annotation-harvest path (PR #20) now routinely surfaces `/i/status/` URLs.
- **Fix:** `skills/fetch-tweets/SKILL.md` step 1 + step 5 now extract the numeric tweet ID from each URL and dedup on `SEEN_IDS` (the set of IDs) instead of full URLs. Seen-file content is unchanged (still appends full URLs at step 6b) — the ID extraction happens on read. Matches the in-run ID-dedup already in `scripts/filter-xai-tweets.py` (`tweet_ids_in(text)` + `splice_annotations`).
- **Impact:** Prevents future duplicate notifications when Grok returns the same tweet under a different URL shape across the 3-day log window. Defensive fix — also makes the skill's behaviour consistent with `filter-xai-tweets.py`'s own dedup model. Zero new deps, zero file moves.
