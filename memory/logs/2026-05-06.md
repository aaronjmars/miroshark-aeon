## Feature Built — Webhook Delivery Log + Manual Retry

- Built feature for `aaronjmars/MiroShark` — operational visibility layer over PR #46's outbound completion webhook.
- **What:** `<sim_dir>/webhook-log.jsonl` per dispatch attempt + `GET /api/simulation/<id>/webhook-log` (admin-token gated) returning last 10 entries newest-first + `total_attempts` counter + `POST /api/simulation/<id>/webhook-retry` (admin-token gated) re-firing the completion webhook for finished sims.
- **Why:** PR #46 shipped the webhook but every Zapier / n8n / Slack / Discord integration built on top has been flying blind on delivery status — a 5xx from the downstream endpoint disappeared into server logs. This is the operational closure that any production webhook eventually needs.
- **Pivot:** Picked from May 4 repo-actions idea #2 over #1 Embeddable Live Belief Widget (autonomous-risky: would conflict with existing SPA `/embed/:id` route serving `EmbedView.vue`), #4 Private Share Link (the writeup framed it as resolving issue #70 but #70 is actually a much larger Private Impact mode collaboration request from Cyril), #5 Simulation Tagging (editorial layer with less operational leverage). #2 was the cleanest small-effort autonomous pick.
- **Files:**
  - `backend/app/services/webhook_service.py` — log helpers (`webhook_log_path`, `_read_log_lines`, `_parse_log_entries`, `_next_attempt_number`, `_append_log_entry`, `_parse_status_code_from_message`, `read_webhook_log`, `_record_delivery`) + `_start_dispatch_thread` shared between auto-fire and retry + `retry_webhook_for_simulation` + extended `_send` body capturing latency and writing the log
  - `backend/app/api/simulation.py` — `GET /<id>/webhook-log` + `POST /<id>/webhook-retry` route handlers, both admin-token gated
  - `backend/openapi.yaml` — both new paths under Publish & Embed + `WebhookDeliveryEntry` and `WebhookDeliveryLog` schemas (drift-detection test passes)
  - `backend/tests/test_unit_webhook_log.py` — 13 offline unit tests
  - `frontend/src/api/simulation.js` — `getWebhookLog` + `retryWebhookDelivery` helpers
  - `frontend/src/components/EmbedDialog.vue` — "📡 Webhook delivery history" panel (admin-token gated, collapsed by default), status chips (✓ green / ✗ red / ⏱ amber) per delivery with HTTP code, latency, trigger label, timestamp; Refresh + Retry delivery buttons
  - README + docs/FEATURES.md + docs/API.md + docs/WEBHOOKS.md (en + zh-CN mirrors) — feature row + full docs section + delivery-semantics callouts
- **Behaviour:**
  - 50-line on-disk cap via atomic read-modify-rename (`os.replace`) — log can never corrupt or grow unbounded
  - URL masked (`scheme://host/***`) before any disk write — Slack / Discord secret never round-trips through the read endpoint
  - Status code parsed from existing `_post_json` 2-tuple message format (`HTTP <N>` → integer, network errors → `null`) so test surface for original webhook stays untouched
  - Retry bypasses per-process `(sim_id, status)` dedup gate (auto-fire dedup exists only to prevent runner's two terminal code paths from double-firing automatically; an explicit retry should always go through). Replay payload carries `retry: true` so downstream consumers can dedupe
  - Returns 400 when no webhook URL configured, 409 when sim not in terminal state
  - `_record_delivery` happens after `_post_json` returns, so dispatch path stays fire-and-forget — log write never blocks the simulation runner
