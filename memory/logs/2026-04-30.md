### Endpoints added
- `GET /api/feed.atom` — Atom 1.0 syndication feed of the public simulation gallery (default; the format browsers auto-discover via `<link rel="alternate">` and the format modern readers prefer).
- `GET /api/feed.rss` — RSS 2.0 variant for older readers / self-hosted aggregators that haven't moved off RSS.
- Both accept `?verified=1` to restrict to simulations with a recorded outcome annotation (the syndication mirror of `/verified`).
- Both: `Cache-Control: public, max-age=300` (5-minute CDN-friendly cache — short enough for newly published sims to appear in next aggregator poll, long enough to absorb aggressive polling without taxing the gallery query).

### Files
- **NEW** `backend/app/services/feed.py` (584 lines) — Atom 1.0 + RSS 2.0 renderer + `select_public_cards()` selection helper. Pure stdlib (`xml.etree.ElementTree` + `html`). Per-entry payload: scenario as title (truncated to 100 chars), bullish/neutral/bearish consensus split as the plain summary, share-card PNG as `<media:thumbnail>` + `<media:content>` (PNG), replay GIF as `<media:content>` (GIF), outcome + quality as `<category>`. RSS variant adds `<enclosure>` for the share-card PNG and `<guid isPermaLink="false">` for stable identification across deployments. Same ±0.2 stance threshold as gallery / share card / replay GIF / transcript / webhook.
- **NEW** `backend/app/api/feed.py` (144 lines) — `feed_bp` blueprint mounted at `/api`. `_resolve_base_url()` honors `Config.PUBLIC_BASE_URL` first (canonical operator-supplied URL — same field webhook + share-card use), falls back to `request.host_url` with `X-Forwarded-Proto` / `X-Forwarded-Host` for reverse-proxy deployments. `_serve_feed()` shared body for both routes — local-imports `_build_gallery_card_payload` + `_read_outcome_file` from `simulation.py` so the selection stays single-sourced with `GET /api/simulation/public`. Fail-soft against `SimulationManager` errors — an empty feed is the right interim state.

- `backend/tests/test_unit_openapi.py` — `feed_bp: /api` added to `_BLUEPRINT_PREFIXES` so drift-detection test passes.

## Self-Improve — Heartbeat day-of-week accuracy

- **Trigger:** Apr 29 heartbeat report opened with `Date: Tuesday Apr 29, 2026 — ~19:40 UTC`. Apr 29 2026 is **Wednesday**. Because the LLM hallucinated the weekday from `${today}`, the report listed `memory-flush (Sun+Wed)` under "Not scheduled today (Tuesday)" and re-classified its on-schedule 18:25 UTC run as "ran off-schedule". Apr 25/26/27/28 were correct; Apr 29 is the first observed slip but the failure mode is hallucination from a YYYY-MM-DD date and could recur any day. Would mask a real "memory-flush failed to fire on Wed" outage if one ever happened.
- **Picked over:** day-of-week noise was the only repeatable quality drift visible in the last 5 days of logs — every other skill had clean output (token-report, fetch-tweets, tweet-allocator, repo-pulse, feature, push-recap, project-lens, repo-article all green). `cron-state.json` shows 100% success across every enabled skill with the lone Apr 17 push-recap failure already addressed in the SKILL.md (sandbox loop guidance at line 20).
- **Fix:** `skills/heartbeat/SKILL.md` adds an explicit **Step 0** section before any schedule check: run `date -u +%A`, `+%u`, `+%d`; anchor the report header on the shell output (`Date: <%A> <Mon DD>, <YYYY> — <HH:MM> UTC`); use shell values for every schedule comparison. Adds an explicit translation note that cron weekday `0=Sun` while `+%u` returns `7=Sun` (silently off-by-one bug). Adds ground-truth guidance for every-other-day cron expressions: when in doubt, check the last 7 days of `cron-state.json` `last_dispatch` history rather than guessing odd/even-day parity.
