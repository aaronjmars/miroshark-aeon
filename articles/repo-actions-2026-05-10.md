### 1. Webhook HMAC Signature Verification

**Type:** Security / Integration
**Effort:** Small (hours)
**Impact:** As integrations multiply — Revault, CancerHawk, n8n, Zapier, custom endpoints — every operator receiving MiroShark webhooks has the same unresolved question: did this payload actually come from my instance, or is something spoofing it? HMAC signature verification (Stripe's model, GitHub's model) closes that question. Every dispatch adds an `X-MiroShark-Signature: sha256=<hex>` header computed from `WEBHOOK_SECRET + payload_bytes`. Operators verify on arrival with two lines of stdlib `hmac`. Backward compatible: when `WEBHOOK_SECRET` is not set, the header is omitted and existing integrations continue working without changes. Zero new deps.
**How:**
1. In `backend/app/services/webhook_service.py`, add a `_sign_payload(payload_bytes: bytes) -> str | None` helper: reads `os.environ.get("WEBHOOK_SECRET", "")`, returns `None` if blank (backward compat), otherwise returns `"sha256=" + hmac.new(secret.encode(), payload_bytes, hashlib.sha256).hexdigest()`. Modify `_send()` to call `_sign_payload(json_bytes)` and, when non-None, inject `X-MiroShark-Signature` into the request headers dict alongside the existing `Content-Type` and `X-MiroShark-Event` headers. The signature is computed once per dispatch; the same bytes land in the header and the body so the recipient can verify. Add 8 new tests to `test_unit_webhook_log.py`: signature header present when `WEBHOOK_SECRET` set, absent when not set, verification round-trip (sign then verify with same secret), tampered body fails verification, tampered header fails verification, empty secret treated as unsigned (no header), signature is `sha256=` prefixed hex string (format guard), retry dispatch carries the same signature header. Extend `docs/WEBHOOKS.md` with a "Verifying webhook signatures" section — Python snippet (3 lines), Node.js snippet (3 lines), cURL/jq snippet for manual testing.
2. Add `WEBHOOK_SECRET` to `openapi.yaml` `WebhookDeliveryEntry` schema description (note that the delivery log stores `url_masked` not the secret, and the signature is transport-only). Add a "Signature verification" callout to the EmbedDialog webhook section beneath the Retry button: when the webhook URL is set and a delivery has succeeded, show a collapsed "🔐 Verify webhook signatures" hint with the `WEBHOOK_SECRET` env var name and a link to `docs/WEBHOOKS.md`. Do NOT show the secret value — just the env var name. Add `WEBHOOK_SECRET` to the environment variable reference in `README.md`.
3. In `backend/app/api/simulation.py`, the `webhook-retry` endpoint already rebuilds the payload via `retry_webhook_for_simulation()`. Ensure the signature is re-computed fresh for each retry attempt (not cached from the auto-fire) — this is already the case if `_sign_payload` reads the env var at call time, but add a test that explicitly dispatches via retry and verifies the header is present and correct. Add `WEBHOOK_SECRET` to the `.env.example` (or equivalent operator configuration reference) with a note: "Set to a random 32+ char string. Compute with `python -c 'import secrets; print(secrets.token_hex(32))'`."

### 2. Jupyter Notebook Export

**Type:** Feature / Research
**Effort:** Small (hours)
**Impact:** The trajectory CSV (PR #66) gave data analysts a clean input. But opening a CSV in Jupyter still requires writing the boilerplate: imports, `pd.read_csv()`, figure setup, belief evolution chart. A `GET /api/simulation/<id>/notebook.ipynb` endpoint returns a pre-populated Jupyter notebook — code cells ready to run, markdown cells framing the simulation context, analysis scaffolded from the reproduce.json metadata. The `.ipynb` format is plain JSON (no new deps). Institutional observers who land on a published simulation — the Lorimer Ventures tier — download this and open it in one step. It's also the format that academic collaboration happens in; researchers who cite MiroShark work in papers need exactly this artifact.
**How:**
1. Add `backend/app/services/notebook_export.py` (~220 LoC, pure stdlib `json` + `os` + `datetime`). The `.ipynb` format is a JSON document with `nbformat: 4`, `cells: [...]`, and `metadata` (kernel info: Python 3). Cell types: `markdown` and `code`. Build the notebook with this cell sequence: (a) markdown header — sim ID, scenario (full text), agent_count, total_rounds, quality_health, lineage_kind, generation timestamp; (b) code cell — `pip install` line for pandas + matplotlib (commented out by default, operator uncomments if needed) + imports; (c) code cell — trajectory CSV loading via `pd.read_csv(io.StringIO(csv_string))` with the raw CSV data embedded as a Python multi-line string (so the notebook runs standalone without network access); (d) code cell — belief evolution line chart (bullish/neutral/bearish pct over rounds, three colored lines, axis labels, title from scenario); (e) code cell — agent stance distribution bar chart (final consensus breakdown); (f) code cell — quality health and participation rate summary `pd.DataFrame`; (g) markdown cell — reproducibility footer with sim ID, reproduce.json URL, `render_json_bytes`-stable SHA-256 of the trajectory CSV. All code cells have `outputs: []` and `execution_count: null` (standard unexecuted notebook). `render_notebook_bytes(sim_dir, sim_id, base_url)` produces the full JSON bytes. Add `GET /api/simulation/<id>/notebook.ipynb` route (publish-gated, `Cache-Control: public, max-age=300`, `Content-Disposition: attachment; filename="miroshark-<id12>.ipynb"`). Add 10 offline unit tests in `test_unit_notebook.py`: nbformat=4 guard, cells list non-empty, markdown header contains sim_id and scenario, code cells are syntactically importable (no obvious truncation), CSV data embedded in notebook (trajectory round count matches), chart cells reference `plt.show()`, reproducibility footer contains sim_id, route-decorator presence, notebook is valid JSON (`json.loads` round-trip), agent count appears in markdown header.
2. Extend `frontend/src/api/simulation.js` with `getNotebookUrl(simId)`. In `EmbedDialog.vue`, add a "📓 Jupyter notebook" row beneath the Reproducibility Config section: `Download notebook.ipynb` button (same `<a download>` pattern as the reproduce.json download), a note "Opens in JupyterLab, VS Code, or Google Colab — trajectory data and charts pre-loaded", and the curl snippet `curl -fsSL '<url>' -o simulation.ipynb`. Publish-gated with the standard 403 empty state. Add "Jupyter Notebook Export" row to `docs/FEATURES.md` under Publish & Embed. Add the endpoint to `docs/API.md` with a note on the embedded-CSV approach (no network dependency, runs air-gapped).
3. Add `/api/simulation/{simulation_id}/notebook.ipynb` to `openapi.yaml` under Publish & Embed with `content: {application/json: {schema: {type: object}}}` (the `.ipynb` JSON schema is complex and not worth inlining; reference the Jupyter nbformat spec). Drift-detection test passes (new path, existing blueprint). Add `SimulationNotebook` to the schema section with a description pointing at the Jupyter nbformat 4 specification. Zero new deps — the `.ipynb` format is JSON, and the chart code cells are strings that execute in the user's environment (Matplotlib is not imported at generation time, only written as cell source).

### 3. Trading Signal JSON Export

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** @BaseCaptainHB's bankrbot feature is the signal that MiroShark is already being consumed as a trading intelligence source. But bankrbot is reading the share card or gallery description — there's no structured output designed for tool consumption. A `GET /api/simulation/<id>/signal.json` endpoint returns a compact, schema-locked JSON document with exactly what a trading screener or bot needs: consensus direction, confidence margin, quality tier, outcome (if resolved), simulation metadata. This is the format that bankrbot, terminal APIs, and screener tools can ingest natively. It complements the reproduce.json (which targets researchers) and the trajectory CSV (which targets analysts) — signal.json targets the trading tool layer.
**How:**
1. Add `backend/app/services/signal_export.py` (~150 LoC, pure stdlib `json` + `os` + `datetime`). `SCHEMA_VERSION = "1"`. `REQUIRED_KEYS` frozenset (same pattern as `repro_export.py`). `build_signal(state_dict, config_data, sim_dir)` returns: `{schema_version: "1", simulation_id: str, generated_at: ISO-8601 UTC, scenario_preview: str (≤140 chars), agent_count: int, total_rounds: int, consensus: {direction: "bullish"|"neutral"|"bearish", confidence_margin: float (dominant_pct - runner_up_pct, ±0.2 threshold), bullish_pct: float, neutral_pct: float, bearish_pct: float}, quality: {tier: "excellent"|"good"|"fair"|"poor", label: str (full quality_health string)}, outcome: {label: "correct"|"incorrect"|"partial"|null, resolved: bool}, lineage: {kind: "original"|"fork"|"counterfactual"}}`. `confidence_margin` is the key field for trading tools: how far ahead the dominant stance is over the runner-up, using the same ±0.2 threshold as every other surface. `null` for `outcome.label` when not yet resolved (sim is still running or operator hasn't called `/resolve`). `render_signal_bytes(blob)` — `sort_keys=True + indent=2 + trailing newline` for bytewise stability (same pattern as `render_json_bytes` in repro_export). Add `GET /api/simulation/<id>/signal.json` route (publish-gated, 5-min cache, `Content-Disposition: inline; filename="miroshark-<id12>-signal.json"`). Add 12 offline unit tests in `test_unit_signal_export.py`: SCHEMA_VERSION literal pin, REQUIRED_KEYS frozenset pin, direction correctly derived from dominant stance (using ±0.2 threshold — matches `gallery_filters.dominant_stance` logic), confidence_margin is `dominant_pct - runner_up_pct` (not `dominant_pct` alone), quality tier first-word match (mirrors `gallery_filters._quality_tier`), outcome null when not resolved, outcome correct/incorrect/partial when resolved, lineage kind original/fork/counterfactual, scenario_preview capped at 140 chars, `render_signal_bytes` is deterministic (same blob → identical bytes), route-decorator presence guard, openapi drift guard.
2. Add `GET /api/simulation/{simulation_id}/signal.json` to `openapi.yaml` under a new `Signal` tag (or under Publish & Embed) with a `SimulationSignal` schema documenting every field including `confidence_margin` semantics (the ±0.2 threshold, matching the surface-level stance computation). Add `getSignalUrl(simId, origin)` + `getSignal(simId)` helpers to `frontend/src/api/simulation.js`. In `EmbedDialog.vue`, add a "📡 Trading signal" section beneath the Reproducibility Config section: direction chip (▲ Bullish / ● Neutral / ▼ Bearish), confidence margin displayed as `+{margin:.1f}pp lead`, quality tier chip, outcome chip (if resolved), copyable signal URL, `curl -fsSL '<url>'` snippet. Collapsed by default (same chevron pattern as other EmbedDialog sections). Wire into `surface_stats.increment_surface_stat(sim_dir, "signal_json")` — extend `SURFACE_KEYS` frozenset to include `signal_json` so the Distribution panel counts it alongside other surfaces. Add `signal_json` to `SimulationSurfaceStats` schema in openapi.yaml (additive, backward compatible).
3. Add "Trading Signal Export" row to `docs/FEATURES.md` under Publish & Embed. Add a "Signal consumers" note to `docs/API.md` listing the intended downstream targets (trading screeners, terminal APIs, alert bots, bankrbot-style aggregators) and documenting the `confidence_margin` semantics explicitly — many tools will sort by this field and operators need to understand what it measures. Add signal.json to README.md Features table. Zero new deps; pure stdlib; the `dominant_stance` and `_quality_tier` logic already exist in `gallery_filters.py` and should be imported rather than re-implemented (single source of truth for the ±0.2 threshold).

### 4. Per-Agent Stance Sparklines

**Type:** Feature / Research
**Effort:** Small (hours)
**Impact:** @wshuyi's organic simulation-to-GIF demo resonated (18L/6RT) precisely because it showed individual agents evolving — not just the aggregate belief bars, but the agents themselves changing position over time. The GIF captures that motion but can't be cited or queried. A `GET /api/simulation/<id>/agent-timeline` endpoint returns per-agent stance sequences: `{agents: [{name, platform, stances: [{round, label, position_value}]}]}`. SimulationView gets a "🧬 Agent timelines" section: one row per agent, a horizontal colored-dot sequence (Bullish=green, Neutral=grey, Bearish=red) for each round, hover-tooltip showing the agent's post excerpt. This is the "who moved the needle?" view researchers cite. It answers "was consensus driven by one outlier who flipped early, or a coordinated shift?". The aggregate belief chart already answers what happened; agent timelines answer who drove it.
**How:**
1. Add `backend/app/services/agent_timeline.py` (~200 LoC, pure stdlib `json` + `os`). `STANCE_THRESHOLD = 0.2` constant (parity). `_label_for_position(pos: float) -> str`: returns `"bullish"` if `pos >= STANCE_THRESHOLD`, `"bearish"` if `pos <= -STANCE_THRESHOLD`, else `"neutral"` (same ±0.2 boundary as every other surface). `_load_round_agents(sim_dir, round_num)` reads per-round agent data from `trajectory.json` (inspect structure — `trajectory.json` likely contains a per-round `posts` array or per-round agent snapshots; if agent-per-round data is only in aggregate form, derive stances from `reddit_profiles.json`/`polymarket_profiles.json` as a per-agent position series). `build_agent_timeline(sim_dir)` assembles `{simulation_id: str, total_rounds: int, agents: [{name: str, platform: str ("reddit"|"polymarket"), stances: [{round: int, label: str, position_value: float, post_excerpt: str|null (≤80 chars)}]}], empty: bool}`. `empty: true` when no trajectory data or no per-agent data available. Cap at `MAX_AGENTS = 30` (large polymarket sims may have hundreds of agents; truncate to the 30 most-active by post count). Add `GET /api/simulation/<id>/agent-timeline` route (publish-gated, `Cache-Control: public, max-age=300`). Add 12 offline unit tests in `test_unit_agent_timeline.py`: STANCE_THRESHOLD parity guard, `_label_for_position` boundary cases (exactly ±0.2 → bullish/bearish, 0.0 → neutral), empty trajectory → `empty: true`, all-agents capped at MAX_AGENTS, agents sorted by post count desc (most active first), stances list covers all rounds (no gaps), position_value is a float in [-1.0, 1.0], post_excerpt capped at 80 chars, platform field is "reddit" or "polymarket", corrupt round skipped without blanking the agent's series, route-decorator presence, openapi drift guard.
2. Add `GET /api/simulation/{simulation_id}/agent-timeline` to `openapi.yaml` under Analytics with a `SimulationAgentTimeline` schema. Add `getAgentTimeline(simId)` helper to `frontend/src/api/simulation.js`. In `SimulationView.vue`, add a "🧬 Agent timelines" section beneath the belief chart. Render only when `payload.empty === false` and `payload.agents.length > 0`. Each agent row: `[platform emoji] [name]` label on the left (left-truncated to 24 chars if needed), then a horizontal sequence of colored squares — one per round, colored by stance (Bullish `#22c55e`, Neutral `#6b7280`, Bearish `#ef4444`). Square size: 8×8px with 1px gap — a 60-round sim produces a 8×60 + 59px = ~539px strip, which fits in the SimulationView content width. Hover on a square shows a tooltip: `Round N · [label] · "[post_excerpt]"`. When `payload.agents.length > 10`, collapse to "Show all N agents →" toggle (first 10 shown by default). Add CSS: `.agent-row`, `.agent-name`, `.agent-squares`, `.agent-sq` (8×8px colored square), `.agent-sq-bullish`, `.agent-sq-neutral`, `.agent-sq-bearish`, tooltip on `:hover`. Fetch on SimulationView mount alongside existing belief data.
3. In `EmbedDialog.vue`, add a "🧬 Agent breakdown" row beneath the Trading Signal section. Show: total agent count, platform breakdown (`N Reddit · M Polymarket`), and a "View in simulation →" link pointing to the `#agent-timelines` anchor in SimulationView (same page-anchor pattern as other EmbedDialog navigation). Do NOT re-render the sparklines in EmbedDialog — they're already in SimulationView; the EmbedDialog row is a discovery point. Add "Per-Agent Stance Sparklines" row to `docs/FEATURES.md` under Analytics. Add the endpoint to `docs/API.md` under Analytics. Zero new deps; all data on disk in existing trajectory format.

### 5. Simulation Archive Bundle

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** As MiroShark gains institutional observers (Lorimer Ventures, academic researchers, tool integrators), the request pattern is predictable: "give me everything for this simulation in one download." Right now that means six separate downloads — share card, GIF, transcript, trajectory, reproduce.json, signal.json — each with its own EmbedDialog button. A `GET /api/simulation/<id>/archive.zip` endpoint packages all available artifacts into a single ZIP: every file that exists in `sim_dir/` for the published surfaces, organized into a flat folder with a `README.txt` describing each file. The `.zip` stdlib module (`zipfile`, `io`) makes this zero-new-deps. Operators can forward the ZIP as a research handoff, attach it to a Substack post, or push it to a GitHub gist for citation. The operator profile env vars (`OPERATOR_NAME` / `OPERATOR_URL`, from the still-unbuilt May 8 idea #5) naturally slot into the README.txt attribution footer — but since that feature hasn't shipped, the attribution is optional and gracefully degrades.
**How:**
1. Add `backend/app/services/archive_export.py` (~180 LoC, pure stdlib `zipfile` + `io` + `os` + `json` + `datetime`). `ARCHIVE_FILES` ordered dict mapping artifact name → relative path within `sim_dir/` and description: `{"share-card.png": ("share-card.png", "Social share card (OG image, 1200×630)"), "replay.gif": ("replay-gifs/<cache-hash>.gif", "Animated belief replay"), "transcript.md": ("transcript.md", ...), "trajectory.csv": (...), "reproduce.json": (...), ...}`. For each file, check if it exists (`os.path.exists`) before adding — not every sim has every artifact (e.g., sims pre-dating PR #50 have no GIF, sims where `./replay.gif` was never requested have no cached GIF). Emit a `README.txt` as the first entry with: sim_id, scenario, generated_at, list of included files with descriptions, reproducibility URL, share URL. `build_archive_bytes(sim_dir, sim_id, base_url) -> bytes` streams all found files into a `zipfile.ZipFile` opened in memory (`io.BytesIO`), returns the bytes. Add `GET /api/simulation/<id>/archive.zip` route (publish-gated, no `Cache-Control` caching — archives include whatever exists at request time, which changes as surfaces are generated, `Content-Disposition: attachment; filename="miroshark-<id12>-archive.zip"`). Size guard: if the raw sum of all artifact sizes exceeds `MAX_ARCHIVE_BYTES = 50_000_000` (50 MB, enough for a few hundred GIF frames and a long trajectory CSV), return HTTP 413 with `{"error": "archive too large — download artifacts individually"}` — prevents memory exhaustion on pathological sims. Add 10 offline unit tests in `test_unit_archive.py`: valid ZIP bytes (`zipfile.ZipFile` round-trip), README.txt is first entry, missing artifact files are skipped without error, sim_id appears in README.txt, output bytes are non-empty, scenario appears in README.txt, 413 when total artifact size exceeds MAX_ARCHIVE_BYTES, route-decorator presence, openapi drift guard, archive filename contains truncated sim_id.
2. Add `GET /api/simulation/{simulation_id}/archive.zip` to `openapi.yaml` under Publish & Embed with `content: {application/zip: {}}` and a note on the 50 MB size limit and the conditional inclusion of artifacts. Add `getArchiveUrl(simId)` helper to `frontend/src/api/simulation.js`. In `EmbedDialog.vue`, add a "📦 Download archive" section at the bottom of the dialog, above the Mark outcome section. Layout: "Download all artifacts as a ZIP bundle" description, `Download archive.zip` button (same dark gradient style as "Download reproduce.json"), a note listing what's included with conditional language ("includes replay GIF if generated", "includes trajectory data if available"). Publish-gated. Wire into `surface_stats` — no surface key needed (archives are transient, not a share surface). Add `getArchiveUrl(simId)` to the copy-handlers array alongside existing surface URLs.
3. Add "Simulation Archive Bundle" row to `docs/FEATURES.md` under Publish & Embed. Add the endpoint to `docs/API.md` with a note on conditional artifact inclusion and the 50 MB cap. Add "One-click archive download" to README.md Features table. Zero new deps; pure stdlib `zipfile` + `io`. The GIF cache lookup is the only non-trivial piece: the cache filename is a hash of the sim state (from `summary_cache_key()` in `share_card_renderer.py` or equivalent) — inspect the GIF generation code to find the cache path pattern and replicate it in `archive_export.py`, or simply glob `sim_dir/replay-gifs/*.gif` and include the first match (most sims have at most one cached GIF).

## Selection Rationale

Two pressures, five ideas, one batch.

- **Webhook HMAC Signature Verification** (#1) — Integration demand is real (Revault, CancerHawk, x402). HMAC signing is the industry-standard response (Stripe's model, GitHub's model). Zero new deps, backward compatible, closes a genuine security gap before the integration count grows further. The existing `webhook_service.py` already handles dispatch and logging; this adds one function and one header.
- **Jupyter Notebook Export** (#2) — Lorimer Ventures is the first institutional-tier observer. The trajectory CSV was the foundation; the notebook closes the gap between "here's the data" and "here's the analysis, ready to run." The `.ipynb` format is plain JSON — no new deps, no runtime cost, one new service file.
- **Trading Signal JSON Export** (#3) — bankrbot is consuming MiroShark output without a structured schema. `signal.json` gives that layer a stable, schema-locked contract: direction, confidence_margin, quality tier, outcome. The `confidence_margin` field (dominant_pct − runner_up_pct, ±0.2 threshold) is the field trading tools sort by. Extends the existing `gallery_filters.dominant_stance` logic rather than reimplementing it.
- **Per-Agent Stance Sparklines** (#4) — @wshuyi's organic GIF demo showed what resonates: individual agents changing position over time. The aggregate belief chart already exists; agent sparklines add the "who moved the needle?" layer that researchers cite. All data is in `trajectory.json`; the endpoint is a read-and-format, not a new write path.
- **Simulation Archive Bundle** (#5) — The "give me everything" request is predictable at the institutional tier. Six separate downloads in the EmbedDialog collapse into one ZIP. Zero new deps (`zipfile` stdlib). The 50 MB cap and conditional file inclusion make it defensive by design.

Each idea is scoped for autonomous implementation: all data on disk, pure stdlib, follows established `_serve_X` / `_record_*` / `publish-gated` patterns. No ambiguous design decisions, no external approvals required.
