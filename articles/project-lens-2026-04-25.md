# The YAML That Matches the Code: API Documentation Stopped Being Allowed to Lie

For about fifteen years a quiet rule held in software organizations: API documentation is wrong. Not maliciously, not catastrophically — just slowly, perpetually wrong. A field gets renamed in code and the docs catch up a sprint later, or never. A new endpoint ships and the spec gets the change a week behind. A response shape evolves and the example in the README hardens into folklore. Most teams treat this as a fact of nature, the same way they treat the screenshot in the onboarding doc that still shows the 2022 dashboard.

In 2026 that resignation got expensive. A widely cited Nordic APIs analysis puts the share of APIs that don't conform to their published specs at 75 percent. A follow-up from earlier this year traces roughly 70 percent of production API failures back to contract drift — schema mismatches, renamed keys without version bumps, status codes the spec never described — and notes that these failures pass CI because tests mock the API instead of hitting it. None of this is news. What's new is who is reading API specs now. Specs are consumed by SDK generators that compile them into Python, TypeScript, and Go clients. They're consumed by LLM tool-callers that translate them into structured function definitions. They're consumed by gateway products like Speakeasy and Stainless that wire them straight into release pipelines. A wrong spec is no longer a documentation bug. It's a load-bearing lie that pushes broken type signatures into production code that other people are pip-installing.

## What the Pull Request Actually Says

On April 25 the open-source multi-agent simulator MiroShark merged PR #45: a 1,966-line OpenAPI 3.1 YAML covering ~85 REST endpoints, a Flask blueprint that serves Swagger UI at `/api/docs`, and YAML and JSON variants of the spec at `/api/openapi.yaml` and `/api/openapi.json`. The visible artifact is a try-it-out page anyone can click. The headline reason to ship it, written in the PR description, is that yesterday's MCP onboarding panel left "what REST endpoints can I call directly?" unanswered for the developer audience that had just been routed in.

The architectural decision underneath is a single test file: `backend/tests/test_unit_openapi.py`. It is the centerpiece of the PR, and it is the unsexy part nobody outside engineering will ever read. Its job is to make the YAML true.

## Why a Regex Test Is the Spec's Backbone

The test does roughly four things. It loads `openapi.yaml` and asserts the minimum OpenAPI 3.x shape. It scans every `app/api/*.py` source file for the regex `@<name>_bp.route('<path>')`, extracts each path, and prefixes it with the blueprint's mount point from a hand-maintained `_BLUEPRINT_PREFIXES` map (`graph_bp` → `/api/graph`, `share_bp` → `""`, and so on). It converts Flask's angle-bracket parameters like `<simulation_id>` and `<int:round_num>` into OpenAPI's curly-brace form `{simulation_id}`. Then it asserts that the set of routes the regex found equals the set of paths declared in the spec, minus an explicit allowlist of internal endpoints — debug routes, operator-only env management, ad-hoc helpers.

The point is what happens when the two sets don't match. The next pull request that adds a Flask route without documenting it fails CI. The next pull request that removes a route without taking it out of the YAML fails CI. The next pull request that renames `<sim_id>` to `<simulation_id>` and forgets to propagate fails CI. There is no human in the review loop responsible for catching drift, because there is a 30-line static scan responsible for catching drift, and it runs on every push.

This is the difference between API documentation that exists and API documentation that is true. Most projects that ship OpenAPI specs treat them as artifacts — generated once, edited by hand, drifting after the second commit. Treating the spec as code, with a test that fails the build when the spec lies, is the principle behind most of the contract-testing tooling that landed in 2026: oasdiff for diffing spec versions, Schemathesis for property-based fuzzing, Spectral for governance linting, Mokapi for transparent request/response validation. MiroShark's version is small enough to fit in one Python file because the project is small enough not to need the ecosystem yet. The shape is the same.

## What That YAML Earns You

Once an OpenAPI spec is verifiably current, three things downstream get cheap. The first is SDK generation: `openapi-generator` takes the file and emits typed Python, TypeScript, Go, and Java client libraries in one command. A consumer who wants to drive MiroShark's API from a Next.js app no longer reads docs and writes a fetch wrapper; they pipe the YAML through one CLI and import the result. The second is discoverability: APIs.guru and similar registries scrape OpenAPI URLs from across the public web and list anything that parses, surfacing a population of API-shopping developers the project could not previously reach. The third — the one most relevant to where MiroShark is heading — is LLM tool calling.

Modern LLMs that interact with external systems through function calling rely, increasingly, on OpenAPI specs as the structured contract that prevents the model from hallucinating parameters or endpoints. The Xano and Gravitee writeups from earlier this year both name OpenAPI as "the single source of truth" for AI tool definitions. Anthropic's Model Context Protocol can be auto-derived from an OpenAPI document via FastMCP and openapi-mcp-generator. Yesterday's MCP onboarding panel exposed eight curated tools over the knowledge graph. Today's OpenAPI spec, kept honest by a regex test, is what lets future agents reach the rest of the API surface without a human writing each tool description by hand.

## Boring Plumbing, Big Multiplier

Most load-bearing decisions in software look small from the outside. A 1,966-line YAML and a 321-line test file are not the parts of a product anyone advertises. They're also the parts that decide whether everything downstream — typed clients, SDK release pipelines, MCP servers, agent integrations, third-party listings — can be trusted. The interesting thing about PR #45 isn't the OpenAPI spec. It's that the YAML is going to keep matching the code, because a test will fail when it doesn't.

In 2026 that's no longer a documentation hygiene improvement. It's the thing that decides whether your software is programmable to other people's machines.
