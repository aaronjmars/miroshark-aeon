## Ecosystem Context

Two arcs just closed in rapid succession. PR #79 (HMAC signing) + PR #80 (Jupyter export) shipped back-to-back: one tightened the integration security layer, the other handed institutional researchers a publish-ready artifact. The reproduce.json → lineage → surface-stats → trending → HMAC → notebook chain that started with PR #74 on May 7 is now 7 PRs deep in five days.

The question this batch asks is: *what does the platform need at exactly this momentum point?*

Three signal threads drive this batch:

- **The integration event-stream gap.** Revault and CancerHawk are live integrations, confirmed by multiple community members. Both are receiving MiroShark webhooks — but only on simulation *completion*. A tool like Revault (sneaker market intelligence, multi-round sims) wants to know when a sim *starts*, when it crosses quality thresholds, and when consensus is reached mid-run — not just when the final round closes. The webhook service, dispatch path, and HMAC signing infrastructure are all in place. The event vocabulary is the only gap.

- **The embed surface gap.** The share card is a static PNG (PR #42), the GIF is an animation (PR #50), the watch page is a full SPA. There is no lightweight interactive embed: a single `<iframe>` snippet that works in Substack, Notion, and Ghost — platforms that support iframes but not the full SPA router. A minimal `/embed/<id>` page (belief bars, scenario title, quality badge, one link) fills this gap without any new deps. The current oEmbed endpoint (if built) would return a `type: "photo"` response — a static PNG. An iframe is `type: "rich"` — interactive, live-updatable while a sim is running.

- **The discovery / SEO gap.** MiroShark's public gallery grows with every publish. But search engines have no structured entry point into the simulation corpus. A `sitemap.xml` that lists every published simulation's `/share/<id>` and `/watch/<id>` URL — auto-regenerated from the public sim index — makes the gallery crawlable. The "Aave risk simulation from May 8" that Aaron tweeted would surface on Google when someone searches the scenario keywords. This compounds over time: each published sim becomes a potential organic entry point.

Two additional ideas round out the batch:

- **The feed filtering gap.** The RSS/Atom feed (PR #60) returns all public sims in date order. An operator who only wants "bullish consensus + excellent quality" sims in their feed can't express that. Adding the same `?consensus=&quality=&sort=` query params the gallery API already supports turns the feed into a structured signal source for n8n, Zapier, Make, and RSS aggregators — without any new data or logic.

- **The step-through API gap.** The trajectory CSV (PR #66) and the Jupyter notebook (PR #80) both dump all rounds at once. There is no endpoint for "give me exactly round 15" — a scrubber API that lets integrators build step-through replay UX, or fetch just the mid-run belief state for a round-in-progress sim. `GET /api/simulation/<id>/round/<n>` fills this gap without any new write paths.

### 1. Simulation Lifecycle Webhooks (granular event subscriptions)

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** Revault and CancerHawk are consuming MiroShark webhooks today — but only on simulation completion. Adding three new event types (`simulation_started`, `consensus_reached`, `quality_milestone`) with opt-in subscription via `WEBHOOK_EVENTS` env var (comma-separated event names, default `simulation_complete` for backward compatibility) gives integrators an event stream rather than a single signal. A sneaker-market intelligence tool that wants to know *when a consensus forms mid-run* — not just when the last round closes — can subscribe to `consensus_reached` only. The `X-MiroShark-Event` header already signals event type (present since PR #73); this extends the event vocabulary and the dispatch trigger points. HMAC signing from PR #79 applies to all event types automatically. Zero new deps; fully backward compatible.
**How:**
1. In `backend/app/services/webhook_service.py`, define `WEBHOOK_EVENT_TYPES = frozenset({"simulation_started", "round_complete", "consensus_reached", "quality_milestone", "simulation_complete"})`. Add `_resolve_subscribed_events() -> frozenset` that reads `os.environ.get("WEBHOOK_EVENTS", "simulation_complete")`, splits on commas, strips whitespace, filters to valid event types (unknown names silently ignored). Add `is_subscribed(event_name: str) -> bool` wrapper. Modify each dispatch call site to call `is_subscribed(event_name)` before firing — if not subscribed, no-op. Add dispatch calls at the new trigger points: `simulation_started` at the beginning of the simulation run loop (after the first-round setup); `consensus_reached` when the dominant stance first reaches ≥50% majority and holds for 2+ consecutive rounds (inspect `simulation_state.json` update logic for the right hook); `quality_milestone` when `quality_health` tier transitions from "fair" → "good" or "good" → "excellent" (read the quality tier computation location in the codebase — likely in `simulation_state.py`). The `round_complete` event fires after each round's belief state is written to disk (at the existing round-write hook) — but only if `round_complete` is in `WEBHOOK_EVENTS` (high-volume by default; operators opt in deliberately). Each event payload includes `{event: str, simulation_id: str, timestamp: ISO8601, ...event_specific_fields}`: `simulation_started` adds `{scenario_preview, agent_count, total_rounds}`; `consensus_reached` adds `{consensus_direction, bullish_pct, neutral_pct, bearish_pct, round: int}`; `quality_milestone` adds `{from_tier, to_tier, round: int}`; `round_complete` adds `{round: int, bullish_pct, neutral_pct, bearish_pct}`. Add 12 offline unit tests in `test_unit_webhook_events.py`: `WEBHOOK_EVENTS` unset → only `simulation_complete` subscribed; comma-separated list parsed correctly; unknown event names silently dropped; `simulation_started` fires before first round; `consensus_reached` fires at ≥50% majority first held for 2+ rounds; `quality_milestone` fires on tier transition; `round_complete` only fires when explicitly subscribed; HMAC signing applies to all event types; `X-MiroShark-Event` header carries the correct event name; backward compatibility — existing `simulation_complete` webhook behavior unchanged when `WEBHOOK_EVENTS` is unset.
2. Extend `docs/WEBHOOKS.md` and `docs/WEBHOOKS.zh-CN.md` with an "Event types" section: table of all 5 event names with trigger conditions, example payloads, and the `WEBHOOK_EVENTS` configuration reference. Add `WEBHOOK_EVENTS` to `openapi.yaml` `WebhookDeliveryEntry` schema description note. Update `README.md` environment variable reference with `WEBHOOK_EVENTS` (default: `simulation_complete`; accepts comma-separated list). Add `WEBHOOK_EVENTS` to `.env.example` with a comment: "Set to `all` for all events, or comma-separated: simulation_started,consensus_reached,simulation_complete".
3. In `EmbedDialog.vue`, add an "Event subscriptions" row beneath the HMAC hint in the webhook section. Show a compact list of the 5 event types with checkmarks indicating which are currently active (derive from `WEBHOOK_EVENTS` env var — expose via a new `GET /api/config/webhook-events` public endpoint that returns `{subscribed: [str]}`, no auth required, no secrets exposed). If `WEBHOOK_EVENTS` is unset or `simulation_complete` only, show "Receiving: completion events only — set `WEBHOOK_EVENTS` for real-time hooks" with a link to `docs/WEBHOOKS.md`. Add the endpoint to `openapi.yaml` under Webhooks; add the path to drift-detection test. Add 3 additional tests: `GET /api/config/webhook-events` returns correct list when `WEBHOOK_EVENTS` set, returns `["simulation_complete"]` when unset, secrets are not exposed (no WEBHOOK_SECRET value in response).

### 2. Interactive Embed Widget (`/embed/<id>` iframe)

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** Platforms that support iframes — Substack, Notion, Ghost, Mirror.xyz — cannot use a static PNG embed and won't render a full SPA route. If the sim is still running, the page polls `GET /api/simulation/<id>/status` every 10 seconds and updates the bars — giving publishers a live-updating embed they didn't have before. If the sim is complete (or unpublished), the bars are static. The entire page is self-contained HTML + inline CSS (~80 lines) with no JavaScript framework, no bundler output — it loads in any iframe sandbox. EmbedDialog gains one new row: "🖼 Embed widget" with a copy-ready `<iframe>` snippet and dimensions note (400×200px default). Zero new deps; pure stdlib Flask response.
**How:**
1. Add `GET /embed/<simulation_id>` route to `backend/app/api/share.py` (or a new `embed.py` blueprint). Publish-gate: 404 for unpublished sims (opting for 404 over 403 to avoid iframe error display on external pages). Load the sim via `SimulationManager.get_simulation(sim_id)` and call `_build_embed_summary_payload(sim)` to get belief state + scenario + quality + consensus. Render the response as `text/html` with `Content-Type: text/html; charset=utf-8` and a permissive `X-Frame-Options: ALLOWALL` header (or `Content-Security-Policy: frame-ancestors *`) so the page embeds in third-party sites. The HTML template (build inline as a Python string — no Jinja2 required): `<!DOCTYPE html>` with inline `<style>` block (flexbox belief bars — `.bar-bullish {background:#22c55e}`, `.bar-neutral {background:#6b7280}`, `.bar-bearish {background:#ef4444}`, each `height:16px; border-radius:4px; transition:width 0.4s`), a 3-bar belief row, scenario title, quality badge, consensus label, and a `<a href="/share/{sim_id}" target="_blank" rel="noopener">View full simulation →</a>` link. If `is_running: true` in the embed summary, add a `<script>` block (inline, no external dependencies) that polls `fetch('/api/simulation/{sim_id}/status')` every 10s and updates bar widths via `element.style.width`. Cap polling at 200 rounds (failsafe) or stop when `run_complete: true`. Add `Cache-Control: no-store` when `is_running: true`; `Cache-Control: public, max-age=300` when complete. Add 10 offline unit tests in `test_unit_embed_widget.py`: published complete sim returns 200 HTML, unpublished returns 404, `X-Frame-Options: ALLOWALL` present, belief bars have correct width percentages, scenario truncated to 100 chars, share link present in body, quality badge present, consensus label present, `is_running: true` sim includes polling script, `Cache-Control: no-store` on live sims.

Updates live while simulation runs." Collapse the preview behind a "Show preview →" toggle to avoid layout shift on dialog open. ### 4. Per-Round Belief Snapshot API (`GET /api/simulation/<id>/round/<n>`)

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The trajectory CSV (PR #66) and Jupyter notebook (PR #80) dump all rounds at once — appropriate for download-and-analyze, wrong for step-through and real-time monitoring. Integrators building a replay scrubber, a round-level citation tool, or a live monitoring dashboard need a "give me exactly round 15" endpoint. `GET /api/simulation/<id>/round/<n>` returns the belief state at a specific round: `{round: int, total_rounds: int, bullish_pct: float, neutral_pct: float, bearish_pct: float, stance_label: str, sample_posts: [{agent_name, platform, stance, text_excerpt (≤80 chars)}], director_events: [{label, description}] | [], is_complete: bool}`. The data is already on disk in `trajectory.json` (or per-round files — inspect the sim_dir structure). This is a read-and-slice, not a new write path. A sim still in progress responds with whatever rounds are complete (`is_complete: false`; requesting a future round returns 404). This turns the trajectory into an addressable document — each round has a stable URL — which is the prerequisite for round-level citations and embeds.
**How:**
1. Add `backend/app/services/round_snapshot.py` (~150 LoC, pure stdlib `json` + `os`). `STANCE_THRESHOLD = 0.2` constant (parity). `_label_stance(bullish, neutral, bearish) -> str`: returns dominant stance name using ±0.2 threshold — same logic as `gallery_filters.dominant_stance`. `_load_round_data(sim_dir, round_num: int) -> dict | None`: reads `trajectory.json` (load the array, index by `round_num - 1`; handle both 0-indexed and 1-indexed depending on what the trajectory uses — inspect the existing `trajectory_export.py` for the exact structure). Returns `None` if round doesn't exist yet (sim in progress, or round_num > completed rounds). `_sample_round_posts(sim_dir, round_num, n=5)`: reads per-round agent post data from the same source as `agent_timeline.py` would use (inspect trajectory structure). Returns up to 5 posts from that round with agent name, platform, stance label, and text excerpt capped at 80 chars. `_load_director_events_at_round(sim_dir, round_num)`: reads `simulation_state.json` or equivalent for director events at the given round. Returns `[]` if none. `build_round_snapshot(sim_dir, sim_id, round_num) -> dict`: assembles the full response payload. Add `GET /api/simulation/<id>/round/<n>` route (publish-gated — 403 for unpublished; 404 when `_load_round_data` returns `None` i.e., round not yet complete or beyond total_rounds; `n` must be a positive integer, 400 for invalid input). `Cache-Control: public, max-age=3600` for complete sims (belief state at a past round never changes); `Cache-Control: no-store` for the current/live round (may still be updating). Add 14 offline unit tests in `test_unit_round_snapshot.py`: valid round 1 returns correct belief state, valid round N (last) matches final consensus, `round_num > total_rounds` returns 404, `round_num = 0` returns 400 (invalid), `round_num` as float string returns 400, unpublished sim returns 403, in-progress sim round beyond current returns 404, in-progress sim completed round returns 200, `STANCE_THRESHOLD` parity guard (0.2 exactly), sample_posts capped at 5, text_excerpt capped at 80 chars, director_events present when injected at that round, `director_events: []` when none, `is_complete` reflects whether total_rounds reached.
2. Add `GET /api/simulation/{simulation_id}/round/{round_num}` to `openapi.yaml` under Analytics with a `RoundSnapshot` schema. Drift-detection test passes. Add `getRoundSnapshot(simId, roundNum)` helper to `frontend/src/api/simulation.js`. In `SimulationView.vue`, add a round scrubber input to the belief chart controls: an `<input type="range" min="1" :max="totalRounds" v-model="selectedRound">` + `<span>Round {{ selectedRound }} / {{ totalRounds }}</span>`. When `selectedRound` changes (debounced 300ms), fetch `GET /api/simulation/<id>/round/<selectedRound>` and update the belief bar data shown beneath the chart (separate from the main belief chart — a "Round detail" card below). The main Chart.js belief chart always shows the full trajectory; the round detail card shows the point-in-time view for the selected round. Show `director_events` at that round as an amber ⚡ annotation beneath the round detail card (same amber color as planned for the Director Event Timeline). If `sample_posts.length > 0`, show a collapsed `<details>` with the post excerpts.
3. Add `GET /api/simulation/<id>/round/<n>` to the surface-stats increment path: call `surface_stats.increment_surface_stat(sim_dir, "round_snapshot")` in the route handler. Extend `SURFACE_KEYS` with `"round_snapshot"`. Add "Per-Round Belief Snapshot" row to `docs/FEATURES.md` under Analytics. Add the endpoint to `docs/API.md` under Analytics with a note on caching behavior (immutable for completed rounds, no-store for live). Add a "Round-level citation" note: each round has a stable URL (`/api/simulation/<id>/round/<n>`) that can be cited in research without downloading the full trajectory. Zero new deps.

### 5. Sitemap.xml for Published Simulations (SEO + web search discoverability)

**Type:** Growth / SEO
**Effort:** Small (hours)
**Impact:** MiroShark's public gallery grows with every publish. But search engines have no structured entry point: `/share/<id>` and `/watch/<id>` pages aren't linked from a crawlable index that Googlebot, Bingbot, or DuckDuckGo can discover. A `GET /sitemap.xml` endpoint auto-generates an XML sitemap from the public sim index — one `<url>` entry per published sim, with `<loc>` pointing to `/share/<id>`, `<lastmod>` from the sim's `updated_at` timestamp, and `<changefreq>weekly</changefreq>` for completed sims (`always` for in-progress). Submitted to Google Search Console once, it enables organic discovery of simulation results via web search: the "Aave rate scenario from May 8" published by Aaron surfaces when someone searches the scenario keywords. This compounds: each new published sim becomes a new organic entry point, compounding the distribution advantage of the reproduce.json citation key that PR #75 introduced. Pure stdlib `xml.etree.ElementTree`, zero new deps. A `robots.txt` entry (`Sitemap: https://<host>/sitemap.xml`) auto-registers it with all compliant crawlers without needing to submit manually.
**How:**
1. Add `backend/app/services/sitemap.py` (~120 LoC, pure stdlib `xml.etree.ElementTree` + `os` + `datetime`). `SITEMAP_NAMESPACE = "http://www.sitemaps.org/schemas/sitemap/0.9"`. `build_sitemap(public_sims: list[dict], base_url: str) -> bytes`: builds a `<urlset>` document with one `<url>` block per sim. Each block: `<loc>{base_url}/share/{sim_id}</loc>`, `<lastmod>{updated_at[:10]}</lastmod>` (date only, ISO 8601 format — day precision is the sitemap spec norm), `<changefreq>always</changefreq>` if `is_running: true` else `weekly`, `<priority>0.8</priority>` for completed sims (main share surface) and `0.6</priority>` for in-progress. Also include the `/watch/<sim_id>` URL for each sim (`<changefreq>always</changefreq>` if running else `daily`, `<priority>0.7</priority>`). Cap at `MAX_SITEMAP_URLS = 1000` entries (Google's practical limit per sitemap file; MiroShark won't hit this soon, but the guard is correct). `render_sitemap_bytes(public_sims, base_url) -> bytes`: serialize the ElementTree to bytes with `xml_declaration=True, encoding="utf-8"` + trailing newline. Add `GET /sitemap.xml` route to `backend/app/api/routes.py` (or a new `sitemap.py` blueprint): calls `SimulationManager.list_public_simulations()` with no filters (all public sims), passes to `build_sitemap`, returns with `Content-Type: application/xml; charset=utf-8` and `Cache-Control: public, max-age=3600` (hourly refresh — a new publish triggers a cache refresh on the next request). Add 10 offline unit tests in `test_unit_sitemap.py`: valid XML (`xml.etree.ElementTree.fromstring` round-trip), `<urlset>` namespace matches sitemap spec, all public sims appear in sitemap, unpublished sims excluded, `<loc>` contains correct `/share/<id>` URL, `<lastmod>` is date-only ISO 8601, `<changefreq>` is `weekly` for completed sims, `<changefreq>` is `always` for in-progress sims, cap at MAX_SITEMAP_URLS (1001st sim excluded), route-decorator presence guard.
2. Add `/robots.txt` route (or update existing one if present) to include `Sitemap: {PUBLIC_BASE_URL}/sitemap.xml`. If `robots.txt` is already served as a static file, add the Sitemap line to it. Add a `User-agent: *` / `Allow: /share/` / `Allow: /watch/` block to explicitly permit crawlers on the public simulation surfaces (if not already present). Add `GET /sitemap.xml` and `GET /robots.txt` to `openapi.yaml` under a new `Discovery` tag (or under a `Site` tag) — or omit from the API spec if these are considered infrastructure rather than API endpoints (reasonable judgment call; the drift-detection test only guards the `_BLUEPRINT_PREFIXES` list, not static file routes). Document the sitemap in `docs/API.md` under a "Search Engine Discoverability" section: explain the sitemap URL, cache TTL, submission instructions for Google Search Console (one-time manual step by the operator), and the robots.txt signal. Add `ENABLE_SITEMAP=true` to `.env.example` with a note: "Set to 'false' to disable sitemap generation if the instance is private or not indexed."
3. In `EmbedDialog.vue`, add a "🔍 Search indexing" row at the bottom of the dialog (publish-gated). Content: "This simulation is included in /sitemap.xml — search engines can discover it." Link to `{base_url}/sitemap.xml` (opens in new tab). If `ENABLE_SITEMAP` is false (expose via a new `GET /api/config/sitemap` public endpoint returning `{enabled: bool}`, same pattern as the webhook-events config endpoint from Idea #1), show "Sitemap disabled — set `ENABLE_SITEMAP=true` to enable search engine indexing" instead. Add "Search Engine Sitemap" row to `docs/FEATURES.md` under Publish & Embed. Add a note to README.md Features table: "Auto-generated sitemap.xml for Google Search Console registration." Zero new deps.

## Selection Rationale

Five ideas, one batch, one moment: the token crossed $1M FDV today and two major PRs (#79 + #80) merged in the last 24 hours. The platform is in active use by external integrators. This batch serves the layer that comes after surface multiplication: event granularity for live integrators, interactive embeds for publishing platforms, filtered feeds for tool consumers, round-level addressability for citations, and search-engine discoverability for long-tail organic growth.

- **Simulation Lifecycle Webhooks** (#1) — Revault and CancerHawk are live. They need an event stream, not a single completion signal. The webhook, dispatch, logging, and HMAC infrastructure are fully in place; the event vocabulary is the only gap. Extending it is additive and backward compatible.
- **Interactive Embed Widget** (#2) — Substack, Notion, Ghost, and Mirror support iframes. None of them can use the current static PNG embed meaningfully. A no-chrome iframe page with live-updating belief bars is the intermediate surface that's been missing from the distribution stack. Zero new deps; pure HTML + inline CSS.
- **Filtered RSS/Atom Feed** (#3) — The feed is a distribution channel. Right now it's all-or-nothing. The gallery filter logic is already tested and proven; composing it with the feed route is 1–2 hours of plumbing, not a new algorithm. Immediately useful for n8n/Zapier/Make integrations and RSS-based trading signal monitoring.
- **Per-Round Belief Snapshot API** (#4) — The trajectory CSV and Jupyter notebook dump everything at once. A round-addressable API (stable URL per round) is the prerequisite for round-level citations, step-through replay UX, and live monitoring of in-progress sims. All data is on disk; this is a read-and-slice.
- **Sitemap.xml** (#5) — Each published simulation is a web page that a researcher might search for. Right now Googlebot has no structured way to discover them. A sitemap, submitted once to Google Search Console, compounds every future publish into organic search discoverability. Pure stdlib `xml.etree.ElementTree`, minimal effort, asymmetric long-term payoff.

Each idea follows the established pattern: pure stdlib, publish-gated, zero new deps, extends existing services rather than adding new data pipelines. Scoped for autonomous implementation by the `feature` skill.
