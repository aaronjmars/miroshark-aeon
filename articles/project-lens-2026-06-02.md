# What Stays vs What Changes: A Small API Decision That Says a Lot

On September 22, 2025, OpenAI quietly published a developer blog post explaining why it had introduced a new API endpoint called Responses, parallel to its older Chat Completions endpoint. Buried in the rationale was a sentence about a design choice that most readers skipped: *"Responses preserves the model's reasoning state across those turns... step-by-step thought processes actually survive into the next turn."* The hidden argument was that you couldn't keep adding more keys to one endpoint forever. The state of a conversation deserved its own resource, separate from the act of asking a single question.

Over the eight months since, the same argument has been showing up in places that have nothing to do with OpenAI. Langfuse, the open-source observability tool that a growing number of AI teams use to debug agents, documents its core data model with the line *"Sessions in Langfuse are a special way to group these observations across traces together and see a simple session replay."* A trace is what an agent did once. A session is the agent across many traces. Different things, modelled separately, queried separately. In December 2025, the engineer Ted Tschopp wrote up the [MCP 2026 roadmap](https://tedt.org/MCPs-2026-Roadmap/) and described an emerging convention: *"MCP Server Cards [are] a standard for exposing structured server metadata through a .well-known URL, allowing external systems to discover what a server can do without first opening a full MCP connection."* The capabilities of an AI server are now their own URL. The runtime is another URL.

This is the boring half of API design — the half that doesn't get written up in launch threads — and it is doing more work in 2026 than it gets credit for.

## The principle

There is a phrase from the early REST literature that has been making a quiet comeback in 2026 writing: *nouns, not verbs.* A February [Medium piece](https://medium.com/@md.mohiuddin/rest-api-design-in-2026-whats-changed-what-still-works-8f2f09e925e2) on REST design in the agent era opens with that line and adds, *"Machine-readable APIs are more important than ever."* The principle behind it is older than the web. When you are designing an interface, the temptation is to fold every fact about a thing into one big response. It feels parsimonious. One endpoint, one payload, one fetch. But the things that change every time you look at them, and the things that almost never change, want different lifetimes — different cache headers, different access patterns, often different consumers entirely. A [2026 best-practices guide](https://dasroot.net/posts/2026/01/restful-api-design-best-practices-2026/) puts the cache-control consequence plainly: *"Ensure that Cache-Control headers are set appropriately for different resource types."* Splitting them is a way of admitting that you have two resources, not one.

## Inside an agent debate

The simulation platform MiroShark runs a kind of synthetic debate. A scenario is posed, dozens or hundreds of AI personas — each one assembled from a Reddit-style profile, a demographic, an MBTI type, an interest set — argue it out over a fixed number of rounds. At the end you have two interesting things. You have *who was in the room*: their names, their backstories, their final stances. And you have *what their minds did*: the round-by-round trajectory of how each agent's belief moved.

Earlier this spring, on May 27, the project merged a small endpoint called `/api/simulation/<id>/agents/sparklines`. It returns the second thing: the trajectory of each agent's belief across rounds, suitable for drawing the little inline charts you see in finance dashboards. Today, on June 2, it merged a sibling called `/api/simulation/<id>/agents.json`. It returns the first thing: the roster — name, handle, 280-character persona preview, demographics, final position, rounds participated. Same agents. Same simulation. Two URLs.

The natural question is: why not one?

## Why two is the right answer

The answer is the boring half. The roster of who debated is *structural*. Once the simulation has finished, those agents are exactly who they were. The 1-hour cache lifetime on `agents.json` reflects this — a downstream tool can fetch it once, sit on the response for an hour, and not be wrong about anything. The belief trajectory, by contrast, sits behind a 5-minute cache, because during a live simulation it changes. Folding both into one endpoint forces one cache lifetime onto both, which forces a choice between giving stale roster data to live consumers or giving stale belief data to roster consumers. The split lets each behave correctly.

There is also a payload argument. A consumer that wants to render a "who was in this debate" card on a third-party site does not need the trajectory arrays. A consumer that wants to chart the trajectories does not need the demographic tags. Bundling them would push every byte of one to every consumer of the other, which on a single-agent endpoint is cheap and on a thousand-agent endpoint is not. The split lets each consumer pay for what it asks for.

And there is a discoverability argument, which is the part most often missed. When the project's catalog of API surfaces — itself an endpoint, `/api/surfaces.json` — lists `agents.json` and `agents/sparklines` as two separate entries, an integrator scanning the catalog sees that the platform thinks of the roster and the trajectory as two different facts. One endpoint with twenty keys hides the seam. Two endpoints make it visible.

## What it tells you about a project

The temptation, when you build an API, is to write one endpoint that returns *everything we know about this thing*. It feels like generosity. The harder discipline is to ask, before merging, which parts of the response change on different clocks, which parts have different consumers, which parts deserve their own URL. The split is a small decision. The fact that someone made it, instead of bundling, is a larger one. It suggests the project expects to be talked to by tools it doesn't know about yet, and is leaving them room to ask for only the part they need. That is what a 2026 API designed for an agent-shaped consumer base ends up looking like, even when the underlying thing being modelled is itself a group of agents.
