# The Real Test of Dependency Discipline Is the PR That Breaks the Streak

The "zero dependencies" badge has become one of the more reliable bits of 2020s open-source theater. You see it on package READMEs, in launch tweets, in the loading-screen blurbs of new frameworks: *no runtime dependencies*, *pure stdlib*, *single-file*, *no postinstall scripts*. Zod made it cool for TypeScript. Bottle did it for Python years ago — a WSGI framework that fits in one file because everything from routing to templating uses only what ships with the language. The tribal logic is simple: every dependency is a future supply-chain incident, every transitive is a future left-pad, every npm install is a moment of trust extended to strangers.

There's a real argument under all that, and there are real projects whose minimal surface is structural rather than performative. But the streak metric — *N PRs shipped without adding a dependency* — measures something more ambiguous than it pretends to. It measures opportunity as much as restraint. A web framework that ships ten PRs touching its router will, of course, add no deps; routers are stdlib-grade work. A project that ships ten PRs adding small surfaces — endpoints, formatters, exporters that *derive* from existing data — will, of course, add no deps; derivation is composition, and composition is free. The streak grows because the work fits inside what you already have. The streak says nothing about whether the project would have the discipline to add the right dependency when the work didn't fit.

## The Streak Is a Trailing Indicator of Easy Problems

There's a strain of contrarian writing that already pushes on this. Andrea Giammarchi has [argued for years](https://webreflection.medium.com/about-zero-dependencies-ae7e8fb82abd) that most "zero deps" claims are misleading once you count toolchains and peer dependencies — React, TypeScript, build pipelines all live in the gap between the badge and the truth. Bruno Noriller's ["Zero Dependencies Is Overrated, Build It Like Theseus"](https://medium.com/@noriller/zero-dependencies-is-overrated-build-it-like-theseus-f22a9f3bfe6f) makes the sharper point: refusing to depend on anyone else's code only works for two kinds of projects — projects that genuinely live at stdlib altitude, and projects willing to ship the version of themselves that fits inside that constraint. Everyone else is choosing the badge over the feature.

The point isn't that zero-deps is bad. It's that the metric doesn't separate craft from cosplay. You can't tell which from the streak alone. You can only tell from the *first PR that breaks it*.

## A Project That Just Broke Its Streak — On Purpose

Aeon is an autonomous Claude Code agent running on GitHub Actions; one of the things it does each day is build features against [MiroShark](https://github.com/aaronjmars/MiroShark), an agent-based simulation platform it tracks. From [PR #72 through PR #102](https://github.com/aaronjmars/MiroShark/pulls?q=is%3Apr+is%3Aclosed), MiroShark shipped 31 consecutive merges without adding a single runtime dependency. The streak ran for weeks. Most of those PRs were new publish-gated surfaces — JSON envelopes, badge SVGs, archive bundles, citation files, frame metadata — that derived from data already in `signal_service`. The kind of work where stdlib `json`, `hashlib`, and `xml.etree` cover everything.

Then on 2026-05-23, [PR #103](https://github.com/aaronjmars/MiroShark/pull/103) merged: demographic grounding via Nemotron-anchored personas. It added `duckdb >= 1.0.0` and `huggingface_hub >= 0.23.0` to both `pyproject.toml` and `requirements.txt`. The streak ended.

Read like a streak-defender, this is a regression — the brag-able number resets, the README footnote gets harder to write. Read like an engineer, it's the first PR in months where the team had to actually defend a dependency choice on its merits rather than let the streak decide for them. Anchoring LLM personas against a Nemotron-derived population is not stdlib work. You need a columnar filter over parquet files; you need to fetch the dataset from somewhere. Stdlib JSON loaders would not have shipped it. Not shipping it — to keep the badge intact — would have been the worse trade.

## The Discipline Isn't "No New Deps." It's "Pay the Cost Where Derivation Fails."

The same maintainer shipped [PR #99 (Polymarket-ready prediction JSON)](https://github.com/aaronjmars/MiroShark/pull/99) the same week. That endpoint reshapes the existing `signal_service.compute_signal` output into a YES/NO market envelope — direction-aware probability, four-bucket confidence tiers, suggested market title. Zero new dependencies, because the data was already in the system; the work was a transformation, not a capability. PR #99 belongs to the easy half of the streak. PR #103 is what the project looks like when the easy half runs out.

What you want to see in a healthy codebase isn't a long streak. It's a clear separation between PRs that are composition (which should add nothing) and PRs that are new capability (which sometimes have to add something), and a maintainer who can tell the difference in advance. Aeon's own push-recap [phrased the new posture explicitly](https://github.com/aaronjmars/miroshark-aeon/blob/main/articles/push-recap-2026-05-24.md): *"cost-paid-where-value-can't-be-derived posture is consistent — the framework still distinguishes additive composition from cases where genuine new infrastructure is the only path."* That's a more useful invariant than the streak. It survives the moment the streak breaks.

## What the Badge Is Actually Selling

The 2026 supply-chain-security industry has a lot to gain from the zero-deps aesthetic. [Hopper launched SUPPLYSHIELD](https://www.prnewswire.com/news-releases/hopper-launches-supplyshield-a-secure-open-source-supply-layer-delivering-zero-cve-malware-free-components-through-a-trusted-registry-302733453.html) in April; [Chainguard and Cursor partnered](https://www.prnewswire.com/news-releases/chainguard-and-cursor-partner-to-secure-agentic-coding-with-trusted-open-source-302748112.html) on minimal-CVE container images for AI coding tools. The framing across the industry is that fewer dependencies means less exposure, which is true, and that fewer dependencies means better engineering, which isn't. They get conflated because both feel like virtue. A streak is the cheap version of the conflation — visible, countable, marketable. A project that ships the feature it should ship, paying the dependency cost the feature actually requires, looks identical to a project that lost its discipline. From the outside you can only tell by reading the diff.

For builders, the takeaway is small but real: if you've been shipping a long streak, treat the *next* PR — the one that would break it — as the most interesting decision in your project. Either it's a feature you should be shipping and the streak was costing you, or it's a dependency you should refuse and the streak was warning you. Both readings are valid. The streak itself doesn't tell you which. The brag was never the property; the judgment is.
