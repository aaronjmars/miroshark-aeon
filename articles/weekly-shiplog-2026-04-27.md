# Week in Review: The Week MiroShark Became Addressable

*2026-04-27 — Weekly shipping update*

## The Big Picture

Seven days ago, MiroShark's knowledge graph and MCP server were live but completely invisible to users — a substrate without a doorway. This morning the project has three formally documented protocols over the same engine (MCP, OpenAPI 3.1, Completion Webhook), a public discovery surface (`/explore`), an Open Graph share card that auto-unfurls every published run, a Cheap-preset onboarding path that anchors a "$1 & under 10 min" landing claim, and a 243-line README that replaces a 698-line one. Seven PRs merged on MiroShark, seven on miroshark-aeon, plus the largest direct-push night of the project's life on Apr 21 (Cheap preset, runner hardening, README rewrite, OASIS→Wonderwall, simulation-page UI overhaul). Stars went 751 → 836, forks 146 → 157, the open-PR count closed at zero on both repos. The shape change is the story: the simulator stopped being an app you log into and started being a service you can wire into Claude Desktop, Slack, n8n, and a TypeScript SDK.

## What Shipped

### Three Protocols in Three Days — MCP, OpenAPI, Webhook
The throughline of the week. **PR #44 AI Integration · MCP Onboarding** (merged Apr 24) put a Settings panel in front of the 8-tool MCP server that landed Apr 21 as a direct-push commit and had been sitting there unreachable. Per-client config snippets for Claude Desktop, Cursor, Windsurf, and Continue, server-side-resolved absolute paths (`backend_dir`, `mcp_script`, `python_executable`) so the JSON copies clean to the user's machine, a Neo4j liveness probe wrapped in defensive try/except so the panel renders even when the DB is down, and a regex-driven drift test that statically scans `mcp_server.py` for `@mcp.tool()` decorators and fails CI if the tool list and the API ever diverge. **PR #45 OpenAPI 3.1 + Swagger UI** (merged Apr 26) is the same architectural move for the REST surface — a 1,966-line handwritten `backend/openapi.yaml` covering ~85 paths under 13 tags, Swagger UI pinned to `swagger-ui-dist@5.17.14` from jsDelivr at `/api/docs`, and the same flavour of drift test: a regex scan of every `@<bp>_bp.route(...)` decorator across `app/api/*.py`, fully-qualified by url_prefix, equal-set-asserted against the spec. **PR #46 Completion Webhook** (merged Apr 26, ~10 minutes after #45) closes the loop on the outbound side — `Config.WEBHOOK_URL` plus a fire-and-forget daemon-thread POST in `webhook_service.py`, deduped per `(sim_id, status)` so the runner's exit-code path and the `simulation_end` event path can both call without firing twice, URL-masked to `scheme://host/***` through `GET /api/settings`, stdlib `urllib.request` with zero new dependencies, 18 offline unit tests. Slack/Discord auto-unfurl with the share-card PNG; pointed at Zapier or n8n it fans out to 5,000+ services. Three machine-readable contracts over the same simulator in three calendar days — inbound (MCP for AI editors), inbound (OpenAPI for HTTP/SDK clients), outbound (Webhook for operator environments). Symmetry closed.

### The Distribution Surfaces — Share Card and Public Gallery
**PR #42 Social Share Card** (merged Apr 22) is the first shippable Open Graph asset — a Pillow-rendered 1200×630 PNG generated server-side from the embed-summary payload, with on-disk caching keyed by render-affecting fields, served via `GET /api/simulation/<id>/share-card.png` and surfaced through a `GET /share/<id>` HTML landing page carrying `og:image`/`og:title`/`twitter:card=summary_large_image` meta tags. Pillow was already pinned ≥12.0 in the project's uv overrides — zero new dependencies. **PR #43 Public Simulation Gallery** (merged Apr 23) gave that share card a place to land. `GET /api/simulation/public` paginated 1-100, sorted by `created_at` desc, `Cache-Control: public, max-age=30`, payload built from `_build_gallery_card_payload()` reading `state.json` + `simulation_config.json` + `quality.json` + `trajectory.json` + `resolution.json` already on disk — no DB schema, no LLM calls, missing artifacts degrade to `null` instead of raising. The `/explore` Vue route renders a responsive 320px-min auto-fill card grid with quality + dominant-stance pills, a belief-split mini-bar, agent/round metadata, and paired `Open →` + `Fork this →` actions wired to the existing fork endpoint. Together with PR #44/#45/#46, this turns every published run into something the rest of the world can find, embed, fork, MCP-query, REST-query, or get notified about.

### The Onboarding Pivot — Cheap Preset, Settings Dropdown, README Slim
Apr 21 evening was a 21:07–22:52 UTC sequence that rewired both *what* models the project recommends and *how* users select them. The **Cheap preset** moved off vintage Llama/Mistral defaults to Qwen3.5-flash (default + Wonderwall), DeepSeek-v3.2 (smart), and Grok-4.1-fast (NER + `:online` for web search), and disabled chain-of-thought by default for ~3× lower latency on chat-routed models. The **Settings modal** got a Current Setup readout, a Cheap/Best/Local preset dropdown, and per-slot overrides for every model slot the backend already had (LLM, Smart, NER, Wonderwall, Embedding, Web Search) — applied server-side instead of forcing users to fork `.env`. The brittle HTML parser in url-import was replaced with an LLM-based fetcher routed through the Web Search slot. A 12-commit cluster on Apr 21 evening turned the 698-line monolithic README into a 243-line landing page with a 5-line `OPENROUTER_API_KEY=… && ./miroshark` quick start, then split everything else into nine separate `docs/` files (`INSTALL.md`, `CONFIGURATION.md`, `ARCHITECTURE.md`, `API.md`, `FEATURES.md`, `MODELS.md`, `OBSERVABILITY.md`, `MCP.md`, `CLI.md`). The new tagline is "$1 & under 10 min" — the project's first concrete cost claim, anchored by a measured ~$0.50 end-to-end run on the new Cheap preset.

### Runner Hardening — The Reliability Tax Paid
**`2fd2532`** on Apr 21 hardened the simulation runner end-to-end. `counterfactual_loader` had been imported nowhere since it shipped — branch sims were silently running as plain forks; now wired into all four loops (Twitter, Reddit, Polymarket, synchronized multi-platform). Per-round error isolation: `asyncio.wait_for` + try/except around `env.step`, `fetch_actions`, and `belief_tracker.after_round`, so a bad LLM response on round 7 stops being "sim died at round 7" and becomes "round 7 had a bad LLM response." New `MIROSHARK_ROUND_TIMEOUT` env var (default 600s) bounds round duration. `BeliefTracker` persists per-agent `BeliefState` to `belief_states_<platform>.json` after each `after_round()` call and reloads them on `__init__` — pause/resume now keeps accumulated stance instead of snapping back to profile defaults. A sibling commit (`2f08f76`) fixed the falsy-zero trap in `/counterfactual` and `/demographics` (`uid = p.get('user_id') or p.get('agent_id') or …` short-circuits to `None` when `user_id == 0`) plus the namespace mismatch where the influence leaderboard surfaces display `name` while the counterfactual endpoint indexed by `user_name`/`username`. Counterfactual Explorer goes from "works in demo, fails in production" to "works."

## Momentum Check

Last week's shiplog headline was "MiroShark stopped being a simulator" — the substrate week, the Apr 20–21 graph-memory + sibling-repo siphon double. This week's headline is the matching one: the substrate became reachable. PR #44/#45/#46 are not new engine; they are three formal contracts over the engine that already existed, each landing with a regex-driven drift test (mcp.py vs mcp_server.py, openapi.yaml vs blueprint routes) so the contracts can't rot. The same architectural beat shows up everywhere: PR #42's `_build_embed_summary_payload`, PR #43's `_build_gallery_card_payload`, PR #46's `_final_consensus_from_trajectory` — three different surfaces, all reading the same per-sim on-disk artifacts, all using the **same ±0.2 stance threshold** so a "62% bullish / 13% neutral / 25% bearish" line stays identical in the share-card PNG, the gallery card, and the webhook payload. The pace ticked from 14 PRs combined last week to 14 again this week (almost exactly), but the externalization changed: a sim's outputs are now reachable from inside Claude, inside Slack, inside any TypeScript app generated by `openapi-generator`. That's the trajectory — last week the engine got smarter, this week it got plug-in-able.
