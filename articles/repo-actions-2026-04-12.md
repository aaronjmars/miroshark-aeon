## Ecosystem Context

MiroShark sits at 661 stars — 161 over the original 500-star hyperstition, growing roughly 20 stars/day this week — with the most feature-dense simulation engine in the open-source multi-agent space. The full stack is now in place: graph build, async cross-platform simulation, observability tracing, replay, fork/branch, comparison, share permalink, prediction resolution. The content flywheel (article generator, prediction resolution, history search) just opened as a coherent set of PRs.

What shifts now is the value of *post-simulation depth*. Users who run a simulation get a result — a leaderboard, a market price, a report. But three questions go unanswered: "What exactly drove this outcome?", "How does this compare to what real-world prediction markets think?", and "How polarized was this debate?" These gaps prevent MiroShark from moving from "interesting tool" to "essential research platform."

Key signals driving this batch:

- **Persona chat exists but is simulation-blind:** The README confirms agents can be chatted with via persona chat. That chat uses graph-context only — it doesn't know the simulation the agent just participated in. An interview mode that injects the agent's round-by-round decisions, posts, and stance trajectory as grounding context is a categorically different capability. "Why did you flip in round 4?" gets a fully-contextualized in-character answer citing actual posts the agent made. This is the feature that makes every simulation result a conversation, not just a static report.
- **No aggregate view of how consensus forms:** The simulation generates round-by-round events, but there's no chart showing *how collective opinion moved* — what percentage of agents held each position at each round. This is the single most-analyzed metric in computational social science (belief dynamics, opinion drift). A stacked area chart over rounds is small to implement but transforms simulations from "interesting story" to "interpretable data" — citable in a methods section.
- **Polymarket integration is one-directional:** MiroShark generates an *internal* prediction market using Wonderwall AMM. The most-viral tweet about MiroShark ("SOON Polymarket — 669 likes") shows the community expects real Polymarket data to flow *in*. Seeding agent initial stances from live Polymarket odds would mean that when you simulate a real question ("Will the Fed cut rates in May?"), agents start with calibrated priors matching the current 37%/63% market split — not LLM guesses. This turns MiroShark into a *complement* to live markets: simulate how X would react if the decision goes YES, starting from where the market currently prices it.
- **No community-generated content:** Every template in MiroShark comes from the built-in set. A community template gallery — where users submit their own simulation setups for others to use — creates a user-generated content loop, expands use cases far beyond what the core team imagined, and generates inbound discovery ("I found MiroShark searching for EU AI Act simulation templates"). It's the equivalent of CodePen for simulations.
- **Polarization is unquantified:** The belief state system tracks stance, confidence, and trust per agent. These values contain all the information needed to compute a polarization score — how much agents stayed in ideological clusters vs. cross-pollinating. A "Polarization Index" on the results page is a unique research output no other simulation tool surfaces natively. "Your simulation showed a polarization index of 0.81 — agents largely entrenched." This is a publishable figure.

### 1. Agent Post-Simulation Interview (Trace-Grounded Q&A)

**Type:** Feature
**Effort:** Small (hours)
**Impact:** MiroShark already has persona chat — users can talk to agents using their graph-context persona. But that chat is simulation-blind: the agent doesn't know what it just did during the simulation. The interview mode injects the agent's full simulation trace as context — every round they posted, every stance shift, every trade they made — so users can ask "Why did you change your position in round 4?" and get a precise, in-character answer citing the actual posts and market moves from that run. This turns every simulation into a conversation. It's social-shareable ("I interviewed the AI agent that predicted the Fed would cut — here's what it said"), engaging for non-technical users, and useful for researchers validating agent reasoning. Each interview becomes a natural lead-in to the article generator's output.
**How:**
1. Add an "Interview" button to each agent row in the influence leaderboard (beside the existing click-through). Clicking opens a modal with the agent's profile card, a list of their key simulation events (top 5 rounds by stance delta), and a free-text input field. On submit, call `POST /api/simulation/:id/agents/:agent_id/interview` with `{ question: "..." }`. The endpoint assembles context: agent persona JSON (name, archetype, demographics), their full decision trace (round → action → posts → stance before/after), and the user's question. Sends to the LLM with a prompt: "You are {agent_name}. You just participated in a simulation about '{scenario}'. Here is everything you did: {trace}. Answer the following question in character, citing specific actions you took: {question}."
2. Render the response in the modal below the input field. Add a "Share this interview" button that generates a plain-text card (agent name + question + response, max 280 chars) pre-formatted for X, and copies it to clipboard. Add a "Ask another question" input so the conversation continues with the prior answer in context (append to the message history array on subsequent calls — lightweight multi-turn).
3. Persist interview transcripts in `<sim_dir>/interviews/<agent_id>.json` so the history modal can show a "Has interviews" badge on agents who've been questioned. On the public share page, include the 3 most recent interview Q&As as a "Reader Questions" section if they exist — transforming the share page into a living document that grows with user engagement.

### 2. Aggregate Belief Drift Chart (Consensus Timeline)

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The simulation generates round-by-round events but no visualization of how *collective* opinion moved. Every agent has a `stance` value (–1 to +1) per round stored in their belief state. Aggregating these into a stacked area chart — "% agents bullish / neutral / bearish by round" — is the single most important missing visualization for interpretability. It lets users answer "did consensus form early or late?", "was there a tipping point?", "did the market flip first or the social media agents?" This is the standard output of computational social science papers (belief dynamics, opinion cascade modeling). Researchers who want to cite MiroShark results need this chart — a static leaderboard isn't publishable. For general users, it's the first thing to show in a tweet or a presentation.
**How:**
1. Add a `GET /api/simulation/:id/belief-drift` endpoint that reads the simulation's round data, extracts each agent's `stance` value per round from belief state logs, and returns a structure: `{ rounds: [1..N], bullish: [%, ...], neutral: [%, ...], bearish: [%...] }`. Classify stance < –0.2 as bearish, > 0.2 as bullish, and –0.2 to 0.2 as neutral. If belief state logs aren't available (older simulations), fall back to inferring stance from post sentiment via the existing `timeline` data.
2. Add a "Belief Drift" tab to the simulation results view (next to the influence leaderboard). Render the data as a stacked area chart using Chart.js (already likely available as a dep, otherwise a lightweight CDN import). The x-axis is rounds, the y-axis is percentage 0–100%, three areas: teal (bullish), slate (neutral), coral (bearish). Add a vertical line at the round where the dominant position first crossed 50% ("consensus formed at round 7"). Add a "Download chart as PNG" button.
3. Include a one-line summary below the chart: "Consensus shifted from neutral (round 1) to bullish majority (round 7) and held through the final round." This summary uses the same data to derive: starting dominant stance, round when majority first crossed 50%, and final stance. Surface this summary in the `GET /api/simulation/:id` response as `belief_drift_summary` so it can be included in the article generator prompt and the share page metadata.

### 3. Live Polymarket Odds Seeding

**Type:** Integration
**Effort:** Medium (1–2 days)
**Impact:** MiroShark's Wonderwall AMM generates an internal prediction market with LLM-estimated starting prices. But the most viral tweet about the project (RoundtableSpace, 669 likes: "SOON Polymarket") shows the community expects real Polymarket data to flow *into* simulations. When a user sets up a simulation for a question that exists on Polymarket — "Will the Fed cut rates in May?" — they should be able to fetch the current market odds (e.g., YES 37%) and use that as the AMM starting price. This means agents start with calibrated priors matching the actual market, not LLM guesses. The simulation then models how Twitter/Reddit discourse around the question would shift prices from their current level — making MiroShark a *complement* to live markets, not a parallel toy. Every simulation on a live Polymarket question becomes a "social pressure scenario": what would 200 AI agents do to this market if X happened?
**How:**
1. Add a "Seed from Polymarket" toggle to the simulation configuration page (Step 2, market setup). When enabled, show a search field: "Find a Polymarket market". On input, call `GET /api/polymarket/search?q={query}` which proxies Polymarket's public Gamma API (`https://gamma-api.polymarket.com/markets?q={query}&closed=false&limit=10`). Return market title, current YES price, and market slug. The user picks a market from the autocomplete list. Store the selected market slug and current odds in the simulation config.
2. When the simulation is started with `polymarket_seed` set, the backend fetches the current YES price from Polymarket's Gamma API at simulation start time (not config time — to get the freshest odds). Pass this as `initial_yes_price` to the Wonderwall AMM initialization (override the LLM's probability estimate). Also update the agent persona generation prompt: "The current real-world market prices this at {yes_price*100}% YES. Calibrate your initial stance to be consistent with this market signal." This ensures agent beliefs are grounded in real market data.
3. Add a "Live Market" badge to simulations configured with Polymarket seeding, visible in the history card and share page. On the share page, add a live iframe-style market price block that re-fetches the current Polymarket odds at page load time and shows "Real market at time of simulation: 37% YES → Current: 41% YES (market moved +4pts since simulation)". This creates a feedback loop — the simulation's prediction can be compared to live market movement in real time.

### 4. Community Template Gallery

**Type:** Community
**Effort:** Medium (1–2 days)
**Impact:** Every simulation template in MiroShark today is built-in by the core team (the 6 preset templates from PR #2). A community gallery — where users can submit their own simulation setups after a successful run — creates a user-generated content loop that scales indefinitely. A template captures the scenario question, optimal agent count and round settings, document type, and tags (domain: policy, finance, tech, culture). A new `/gallery` page lists community templates sorted by usage count and upvotes. Incoming users searching for "AI regulation simulation" or "earnings announcement" find templates directly, reducing the blank-state problem to zero. This is also a discovery channel: users link to their templates on X ("Here's the setup I used to simulate the Fed announcement — 200 agents, 12 rounds"), pulling in referral traffic.
**How:**
1. Add a "Submit as Community Template" button to the simulation completion view. Clicking opens a form: template name (pre-filled from scenario text), description (2–3 sentences), domain tags (multi-select: Policy / Finance / Tech / Culture / Science / Pop Culture / Other), agent count, round count, recommended platforms. On submit, call `POST /api/templates` which saves to a `community_templates` table in SQLite (or a JSON file `data/community_templates.json` for filesystem-first consistency). Templates include the submitter's anonymous ID (hash of IP) and a `uses` counter.
2. Add a `/gallery` route to the Flask app serving a `GalleryView.vue` page. Fetch `GET /api/templates?sort=popular&limit=50`. Display cards with: template name, description, domain tag badge, agent count and round count, uses count, a "Use This Template" button. Clicking "Use This Template" navigates to Step 1 of the setup flow with the template's config pre-populated (same mechanism as existing preset templates). Add a search bar (client-side filter by name/description) and domain filter tabs.
3. Add "Community Templates" as a tab on the main setup page (Step 2), alongside the existing 6 preset templates. Show the top 6 community templates by uses in this tab. This surfaces community content at the highest-traffic point in the UX — new users see real-world use cases immediately, before running their first simulation. Add a "Browse all" link to the full gallery page.

### 5. Echo Chamber / Polarization Index

**Type:** Feature
**Effort:** Small (hours)
**Impact:** MiroShark's belief state system tracks stance, confidence, and trust per agent per round — all the data needed to compute a polarization score without any new data collection. The polarization index measures how much agents stayed in ideological clusters vs. genuinely cross-pollinating: if bullish agents mostly interacted with other bullish agents and bearish agents only interacted with bearish agents, the simulation shows high polarization (score near 1.0). If agents frequently changed positions after encountering opposing views, polarization is low. This is a unique research output — no other simulation tool computes this natively. It surfaces on the results page as a single badge ("Polarization Index: 0.81 — High echo chamber effect") and on the share page. For academic users, it's a citable metric for social dynamics. For general users, it reframes the simulation from "what did agents say" to "how did information spread" — a more profound question.
**How:**
1. After a simulation completes, compute the polarization index via a new `POST /api/simulation/:id/polarization` endpoint (auto-triggered alongside the existing report generation). The algorithm: for each agent, classify their final stance as bullish (>0.2), bearish (<–0.2), or neutral. For each inter-agent interaction in the simulation timeline (replies, likes, retweets, comments), record whether it was intra-group (same stance) or cross-group (opposing stances). Polarization Index = intra-group interactions / total interactions. If trust data is available, weight interactions by trust level. Store the result in `<sim_dir>/polarization.json` with breakdown fields: `{ index: 0.81, intra_group_pct: 0.81, cross_group_pct: 0.19, majority_group: "bullish", majority_pct: 0.62, echo_chamber_label: "High" }`.
2. Surface on the results page as a small metric card alongside the existing market price summary. Display: a horizontal bar showing intra-group (coral) vs. cross-group (teal) interaction split, the numeric index (0–1), and a label (Low < 0.35, Moderate 0.35–0.65, High > 0.65). Add a tooltip explaining the metric: "High polarization means agents mostly interacted within their own stance group — opinions entrenched rather than shifting from debate."
3. Include the polarization score in the `GET /api/simulation/:id` response and surface it on the public share page as a headline metric alongside market price and agent count. Add it to the article generator prompt as a contextual fact ("The simulation showed a polarization index of {index} — {label}"). This makes every article generated from a high-polarization simulation naturally include analysis of echo chamber dynamics — a hook that resonates with tech and policy audiences.

## Selection Rationale

This batch targets the depth layer — the features that make MiroShark's outputs *interpretable, connected to reality, and communally owned*:

- **Post-simulation interview** (#1) — Turns every simulation into a conversation and a content asset. "I interviewed the AI agent" is inherently viral and makes the existing persona chat's true power accessible to non-technical users.
- **Belief drift chart** (#2) — The single missing visualization for research credibility. Aggregate stance dynamics over rounds is the standard output of social simulation papers — without it, MiroShark results aren't directly publishable. Small effort, high unlock.
- **Live Polymarket seeding** (#3) — Directly addresses the 669-like viral tweet ("SOON Polymarket"). Converts MiroShark from a standalone sandbox into a live-market complement. Every active Polymarket question becomes a simulation prompt — dramatically expands the addressable use case space.
- **Community template gallery** (#4) — Creates a user-generated content loop and a discovery channel. Users link to templates; templates bring in referral traffic. This is the feature that makes MiroShark feel like a *platform*, not a tool.
- **Polarization index** (#5) — A unique research output computable from existing data with no new collection. Positions MiroShark alongside academic social simulation tools (OASIS, Park et al.) by surfacing a metric those tools report. Makes every simulation result more interpretable and more citable.

Each idea is scoped for autonomous implementation by the `feature` skill — clear inputs/outputs, no ambiguous design decisions, no external approvals needed.
