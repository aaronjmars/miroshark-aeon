## Ecosystem Context

Today's two merges closed a specific chapter: the discovery-and-reusability pair. `GET /api/surfaces.json` is how a machine finds out what's available on a deployment. `GET /api/simulation/<id>/clone.json` is how it gets the inputs to reproduce or fork any published sim. Together they're the MCP primitive shape — `listTools` / `callTool` — transplanted to the MiroShark surface layer. That pair is now live on `main`.

The 12 named integrators in ECOSYSTEM.md are now operating against a platform with 28 publish-gated surfaces, a machine-readable catalog, and a clone endpoint. Three structural gaps remain visible at this new baseline:

**The agent data gap.** Every published sim exposes what the swarm concluded (signal.json, volatility, peak-round) and how belief evolved (chart.svg, sparklines). None expose *who* was in the debate in machine-readable form. `transcript.md` has agent names, roles, and initial beliefs but requires parsing Markdown. `GET /api/simulation/<id>/agents.json` exports the full agent roster as a structured JSON array — the 29th publish-gated surface and the first to focus on the debate participants rather than the outcome. Re-eligible from May-24 #4.

**The query gap.** The gallery HTML at `/explore` has interactive filters; `/api/simulation/public` (the pre-existing gallery JSON) returns an unfiltered index. Neither serves the pattern a research script or integration actually needs: "find me Bullish sims with confidence > 75% on crypto topics, sorted by most recent." `GET /api/search.json?q=&consensus=&min_confidence=&sort=` is the query-driven companion to the full-index gallery endpoint. Re-eligible from May-24 #5.

**The operator identity gap.** There are 12+ named integrators and an unknown number of operators running published sims, but no per-operator view exists. An operator who has published 20 sims has no shareable URL for their work. `GET /api/operator/<name>/profile` + `/profile/<name>` page creates the per-operator identity layer — all their published sims, aggregate consensus distribution, total views. Every share page becomes a backlink to the operator's profile. Re-eligible from May-24 #3.

Two structural gaps beyond the re-eligible set:

**The engagement visibility gap.** Gallery.json returns published sims ordered by recency. There is no view of what is *actively being read* — which sims are accumulating surface views right now. `GET /api/gallery/trending.json` returns the top-10 most-viewed public sims in a rolling 7-day window, sorted by `surface_views`. The `surface_stats` counters per sim already track this; the trending surface aggregates and ranks them. Enables "What's popular" widgets for directories, the Aeon digest, and AntFleet's benchmark discovery loop.

**The per-sim analytics gap.** The platform-level `/api/stats` endpoint returns aggregate surface views across all sims. There is no per-sim breakdown: which surfaces of *this* published sim are being consumed, and how often? `GET /api/simulation/<id>/surfaces/stats.json` exposes the existing per-sim `surface_stats` counters publicly — which endpoints have been hit, with what frequency. Operators see whether their published sim's volatility surface is being consumed (AntFleet benchmark jobs) vs. the badge SVG (social sharing). No new tracking needed; this is a read path over already-stored counters.

### 1. Operator Profile Page

**Type:** Community
**Effort:** Medium (1–2 days)
**Impact:** Every operator who has published a dozen sims has no shareable URL for their body of work. Every share page links to a specific sim but not to its creator. `/profile/<operator_name>` fills this: a per-operator gallery of all their published sims, plus a summary card (total sims, consensus distribution, avg quality, total views). `GET /api/operator/<name>/profile` backs it. Every `/share/<id>` page gains a "by [operator_name]" attribution link, propagating backlinks organically from published sims to the profile. Researchers can follow a specific operator's output without manual search. Re-eligible from May-24 #3.

**How:**
1. Add `backend/app/services/operator_service.py` (~160 LoC, pure stdlib `json` + `os`). `get_operator_sims(operator_name, sim_root) -> list[dict]`: scan all sim dirs, filter `is_public=True` and `operator.lower() == operator_name.lower()` (case-insensitive), sort by `created_at` desc, cap at 100. Per sim: `{sim_id, scenario_title: str[:80], created_at, direction, bullish_pct, neutral_pct, bearish_pct, quality_health, total_rounds, surface_views: int}`. `build_operator_profile(operator_name, sims) -> dict`: `{operator, total_sims, consensus_distribution: {bullish_count, neutral_count, bearish_count}, avg_quality_tier: str (mode of quality_health values), total_surface_views, sims}`. Unknown operator → `total_sims: 0`, empty sims list (not 404). Add `GET /api/operator/<name>/profile` (no auth; public). Sanitize `operator_name`: regex `[a-zA-Z0-9_.-]{1,80}`, 400 on invalid. Add 10 offline unit tests: unknown operator → `total_sims: 0`, 3 published sims → `total_sims: 3`, unpublished excluded, `consensus_distribution` counts correct, `total_surface_views` sum correct, 100-sim cap respected, case-insensitive name match, `avg_quality_tier` non-empty string, valid name → 200, special chars in name → 400.
2. Add `OperatorProfileView.vue` in `frontend/src/views/`. Route `/profile/:operatorName` (add to router). On mount: validate `operatorName` regex; redirect to `/explore` with error toast if invalid. Fetch `GET /api/operator/<operatorName>/profile`. Layout: header card with operator name + stats row (total sims, consensus mini-bar in belief colors, avg quality chip, total views). Below: sim grid using existing `SimCard` component. Empty state for unknown or zero-sim operators. "Share this profile" button copies `/profile/<name>` URL to clipboard. Add `getOperatorProfile(operatorName)` to `frontend/src/api/simulation.js`.
3. In the share page (`/share/<id>`), add "by [operator_name]" attribution text linking to `/profile/<operator_name>` — visible only when the `operator` field is non-empty in state. Add `GET /api/operator/<name>/profile` to `docs/API.md` under Gallery & Discovery with a `OperatorProfileResponse` + `OperatorSimSummary` schema. Add "Operator Profile" to `docs/FEATURES.md`. Add to `openapi.yaml`. Zero new deps.

### 2. Agent Persona Export JSON

Researchers comparing swarm convergence across sim configurations — do financial-analyst agents align faster than retail traders? — get the answer with one curl command instead of Markdown regex. AntFleet's benchmark pipeline can cross-reference agent pool composition with outcome quality. Re-eligible from May-24 #4.

Add `GET /api/simulation/<id>/agents.json` (publish-gated; 404 `{"error": "no agent data"}` when empty). `Cache-Control: public, max-age=300`. Content-Disposition: `inline; filename=miroshark-<id12>-agents.json`. Extend `SURFACE_KEYS` + `surface_stats` with `agents_json`. Add 10 offline unit tests in `test_unit_agent_export.py`: published sim → list of dicts with required keys, unpublished → 403, empty agents → 404, `total_agents` matches list length, `final_stance` one of Bullish/Neutral/Bearish, `rounds_participated` is int ≥ 0, `sim_id` in payload matches route param, `completed_at` is ISO-8601 string, response is JSON-serialisable, `surface_stats` increment called.

In `EmbedDialog.vue`, add a "🤖 Agent Roster" section (publish-gated; positioned after the Sparklines section). Below: collapsible agent table showing name chip (20-char truncated), role tag, final stance colored chip (green/gray/red).
3. ### 3. Simulation Search JSON API

**Type:** Integration
**Effort:** Small (hours)
**Impact:** The gallery HTML at `/explore` has interactive filters; `/api/simulation/public` returns a full unfiltered index. Neither serves the query-driven pattern an LLM agent, research script, or integration actually needs: "give me all Bullish sims with confidence > 75% on crypto, sorted by most recent." `GET /api/search.json?q=&consensus=&min_confidence=&sort=&limit=` is the query-filtered companion to the full index. AntFleet's benchmark pipeline can programmatically find the highest-confidence sims in a topic area. Aeon's self-improvement loop can query for sims on a specific topic to inform the weekly digest. External dashboard builders get a single curl command for a filtered result set. Re-eligible from May-24 #5.

**How:**
1. Add `backend/app/api/search.py` (new blueprint, ~130 LoC, pure stdlib `json` + `os` + `re`). `GET /api/search.json` — query params: `q` (case-insensitive substring match against `scenario_title` and `scenario_context`), `consensus` (`bullish`/`neutral`/`bearish`; filters by `direction`), `min_confidence` (float; filters sims where `confidence_pct ≥ value`), `max_confidence` (float), `sort` (`recent` default / `confidence_desc` / `views_desc`), `limit` (int, 1–100, default 20). Reads from same sim_root scan as gallery service. Filter order: is_public + status=completed → q → consensus → confidence range → sort → limit. Response: `{results: list[{sim_id, title, created_at, direction, confidence_pct, quality_health, total_rounds, surface_views}], total_matching: int, query: {q, consensus, min_confidence, max_confidence, sort, limit}}`. `Cache-Control: public, max-age=30`. Invalid `consensus` value → 400 `{"error": "consensus must be bullish, neutral, or bearish"}`. Register blueprint in `backend/app/__init__.py`. Add 10 offline unit tests in `test_unit_search.py`: empty query returns all public completed sims up to limit, `q=bitcoin` matches title substring case-insensitively, `consensus=bullish` filters correctly, `min_confidence=75` excludes lower-confidence sims, `sort=confidence_desc` ordered correctly, `limit=5` caps results, unpublished/incomplete excluded, `total_matching` correct after filter, combined params stack, invalid consensus → 400.
2. In the `/explore` gallery page `<head>`, add `<link rel="search" type="application/json" href="/api/search.json">`. In the `/explore` gallery body, add a `[JSON ↗]` link near the active search bar that opens `/api/search.json?q={currentQuery}` when a search term is active — zero new views, just a link. Add `searchSimulations(params)` to `frontend/src/api/simulation.js`.
3. Add `GET /api/search.json` to `docs/API.md` under Gallery & Discovery with `SearchResults` schema and all query param definitions. Add to `openapi.yaml` under Gallery & Discovery. Add "Programmatic Simulation Search" to `docs/FEATURES.md`. Zero new deps.

### 4. Gallery Trending JSON

**Type:** Integration
**Effort:** Small (hours)
**Impact:** Gallery.json (`/api/simulation/public`) returns published sims in chronological order — what was published most recently. There is no view of what is *actively being read*: which published sims are accumulating the most surface views right now. `GET /api/gallery/trending.json` fills this gap: top-10 most-viewed public sims in a rolling 7-day window (based on `published_at`), sorted by `surface_views` desc, with fallback to all-time if fewer than 5 sims were published in the window. The `surface_stats` counters per sim already track total surface views; this endpoint aggregates and ranks them. Enables directory builders to surface "popular simulations" widgets, gives Aeon a signal for what's resonating, and lets AntFleet find high-engagement sims as benchmark candidates without scanning the full gallery. Net-new.

**How:**
1. Add `backend/app/api/gallery_trending.py` (new blueprint, ~110 LoC, pure stdlib `json` + `os` + `datetime`). `GET /api/gallery/trending.json` — no query params. Scans sim_root: filter `is_public=True` + `status="completed"`. For 7-day window: filter `published_at ≥ now - 7d` → sort by `surface_views` desc → take top 10. If fewer than 5 results in window: fall back to all-time top-10 by `surface_views`, adding `fallback_reason: "sparse_window"` to the response envelope. Per sim entry: `{sim_id, title: str[:100], direction, confidence_pct, quality_health, surface_views: int, published_at: str}`. Response envelope: `{window_days: 7, window_start: ISO-8601, count: int, fallback_reason: str | null, trending: list[dict]}`. `Cache-Control: public, max-age=300` (5-min cache — views change on each surface hit). `ETag: trending-{window_start[:10]}-{total_views_sum}`. Register blueprint in `backend/app/__init__.py`. Add 8 offline unit tests in `test_unit_gallery_trending.py`: returns valid envelope with all required fields, `count` matches list length, unpublished/incomplete excluded, sorted by `surface_views` desc, `window_days: 7` in response, fewer than 5 in window → `fallback_reason` non-null, ETag header present, empty sim_root → valid response with `count: 0`.
2. In the `/explore` gallery page, add a "Trending" tab or toggle alongside the existing filter options that fetches `GET /api/gallery/trending.json` and renders the top-10 in the existing `SimCard` grid layout. Fallback state: if `fallback_reason` is set, show "(All-time top sims)" subtitle. Add `getTrending()` to `frontend/src/api/simulation.js`.
3. Add `GET /api/gallery/trending.json` to `docs/API.md` under Gallery & Discovery with a `TrendingResponse` schema defining the 7-day window and fallback logic. Add to `openapi.yaml`. Add "Gallery Trending" to `docs/FEATURES.md` under Gallery & Discovery. Zero new deps.

### 5. Per-Sim Surface Engagement JSON

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The platform-level `/api/stats` endpoint returns aggregate `total_surface_views` across all sims. No per-sim breakdown exists: which surfaces of *this particular published sim* are actually being consumed, and how often? `GET /api/simulation/<id>/surfaces/stats.json` exposes the existing per-sim `surface_stats` counters (tracked by `surface_stats.py` already on every surface hit) in a public read endpoint. An operator publishing sim `abc123` can see that its `volatility` surface has 47 views (AntFleet benchmark jobs) while its `share-card.png` has 3 (organic sharing). AntFleet can verify their benchmark pipeline is hitting the right surfaces on each sim. Aeon can monitor surface usage patterns across its feature builds without reading internal state files. No new tracking needed — this is a read path over counters already being written. Net-new.

**How:**
1. In `backend/app/services/surface_stats.py`, add `read_sim_surface_stats(sim_id, sim_root) -> dict | None`: reads the per-sim surface stats file (the file already written by `increment_surface_view`) and returns `{sim_id, total_surface_views: int, surfaces: [{key: str, endpoint: str, views: int}]}` for all keys in `SURFACE_KEYS`. Returns `None` if the sim is unpublished. Surfaces with 0 views are included (so integrators know what's available, not just what's been accessed). Sort by `views` desc. Add `GET /api/simulation/<id>/surfaces/stats.json` to `backend/app/api/simulation.py` (publish-gated; no auth; `Cache-Control: public, max-age=60`; 404 `{"error": "simulation not found or not published"}` when None). Do NOT count this endpoint hit as a surface view (avoid self-referential inflation — skip the `increment_surface_view` call for this route). Add 8 offline unit tests in `test_unit_sim_surface_stats.py`: published sim with stats → returns dict with all required fields, unpublished sim → 404/None, `total_surface_views` equals sum of all surface views, all SURFACE_KEYS present in response even if 0 views, sorted by views desc, zero-views surfaces included, `sim_id` in response matches param, response JSON-serialisable.
2. Add `getSimSurfaceStats(simId)` to `frontend/src/api/simulation.js`. In `EmbedDialog.vue`, add a "📊 Surface Usage" section (publish-gated; visible only to the operator — gate via a URL param or simply make it public like other surfaces). Layout: total views badge + a compact bar-chart-style list (surface key + mini-bar + count) for top-8 surfaces by views. "Copy surface stats URL" button.
3. Add `GET /api/simulation/<id>/surfaces/stats.json` to `docs/API.md` under Platform with a `SimSurfaceStats` schema. Document the `total_surface_views` definition (sum of all per-surface counters for this sim). Note that the endpoint itself is not counted to avoid inflation. Add "Per-Sim Surface Engagement" to `docs/FEATURES.md`. Add to `openapi.yaml`. Zero new deps.

## Selection Rationale

Today's batch reflects the baseline after 28 surfaces, 12 named integrators, and the discovery+reusability pair landing atomically. The analytical quadrant is closed. The catalog and clone surfaces just shipped. What's left is the layer *around* the sims — who runs them, how to find them by query, how they're being consumed.

- **Operator Profile** (#1) — Re-eligible from May-24. Three weeks unbuilt. The platform now has 12+ named integrators and an unknown number of operators running dozens of published sims, but no per-operator identity layer. Every share page is a dead end for discoverability of the operator's wider body of work. Medium effort, but the organic backlink network effect scales with star count — at 1,222 stars it's more valuable than it was at 1,194.
- **Agent Persona Export JSON** (#2) — Re-eligible from May-24. Sparklines give agent *trajectories*; personas.json gives agent *identities* — who was in the debate. These are different surfaces. Researchers comparing convergence rates across persona configurations need both. The 29th publish-gated surface; follows the same stdlib-only pattern as the 28 before it.
- **Simulation Search JSON** (#3) — Re-eligible from May-24. Gallery.json is a full index (pre-existing); search.json is query-driven. The distinction matters for LLM agents and research scripts. AntFleet's benchmark discovery loop — find the highest-confidence crypto sims published in the last month — can't be served by a full-index download. One filtered endpoint collapses N full-index scans into one curl command.
- **Gallery Trending JSON** (#4) — Net-new. The surface_stats counters already exist; trending is a ranking over them. This is the "what's actually popular" view missing from a chronological gallery. Enables the Aeon digest to cite engagement rather than just recency, and gives directory builders a signal beyond "most recent." ~110 LoC, no new tracking infrastructure.
- **Per-Sim Surface Engagement JSON** (#5) — Net-new. The tracking is already happening; this endpoint exposes it. Operators and integrators currently have no visibility into which surfaces of a given sim are being hit. AntFleet can verify their benchmark pipeline is consuming the right surfaces on each run without reading internal state files. A transparency layer that costs ~80 LoC of read-path code over already-written counters.
