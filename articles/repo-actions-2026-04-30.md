## Ecosystem Context

The past week sealed the substrate arc. MiroShark now has four machine-readable contracts (MCP / OpenAPI / webhook / transcript-JSON), four share surfaces (PNG / GIF / Markdown / RSS), and a verified-prediction gallery. Every simulation produces artefacts that land in every distribution channel the research and DeFi audiences use.

The WWII "talkie" demo (pre-1930 data, 34 agents arguing about the outbreak of WWII on simulated Reddit/Polymarket) was the strongest viral moment of the week — not a DeFi protocol sim, but a *historical what-if* framed as an AI time-travel experiment. The per-slot endpoint override (PR #59) is what made it possible at the infrastructure level: individual agent slots can now point at `api.talkie.ai` or any time-bounded model. But there is no product concept around it. The UX to run a historical sim is currently: set five env vars, pick the Wonderwall preset, manually write a period-accurate scenario. That's too much friction to replicate the demo.

Four signal threads drive this batch:

- **Historical mode capitalisation:** The WWII demo proved the market. The per-slot infrastructure is live. The missing piece is a UX layer: a date-anchor field, auto-selection of the historical Wonderwall endpoint if configured, and curated templates (Cuban Missile Crisis, 1929 Crash, Moon Landing, 9/11). A `historical_anchor` field in sim config costs almost nothing to add — and makes every future "what if it was 1989?" demo a 2-click flow instead of a 5-env-var setup.
- **Scientific credibility gap:** MiroShark's quality score is behavioural (engagement, coherence, diversity). It says nothing about whether the agents are *reasoning correctly* — whether a "bearish" agent is actually citing sound risk factors or just producing confident-sounding noise. An LLM-as-judge audit panel, triggered on demand and backed by the just-shipped transcript.json, closes that gap. The transcript was explicitly designed for "LLM-as-judge pipelines" (README) — the pipeline now needs an entry point.
- **Reproducibility as research currency:** A single sim result is an anecdote. Running the same scenario three times and measuring variance ("67% bearish ± 5% across 3 runs") is a finding. With the cost reductions from PR #52 (30–40% cheaper sims), running three replicates of a Cheap preset scenario now costs ~$0.50 total. The infrastructure for parallelism is already there (the simulation runner is stateless); the product concept just isn't.
- **Analyst-grade data export:** Pandas, Excel, Tableau, and R — the tools academic and quantitative researchers reach for automatically — consume CSV or JSONL, not Markdown or GIF. The belief trajectory data is in `trajectory.json` already. A per-round CSV export (round, bullish_pct, bearish_pct, neutral_pct, agent_count, quality_score) is one stdlib-csv render away from completing the data-consumer surface alongside transcript.json.

### 1. Historical Simulation Mode

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** The WWII "talkie" demo was the single strongest tweet of the week. The per-slot endpoint override (PR #59) already lets you point agent slots at a time-bounded model. But reproducing the demo requires manual env-var configuration and a period-accurate scenario you write yourself — the blank-page friction that killed first-sim conversion is doubly worse for historical sims where the user doesn't know what "grounded" period-accurate language looks like. A `historical_anchor` field (year picker, optional) added to the simulation config and rendered in the New Sim form changes the UX to: pick year → pick historical template → set Wonderwall endpoint → run. A pre-built template library of 6–8 historical moments (1929 Crash, WWII Outbreak, Cuban Missile Crisis, Moon Landing, 1987 Black Monday, Dotcom Bubble, 2008 Lehman, 9/11) with period-accurate scenario text and agent briefings eliminates the blank-page problem entirely. In the SimulationView and share card, a small "Historical (1939)" badge communicates the temporal framing to viewers who see the result. On the `/explore` gallery and `/verified` page, a category filter "🕰 Historical" isolates time-anchored sims from present-day event forecasting. Makes every "what if it was [year]?" moment a 2-click flow instead of a 5-env-var setup — and creates a compelling content format that continues performing long after the original WWII tweet fades.
**How:**
1. Add `historical_anchor` (nullable integer year, 1800–2024) to `SimulationConfig` dataclass. Persist it in `simulation_config.json`. In `GET /api/simulation/:id`, expose it in the response. Add a "Historical simulation" toggle in the New Sim form (`SimulationRun.vue` or equivalent): when toggled on, show a year input (1800–2024 range, defaults to 1939) + a "Browse historical templates →" button. If `WONDERWALL_BASE_URL` is set and the toggle is on, auto-check the "Use Wonderwall endpoint" box and show a banner: "Historical agents will use your Wonderwall endpoint — set it to a time-bounded model." Add `historical_anchor` + 6–8 historical scenario objects to `frontend/src/data/scenario_templates.json` (new `category: "Historical"` with `anchor_year` field). Add `historical_anchor` to `openapi.yaml` `SimulationConfig` schema — drift test passes.
2. In `SimulationView.vue`: if `historical_anchor` is set, show a "🕰 Historical (YYYY)" badge next to the scenario title — same styling as the "📍 Verified" badge introduced in PR #47. Pass `historical_anchor` through to `_build_gallery_card_payload()` so it surfaces in gallery cards, share cards, RSS feed entries, and webhook payloads. In the gallery card, add a "Historical" category label to the existing quality/stance pills row. Add `historical_anchor` to `_build_embed_summary_payload()` so the share-card PNG can include a bottom-left year watermark (e.g. "1939") in a small serif font — visually distinct from the present-day forecasting format.
3. Add a `GET /api/simulation/templates?category=historical` path alongside the existing template endpoint. Add "Historical" as a filter option on `/explore` and `/verified` (a chip in the filter row, renders only when `has_historical: true` sims exist). Log historical sim usage in `state.json` under a `mode: "historical"` field — feeds into future analytics and the engagement leaderboard (still unbuilt from Apr 24). Update README Features table with a "Historical Mode" row and the Use Cases section with a dedicated "Historical what-if" example pointing at the talkie demo.

### 2. LLM-as-Judge Audit Panel

**Type:** Feature / Research
**Effort:** Small (hours)
**Impact:** The quality score (engagement / coherence / diversity / variance) is behavioural — it measures how agents *interact*, not whether they're reasoning correctly. A DeFi researcher asking "did the bearish agents actually cite real protocol risk factors or just vibes?" can't answer that from quality.json. The transcript was explicitly designed for "LLM-as-judge pipelines" (README) but there's no in-product judge. `GET /api/simulation/:id/audit` sends transcript.json to a configurable judge model (any OpenRouter model, defaults to a fast/cheap option like `gpt-4o-mini` or `qwen/qwen-2.5-72b-instruct` from the Cheap preset) with a structured prompt: "Rate each agent post on reasoning quality (1–5), factual groundedness (1–5), and stance consistency across rounds (1–5). Return JSON." The response surfaces in SimulationView as an "Audit" tab: a per-agent table (name, archetype, avg reasoning, avg factual, avg stance consistency, sample quote from highest-scoring post). For the academic and quant audience evaluating whether to cite a MiroShark result, this is the difference between "a simulation ran" and "the simulation passed a reasoning quality bar." Operator-triggered (never auto-run — the judge call adds cost), cached in `<sim_dir>/audit.json` once run.
**How:**
1. Add `GET /api/simulation/:id/audit` (publish-gated, same 403 as share card). Reads `transcript.json` (already generated by PR #57). If `audit.json` already exists in sim_dir, returns the cached result immediately with `{cached: true, audited_at}`. Otherwise: reads `Config.LLM_API_KEY` + `Config.LLM_BASE_URL` (same env vars the simulation itself uses, so no new secrets). Builds a structured prompt: system = "You are a rigorous academic evaluator of multi-agent simulation quality", user = "Here is a simulation transcript: [first 12,000 chars of transcript.json as JSON]. Rate each agent (by name) on: reasoning_quality (1–5, cites specific evidence vs vague assertions), factual_groundedness (1–5, uses accurate domain facts vs hallucinated details), stance_consistency (1–5, coherent viewpoint across rounds vs arbitrary flip-flopping). Return: `{agents: [{name, archetype, reasoning_quality, factual_groundedness, stance_consistency, highlight_quote}], overall_grade: A/B/C/D, summary: one-sentence synthesis}`." Sends to LLM via existing `wonderwall_client` or direct OpenRouter call. Writes response to `<sim_dir>/audit.json`. Returns `{available: true, agents: [...], overall_grade, summary, audited_at, cached: false}`. Add `AUDIT_MODEL` to `.env.example` (optional, defaults to cheapest available model).
2. In `SimulationView.vue`, add an "Audit" tab alongside the existing tabs (Report, Interaction, Demographics, Observability). The tab shows a "Run quality audit →" button when `audit.json` doesn't exist — clicking triggers `POST /api/simulation/:id/audit` and shows a spinner. Once loaded (cached or fresh): overall grade pill (A=green, B=blue, C=yellow, D=red) + one-sentence summary + per-agent table (sortable by any column, click a row to expand the highlight quote). Add a small "🔬 Audit" badge to SimulationView's header when `audit.json` exists — makes it discoverable from the sim list without entering the tab. Add `getAuditUrl()` + `runAudit()` helpers to `frontend/src/api/simulation.js`.
3. Surface `overall_grade` in `_build_gallery_card_payload()` as an optional field (null when not audited). In gallery cards, show a faint "🔬 A" or "🔬 B" badge alongside the quality health pill when the audit grade is present. Add `GET /api/simulation/:id/audit` to `openapi.yaml` under Analytics with a new `SimulationAudit` schema — drift test passes. Add a "Quality Audit" row to `docs/FEATURES.md` under Observability. Update README Features table. This is the credibility layer that turns a simulation result from "interesting output" into "citable finding" for the academic audience.

### 3. Batch Rerun / Reproducibility Badge

**Type:** Feature / Research
**Effort:** Medium (1–2 days)
**Impact:** A single sim result is an anecdote. Running the same scenario three times and showing "67% bearish ± 5% across 3 runs" is a finding. With the cost reductions from PR #52 (~30–40% cheaper) and the Cheap preset capped at 30–40 default rounds, three replicates now cost ~$0.50 total — within the "$1 & under 10 min" value proposition. Right now there is no way to measure whether a simulation result is stable or random; this is the #1 scientific credibility objection for the academic and quant audience. `POST /api/simulation/:id/batch-rerun?n=3` spawns N copies of the simulation with identical config but different random seeds, waits for all to reach `completed`, then writes a `<sim_dir>/reproducibility.json` summarising the distribution across runs: `{n_runs, consensus_mean: {bullish, bearish, neutral}, consensus_std: {bullish, bearish, neutral}, quality_mean, quality_std, run_ids: [...]}`. Frontend: a "Run reproducibility check (N=3)" button in SimulationView that triggers this and shows a spinner; once complete, a "Reproducibility" card showing a mini belief-bar-with-error-band and the run IDs as links. A "🔄 Reproducible (n=3)" or "⚠ Varies (n=3)" badge in the gallery card. Makes "how stable is this result?" a first-class question the product can answer.
**How:**
1. Add `POST /api/simulation/:id/batch-rerun` (publish-gated, admin-token gated — this spawns N sims so it needs the same auth as `/outcome`). Accept `?n=2|3|5` (cap at 5 to bound cost). For each i in range(n): call the existing `SimulationManager.create_simulation()` with the same config dict as the source sim (deep-copied from `simulation_config.json`) but with `random_seed: random.randint(0, 2**31)` injected. Start each run. Return immediately with `{batch_id: uuid, run_ids: [...], status: "started"}`. Write `<source_sim_dir>/batch_runs.json` with the `{batch_id, run_ids, n, started_at}` record so the source sim knows it has associated batch runs. Add `GET /api/simulation/:id/batch-rerun` to check status: reads each run_id's state.json, returns `{complete: bool, run_results: [{run_id, status, consensus}], reproducibility: null|{...}}`. When all runs reach `completed`, compute and write `<source_sim_dir>/reproducibility.json`: mean/std of bullish/bearish/neutral and quality_health across runs.
2. In `SimulationView.vue`, add a "Reproducibility" section to the Metrics card (below quality health). If `reproducibility.json` exists: show mean ± std belief bars (using the existing bar component, add thin error-cap lines via CSS), quality mean ± std, "Across N runs" label, and run IDs as links. If `batch_runs.json` exists but runs are still in progress: show a progress bar ("2/3 runs complete"). If neither exists: show a "🔄 Check reproducibility (N=3, est. $0.40)" button that triggers the POST. Poll `GET /api/simulation/:id/batch-rerun` every 15s while runs are active. Add `getBatchRerunStatus()` + `startBatchRerun()` helpers to `frontend/src/api/simulation.js`.
3. In `_build_gallery_card_payload()`: if `reproducibility.json` exists, add `reproducibility: {badge: "stable"|"variable", n: int, bullish_std: float}` to the payload. In gallery cards: a "🔄 Stable (n=3)" chip (green) when `bullish_std < 5` and bearish_std < 5`, or "⚠ Varies (n=3)" (amber) otherwise. Add both endpoints to `openapi.yaml` under Simulation Lifecycle — drift test passes. Add `POST /api/simulation/:id/batch-rerun` to the admin endpoint allowlist alongside `/outcome`. Add a "Reproducibility" row to `docs/FEATURES.md` under Analytics. Update README with "Batch Rerun" in Features table.

### 4. Belief Trajectory CSV / JSONL Export

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** Pandas, Excel, Tableau, R, and Observable — the analyst's default toolkit — consume CSV or JSONL natively. The transcript.json covers reasoning (agent posts per round); it does not cover the quantitative trajectory (per-round belief percentages, agent participation counts, quality scores). `GET /api/simulation/:id/trajectory.csv` gives analysts a file they can `pd.read_csv()` directly: one row per round, columns `round,bullish_pct,bearish_pct,neutral_pct,participating_agents,quality_score,round_timestamp`. `GET /api/simulation/:id/trajectory.jsonl` gives one JSON object per line for streaming pipeline consumers. Both are publish-gated, stdlib-only (csv module), zero new dependencies. This is the fifth data-consumer surface: PNG (share card, visual) → GIF (replay, motion) → Markdown (transcript, prose) → JSON (transcript, SDK) → **CSV/JSONL (trajectory, analytics)**. The quant audience evaluating MiroShark as a research data source will `curl /trajectory.csv | python3 -c "import pandas as pd, sys; df=pd.read_csv(sys.stdin); print(df.describe())"` as their first action — this is the surface that tells them the answer in <10 seconds.
**How:**
1. Add `GET /api/simulation/:id/trajectory.csv` and `GET /api/simulation/:id/trajectory.jsonl`. Both publish-gated (same 403 as share card + transcript). Read `trajectory.json` (per-round agent states), `quality.json` (overall scores — use per-round quality if available, else `null`). For each recorded round: compute `bullish_pct`, `bearish_pct`, `neutral_pct` using the existing ±0.2 stance threshold (same as every other surface — add a `_compute_stance_split(agents, threshold=0.2)` helper to `utils.py` and refactor the other three callers to use it). `participating_agents` = count of agents with at least one post in that round. `round_timestamp` = `null` for now (trajectory.json doesn't currently record wall-clock time per round — add it as a TODO comment). CSV: write with `csv.DictWriter`, `Content-Disposition: attachment; filename="trajectory-{sim_id[:8]}.csv"`, `Content-Type: text/csv`. JSONL: one `json.dumps(row) + "\n"` per round, `Content-Type: application/jsonl`. Add 60s `Cache-Control: public, max-age=60` — same as transcript. `_serve_trajectory(simulation_id, fmt)` shared body, two route decorators, same pattern as transcript.
2. In `EmbedDialog.vue`, add a "📊 Export trajectory data" section alongside "Export transcript" (below or adjacent). Three download buttons: "Download .csv" (triggers `<a href=blob>` download, same pattern as transcript buttons), "Download .jsonl", and a copyable CSV URL. Add a one-liner example beneath: `` `pd.read_csv(url)` — pandas, Excel, Tableau-ready. `` Add `getTrajectoryUrl()` helper to `frontend/src/api/simulation.js`.
3. Add both paths to `openapi.yaml` under Export tag (alongside transcript endpoints) with `text/csv` and `application/jsonl` response schemas and a `SimulationTrajectoryRow` schema object — drift test passes. Add `feed_bp: /api` → ensure `trajectory_bp` also added if it's a separate blueprint (or merge into existing simulation blueprint with a note). Add a "Trajectory Export (CSV / JSONL)" row to `docs/FEATURES.md` under Sharing/Export. Update README Features table with "Trajectory Export" row. Add a code snippet to `docs/API.md` showing `curl /trajectory.csv | python3 -c "import pandas, sys; ..."` as the "analyst quickstart" pattern.

### 5. Spectator Watch Page

**Type:** Feature / Growth
**Effort:** Small (hours)
**Impact:** Right now, tweeting a live MiroShark simulation requires tweeting the SimulationView URL — which opens into the full product UI, requires navigating to the right sim, and has no "broadcast this run in real time" affordance. A `/watch/:id` route renders a clean, minimal fullscreen page purpose-built for live spectating: the belief bar (auto-refreshing every 15s via `GET /api/simulation/:id`), the latest 3–5 agent posts from the most recent round (read from `GET /api/simulation/:id/events?type=post&limit=5`), a round counter, and a progress bar. No sidebar, no settings, no navigation chrome — just the sim, live. Once the sim completes, the page transitions to a static result card: final consensus, quality badge, and two CTAs — "Fork this scenario →" and "View full simulation →". The `/watch/:id` URL is share-ready: it returns a proper OG meta preview (scenario title, current belief split as description, share-card PNG as `og:image`), so when tweeted it auto-unfurls as an image card. Zero new backend: reads the existing `/api/simulation/:id` and `/api/simulation/:id/events` REST endpoints with polling. The "watch this sim in real time" tweet format is different from the "share a finished result" format — it creates a reason to engage immediately and follow the thread as rounds progress.
**How:**
1. Add a `WatchView.vue` route at `/watch/:id` (no auth, public — or 404 if sim is private). Layout: full-viewport dark background, centered 600px-wide column. Top: scenario title (truncated to 120 chars), "🔴 Live" blinking dot if status is `running`, else status badge. Middle: `BeliefBar` component (reuses existing `BeliefChart` or creates a simplified bar component — bullish/neutral/bearish percentage bars, no chart.js required). Below the bar: "Round N of M" + a progress bar (`width: (n/m)*100%`). Bottom: latest posts panel — last 3 agent posts from `GET /api/simulation/:id/events?type=agent_post&limit=3`, each showing agent name, stance emoji (🔺 bullish, ◉ neutral, 🔻 bearish), and post excerpt (first 160 chars). On sim completion: swap to a result card — final belief split, quality badge, big "Fork this scenario →" button (calls existing fork endpoint), smaller "View full simulation →" link. Poll every 15s while running; stop polling on completion. 
2. Add `<meta property="og:title">` = scenario excerpt (first 80 chars), `<meta property="og:description">` = "Round N · Bullish X% / Neutral Y% / Bearish Z% — watch live", `<meta property="og:image">` = share-card PNG URL (existing endpoint), `<meta property="og:url">` = `/watch/:id` canonical. This requires server-side rendering or a Flask-served SSR page for bots. The simplest approach: add `GET /watch/<sim_id>` to a new Flask blueprint that returns a static HTML shell with the OG tags pre-populated from `_build_gallery_card_payload()` and a `<script>` tag that mounts the WatchView Vue component. Same pattern as the existing `/share/:id` landing page (PR #42) — reuse its Flask route handler, just render different Vue component.
3. Add a "Watch live →" button to `SimulationView.vue` (visible only when status is `running`) — clicking copies the `/watch/:id` URL to clipboard and shows a "Link copied — tweet it!" toast. Add a "👀 Watch live" link to `EmbedDialog.vue`'s sharing section alongside the existing share-card URL and embed code. Add `getWatchUrl(simId)` helper to `frontend/src/api/simulation.js`. Add `GET /watch/:id` to the Flask router (SSR shell page — not a pure API endpoint, so it goes in `app/views.py` or equivalent, not `openapi.yaml`). Add a "Spectator Watch Page" row to `docs/FEATURES.md` under Sharing. Update README with a "Live Watch Page" entry in the Features table.

## Selection Rationale

Today's batch targets the research credibility gap and the live-broadcasting use case — two areas the past week's distribution arc (GIF + transcript + RSS) left untouched:

- **Historical Mode** (#1) — capitalises on the WWII demo's viral moment while the thread is still warm. The per-slot endpoint override is live; the UX layer takes hours to add. Every future historical demo becomes a 2-click flow rather than a manual env-var exercise.
- **LLM-as-Judge Audit** (#2) — the transcript was designed for judge pipelines (README says so explicitly). The judge entry point is the missing half. Gives academic and quant researchers the "reasoning quality score" they need before citing a MiroShark result.
- **Batch Rerun / Reproducibility** (#3) — makes a single sim result scientifically credible. "67% bearish ± 5% across 3 Cheap-preset runs" costs ~$0.50 post-PR-#52. The objection "but couldn't you just get lucky?" disappears once there's a badge.
- **Trajectory CSV / JSONL** (#4) — Pandas/Excel/Tableau is the analyst's default workflow. transcript.json covers prose; trajectory.csv covers the numbers. Zero new deps, pure stdlib, fifth surface in the export arc. The analyst "quickstart" is `curl /trajectory.csv | python3`.
- **Spectator Watch Page** (#5) — the "tweet a live sim" format is different from "share a finished result." A `/watch/:id` page with OG meta that auto-unfurls as an image card when tweeted creates a real-time engagement format. Zero new backend; reuses existing REST polls + the `/share/:id` SSR pattern from PR #42.

Each idea is scoped for autonomous implementation by the `feature` skill — all required data is on disk, no ambiguous design decisions, no external approvals needed.
