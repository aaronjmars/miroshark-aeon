# Status Pages Are Politics. One Endpoint This Week Wasn't.

In April 2026, IsDown's monthly report logged 45 outages the third-party monitor detected up to 3.6 hours before the vendors acknowledged them, and 104 separate incidents the vendors never reported at all. The average lag between when a degradation began and when it appeared on the official status page was 22 minutes. Status pages are not a neutral instrument. They are a publication, owned and operated by the same party whose uptime they describe, and the gap between the publication and the reality is wide enough that an entire third-party monitoring industry has organised around closing it.

Better Stack runs over 50,000 status pages as of 2026. IsDown aggregates 6,320 vendors into one dashboard. Upptime, Cachet, Statping — each exists because the version of the truth the vendor publishes lags, omits, or rounds. The structural problem isn't deliberate lying. It's that the vendor holds the pen and the trigger at the same time. The status page is the marketing surface for whether the marketing claim is currently true. The pen pauses when it's about to embarrass.

This is the frame to read PR #149 against, which merged into MiroShark at 13:01 UTC this morning. The PR ships `GET /api/status.json`, a small JSON envelope: `ok: true`, `queue_depth`, `completed_24h`, `last_completed_at`, `total_sims`, `surface_count`, `check_at`. None of that is unusual in isolation. What is unusual is the auth posture — and the auth posture is the answer to the IsDown report.

## The Vendor-Owned Status Page

The default contract between an API and its status page is mediated. The vendor's monitoring service watches the deployment, the vendor's incident commanders decide what counts as an incident, and the vendor's communications team writes the update. Each link is staffed by the vendor. Cloudflare's status feed is one Cloudflare controls; Google Cloud routinely lags hours behind user reports. This isn't a Cloudflare or Google problem; it's a property of any system where the publisher and the subject are the same entity.

The alternative the industry has converged on is to layer external probes over the vendor's surface. Upptime polls on GitHub Actions. Better Stack polls on its own infrastructure. IsDown crowdsources reports from end users. These systems work because they sit outside the vendor's control plane — observing the public surface and inferring the answer from behaviour. For the inference to work, the vendor has to expose a public surface that isn't gated by the same control plane the status page lives on. That surface is precisely what most APIs don't ship, because most APIs lock down `/api/*` by convention.

## The Endpoint That Had To Have Its Auth Removed

`/api/status.json` looks like a routine envelope until you read the commit history of the PR that landed it. The merged PR squashes three review-commits. The first was the initial implementation: scanner, blueprint, OpenAPI entry, 28 offline tests. The second was a drift-test fix — the documented path looked like a phantom to the static route scanner because the new blueprint wasn't in `_BLUEPRINT_PREFIXES`. The third one is the interesting one. It removed the default `internal_auth_guard` from this specific path. Every other `/api/*` endpoint on MiroShark inherits that guard. `/api/status.json` had to actively drop it, mid-PR, to deliver the contract the OpenAPI spec already documented. The auth-exemption list now reads `/openapi.json`, `/health`, `/api/status.json`. The probe is the third entry, and the only one with a JSON envelope describing the live deployment.

Two more details in the same commit make it a probe an external monitor can actually use. `ok: true` is a literal, not a derived value — the field is a stable anchor for status-page body-matchers, the one-line string check Upptime and Better Stack use to decide whether a page is healthy. A regression should bubble up through envelope drift or a 500; it should not silently flip the boolean. And `total_sims` was tightened from cumulative-everything to public-and-completed only, so an anonymous caller can't infer the volume of private, in-flight, or failed simulations from the probe. The third commit was, in effect, the entire IsDown thesis applied to one endpoint: the probe is the contract, the contract is publicly verifiable, and the publisher cannot use auth to filter what callers see.

## One Function as the Source of Truth

The `surface_count` field is derived from `surfaces_catalog.catalog_count()` — the same function the existing `/api/surfaces.json` reads to publish the capability catalog. A downstream alert keyed on "MiroShark advertises 32 surfaces today" doesn't need to poll the catalog separately or correlate two endpoints. The probe and the marketing surface read the same constant. Drift between what the platform advertises and what it reports can only come from the function being wrong, which a single test pins. The catalog moved from 31 to 32 with this PR. The new entry is `platform_status`. The surface machine readers query to discover other surfaces already knows about the one they would use to verify it.

## What This Looks Like in an Agent World

Composio's 2026 integration-patterns guide notes that OpenAI's `/v1/models` has become the de facto unauthenticated endpoint on self-hosted LLM stacks — the slot every operator leaves open because agents need a way to ask "are you up, and what do you do" before paying for credentials. MiroShark's platform-surface family is the same instinct generalised: `/api/stats` describes the corpus shape, `/api/surfaces.json` enumerates the capabilities, `/api/status.json` reports current health. The three together are the public probe surface for a deployment, and the third — the one that landed today — is the only one with auth actively stripped.

In a month when IsDown is reporting 104 outages no vendor admitted, the question isn't whether you should publish a status page. It's whether what you publish can be poll-verified by anyone, without credentials, by services with no stake in the answer. The default on most APIs is no. The default on `/api/status.json` had to be actively made yes, and that is the substance of what the PR shipped.
