## aaronjmars/MiroShark

### Theme: Inverse-direction share surface — Shareable Scenario Links (PR #71, filed and merged today)

**Summary:** Every share surface MiroShark has shipped to date points readers at a finished simulation. `/share/<id>` is the still-image preview of one. `/watch/<id>` is the live-polling view. Replay GIF is the motion version. Transcript MD+JSON is the prose version. RSS / Atom is the syndication version. Trajectory CSV+JSONL is the analyst version. Gallery search (PR #69, May 3) is the *index* across the prior seven. PR #71 changes direction: it's a **share surface for a sim that hasn't run yet**. The home page now reads four optional URL query params on mount — `?scenario=...` pre-fills the Simulation Prompt textarea (≤500 chars), `?url=...` auto-fetches into the URL Import list (must be `http://` or `https://`, ≤2000 chars), `?ask=...` pre-fills the Just Ask question field but does *not* auto-run (the "do not silently spend the operator's money" rule the May 2 cost-compression cluster encoded), and `?template=<slug>` auto-launches one of the preset templates by skipping the home page entirely and routing to `/process/new` via the existing `setPendingTemplate` flow that the gallery launch button already uses. Inputs are sanitized on read — DOMPurify strips HTML and `javascript:` URIs, control characters are stripped (newline / tab / CR preserved so multi-line scenarios survive), length caps prevent megabyte payloads, the URL is rejected unless its parsed protocol is `http:` or `https:`, the template slug is restricted to `[a-z0-9_-]+`. Once any field is touched the params are stripped via `router.replace({path:'/', query:{}})` so a refresh / address-bar copy reflects the user's edited state, not the original shared link.

The reverse direction lives in two new buttons. On the home page, a discreet **🔗 Share as link** button beneath the Simulation Prompt textarea constructs a `?scenario=...&url=...&ask=...` URL from the current form state and copies it to the clipboard with a 2.2 s "✓ Link copied" toast — only the first http(s) URL doc travels with the link (multi-doc setups are advanced and rare). On every preset template card a small **🔗** icon next to the Launch button copies a `?template=<slug>` URL with a brief ✓ flash. The flow is symmetric: read on `/`, write on `/` and on every template card, all backed by one `frontend/src/utils/urlParams.js` module so both directions can't drift in their definition of "valid input".

## Developer Notes

- **New dependencies:** None. DOMPurify was already pinned in `frontend/package.json` for the markdown renderer; PR #71 is the second consumer of the same dep. The zero-new-deps streak now extends across 10 consecutive PRs (#57 / #58 / #60 / #61 / #62 / #65 / #66 / #67 / #69 / #71 — #63 / #64 README-only).
- **Breaking changes:** None. PR #71 is purely additive — a fresh visit to `/` with no query params behaves exactly as before. The URL params are *optional and validated*; a malformed param is silently dropped, not a hard error.
- **Architecture shifts:** The substrate's share graph is now closed in both directions. Reads from a finished sim → the eight prior surfaces. Writes to start a new sim → `?scenario=…` / `?url=…` / `?ask=…` / `?template=<slug>`. The same `frontend/src/utils/urlParams.js` module is the spec for both halves of the inverse direction (read on `/`, write on `/` + per-template card), preventing the two halves from drifting in their definition of "valid input".
- **Tech debt:** `frontend/src/utils/urlParams.js` carries 27 standalone parser assertions in the PR description but no checked-in Vitest suite — the assertions live in PR-message scratch space rather than `frontend/src/utils/__tests__/urlParams.spec.js` or equivalent. If the param shape ever has to expand (a `?seed=<int>` for deterministic reruns, say), the parser tests would have to be re-derived from the PR thread. Worth a small follow-up to land a checked-in spec file alongside the module.
- **Sanitization notes:** the control-character regex is constructed from a string template and contains a literal `\\^` block — Grep confirms this matches the `^@-\^H` / `^L` / `^N-\^_` ranges plus DEL via the bracket-class shorthand. Effective coverage of C0 control chars while preserving `\t` `\n` `\v`(actually U+000B is excluded explicitly) `\r` is correct, but the construction-from-template idiom is harder to read than a plain literal `/[\x00-\x08\x0B\x0E-\x1F\x7F]/g` — could be tightened in a follow-up without behavior change.
