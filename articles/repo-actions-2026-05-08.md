## Ecosystem Context

PR #75 closed the reproducibility gap — the last barrier between a published MiroShark result and a citeable research artifact. That arc is complete: fifteen surfaces, one folder, one threshold, one bytewise-stable export. The question this batch asks is: *what comes after the surface layer?*

Three signal threads drive this batch:

- **The interconnection gap.** Each of the fifteen share surfaces operates independently. A share card links to `/share/<id>`; a thread exports to X; a trajectory goes to a CSV. But there's no way to navigate between related simulations — "what was forked from this run?", "which counterfactual did the operator trigger at round 30?" PR #75 records `parent_simulation_id` and `lineage.kind` in every reproduce.json. That data is on disk but invisible in the UI. A lineage endpoint + compact tree view in SimulationView turns the fork/counterfactual network from a metadata field into a navigable graph — the first time a viewer can trace the intellectual ancestry of a simulation result.

- **The discovery ceiling.** Gallery search (PR #69) enables keyword + filter + sort discovery. But the sort options — date, rounds, agents — are structural. Surface-stats (PR #74) is the first behavioral signal on disk: which sims are actually being served? The most-served sims this week are also the most-distributed — a proxy for most-interesting. Extending the gallery with `sort=trending` turns an observability data point into a discovery primitive. The first operator to publish a viral sim becomes the first operator whose work floats to the top of the gallery by merit rather than by recency.

- **The embed gap.** MiroShark's share card unfurls automatically on Twitter and Discord via OpenGraph tags (PR #42). But pasting a `/share/<id>` URL into Substack, WordPress, Ghost, Notion, or Bear gets a raw link — those platforms follow the oEmbed spec, not OpenGraph. An `oEmbed` endpoint (`GET /api/oembed?url=...`) returns the structured JSON response those platforms expect: photo type, share-card URL, dimensions, title, scenario preview. Zero new deps. Makes every existing share card URL auto-embed-ready in a dozen long-form writing platforms that the thread/transcript audience uses.

Two additional ideas round out the batch:

- **The peak-moment gap.** Share cards show the final consensus. Transcripts show every round. There is no surface for the single most interesting moment in a simulation — the round of peak belief divergence. A `GET /api/simulation/<id>/peak-round` endpoint identifies that moment from the trajectory and returns the belief split + a sample of the most polarized agent posts. The EmbedDialog gets a "Peak moment" export alongside the share card — a second shareable image (same 1200×630 generator) capturing the inflection rather than the resolution. The inflection is the story; the resolution is the verdict.

- **The operator identity gap.** MiroShark results are circulating externally — Umia leaderboard, Grok citations, Bankr Terminal references. But every published sim is anonymous: there's no way for a viewer to know who operates the instance, whether the results are curated, or whether a sim was published by the same person who published related work. A lightweight operator profile (`GET /api/operator/profile` from env vars: `OPERATOR_NAME`, `OPERATOR_BIO`, `OPERATOR_URL`) adds the first authorship layer. Published sims get a small "By [name]" attribution in the share page footer. Zero runtime cost — read-only, env-configured, no database.

### 1. Trending Simulations Sort (surface-stats → discovery signal)

**Type:** Feature / Discovery
**Effort:** Small (hours)
**Impact:** Surface-stats (PR #74) gives operators per-sim serve counts. But the data is siloed inside EmbedDialog — operators see their own stats, not the platform-wide distribution signal. Extending `GET /api/simulation/public?sort=trending` to rank public sims by their surface-stats total (sum of all share surface serves) turns an operator-facing analytics tool into a platform-wide discovery primitive. The most-served sim this week floats to the top of the gallery. This is the first feedback loop from distribution analytics into discovery ranking: sims that get shared get found more easily, compounding the distribution advantage of high-quality outputs.
**How:**
1. In `backend/app/services/gallery_filters.py`, add `"trending"` to the `VALID_SORT_OPTIONS` frozenset. Add a `_get_surface_total(card)` helper that reads `surface_stats.read_surface_stats(sim_dir)` for the sim (the card's `sim_dir` field, or derive it from `simulation_id` via `SimulationManager.get_simulation_path()`). In `sort_cards()`, add the `trending` case: sort by `_get_surface_total(card)` descending, tie-break by date descending (most-served-and-most-recent floats above most-served-and-stale). Gracefully degrade: if `surface-stats.json` is missing for a sim, treat `total = 0` — old sims without the file sort to the bottom without raising. Add 6 offline unit tests to `test_unit_gallery_filters.py`: trending sort with all stats present, missing-stats-file degrades to 0 (no exception), tie-break on date, mixed-presence corpus (some sims with stats, some without), trending unknown → falls back to date sort, end-to-end `select_filtered_cards(sort="trending")` returns highest-served sim first.
2. Extend `openapi.yaml` `sort` query param enum to include `"trending"` — drift-detection test passes. Update `frontend/src/api/simulation.js` `getPublicSimulations()` to forward `sort: "trending"` when provided. In `ExploreView.vue`, add a "🔥 Trending" option to the sort dropdown (same `<option>` element pattern as existing date/rounds/agents). Add a subtle "🔥" flame icon prefix in the sort-option label to visually differentiate it from structural sorts. Preselect "🔥 Trending" when the user lands on `/explore` for the first time (overridable by explicit `?sort=` query param). Mirror the selected sort in the URL via `router.replace` (same pattern as existing sort options — no back-stack pollution). Add "Trending sort" row to `docs/FEATURES.md` under Gallery & Discovery + a note to `docs/API.md` under the `sort` param.
3. Performance note: `gallery_filters.select_filtered_cards()` already builds all gallery cards before filtering (since keyword/consensus filters need the assembled card). The `_get_surface_total()` call adds one `json.load()` per public sim. Mitigate with a pass-over: compute all surface totals in a single sweep before the sort (same pattern `_build_gallery_card_payload` uses), store them on a per-card `_serves_total` transient field, sort against that. No new caching layer needed — the existing `Cache-Control: public, max-age=30` on the gallery endpoint amortises the disk reads across many requests.

### 2. oEmbed Endpoint (paste-to-embed for Substack / Notion / Ghost)

**Type:** Integration / DX
**Effort:** Small (hours)
**Impact:** MiroShark's share card unfurls automatically on Twitter and Discord (OpenGraph). But pasting `/share/<id>` into Substack, WordPress, Ghost, Notion, Bear, or any oEmbed-supporting platform produces a raw link — no card, no image. The oEmbed spec (oembed.com) is the industry standard that those platforms follow: they fetch `?url=<share-url>` from a registered `oEmbed` endpoint and get a structured JSON response describing the embed. A `GET /api/oembed?url=<share-url>` route returning `{type: "photo", url: <share-card-url>, width: 1200, height: 630, title: <scenario-preview>, provider_name: "MiroShark", ...}` makes every existing share card URL paste-to-embed in a dozen platforms. The long-form writing audience — the same one that downloads transcripts and subscribes to the Atom feed — switches between Substack and Twitter constantly; this closes the gap. Zero new deps; pure stdlib Flask JSON response. Registerable on the official oEmbed provider registry (oembed.com/providers) for automatic discovery by platform crawlers.
**How:**
1. Add `GET /api/oembed` route to `backend/app/api/simulation.py` (or a new `oembed.py` blueprint). Accept a single `url` query parameter. Parse the URL: validate it matches the `/share/<id>` pattern for this MiroShark instance (use `Config.PUBLIC_BASE_URL` as the expected host — same guard as `_resolve_base_url()`). Extract `<simulation_id>`. Load the simulation via `SimulationManager.get_simulation(sim_id)`. If not found or not published: return HTTP 404 (oEmbed spec requires 404 for "not found or not embeddable"). Otherwise: call `_build_embed_summary_payload(sim)` to get scenario + consensus data. Compose oEmbed response: `{type: "photo", version: "1.0", title: <scenario truncated to 100 chars>, author_name: <OPERATOR_NAME env var if set, else "MiroShark">, provider_name: "MiroShark", provider_url: <PUBLIC_BASE_URL>, url: <share-card-png-url>, width: 1200, height: 630, thumbnail_url: <share-card-png-url>, thumbnail_width: 1200, thumbnail_height: 630}`. Return as `Content-Type: application/json`. Honor the `maxwidth`/`maxheight` oEmbed parameters if supplied (for MiroShark the image is fixed at 1200×630, so just echo the requested dimensions as the actual — platforms handle the scale client-side). Add 10 offline unit tests in `test_unit_oembed.py`: valid share URL extracts correct sim_id, published sim returns 200 with type="photo", unpublished sim returns 404, invalid URL returns 404, missing url param returns 400, title truncated at 100 chars, provider_name always "MiroShark", url field points to share-card PNG, width/height always 1200/630, route-decorator presence guard.
2. Add `<link rel="alternate" type="application/json+oembed" href="/api/oembed?url=<encoded-share-url>" title="<scenario>">` meta tag inside `<head>` of the share page (`backend/app/api/share.py` or wherever `_build_share_html()` lives). This is the oEmbed auto-discovery signal — when Substack's crawler fetches `/share/<id>`, it sees the `<link>` tag and knows where to find the oEmbed JSON without being pre-configured. Add `/api/oembed` to `openapi.yaml` under Publish & Embed with a `OEmbedResponse` schema (all standard oEmbed fields). Drift-detection test passes. Add `getOEmbedUrl(simId, origin)` helper to `frontend/src/api/simulation.js`.
3. In `EmbedDialog.vue`, add an "🔗 Embed anywhere" section beneath the share card row. Layout: a copy-ready `<img>` HTML snippet (`<img src="<share-card-url>" alt="<scenario>" width="1200" height="630">`) for manual paste into any HTML context, a note "Substack, Notion, and Ghost auto-embed this URL — just paste `/share/<id>`", and a "Copy oEmbed URL" button for platforms that ask for the oEmbed endpoint directly. Publish-gated (same 403 pattern as every other EmbedDialog section). Add "oEmbed / Rich Embed" row to `docs/FEATURES.md` under Publish & Embed. Add a "Supported platforms" note to `docs/API.md` listing Substack, WordPress, Ghost, Notion, Bear, and Micro.blog as platforms that support oEmbed auto-discovery.

### 3. Simulation Lineage Navigator (fork tree from reproduce.json)

**Type:** Feature / Research
**Effort:** Small (hours)
**Impact:** PR #75 records `lineage.kind` (original/fork/counterfactual), `parent_simulation_id`, and a 140-char counterfactual label in every reproduce.json. But the lineage is one-directional: a child sim knows its parent, but the parent has no visibility into its children. A researcher who ran the base scenario then triggered three counterfactual branches can't navigate from the parent to the branches — they have to remember each sim ID. A `GET /api/simulation/<id>/lineage` endpoint reads the public sim index, identifies children (public sims whose reproduce.json `parent_simulation_id` matches the requested ID), and returns `{root: <id|null>, parent: {id, scenario_preview}|null, children: [{id, kind, scenario_preview, created_at}]}`. SimulationView gets a compact "🌳 Lineage" section beneath the belief chart: parent card (if any) + children list (forks and counterfactuals with their kind labels). Click on a child navigates to `/watch/<child_id>`. This closes the navigation gap that the fork/counterfactual feature (PR #67's "Fork this scenario" button) created — operators who build multi-branch simulation trees can now traverse them as a graph rather than a flat list of IDs.
**How:**
1. Add `GET /api/simulation/<id>/lineage` to `backend/app/api/simulation.py`. Logic: load the reproduce.json for the requested sim to get `parent_simulation_id` and `lineage.kind`. Load all public sim IDs via `SimulationManager.list_public_simulations()`. For each public sim, read its reproduce.json `parent_simulation_id` field (with graceful fallback — most sims won't have a reproduce.json; treat as `parent_simulation_id: null`). Collect those whose `parent_simulation_id == requested_id` as children. Resolve parent: if the requested sim has a `parent_simulation_id`, load that sim's reproduce.json for its `scenario` field (truncated to 80 chars as `scenario_preview`). Return: `{simulation_id: str, lineage_kind: str, parent: {id, scenario_preview, created_at}|null, children: [{id, kind, scenario_preview, created_at}], total_children: int}`. Publish-gate: the endpoint is public for published sims (same `is_public` check). 403 for unpublished. Children list only includes public sims (private children are not exposed). Limit to 50 children max (paginate via `?offset=` if needed — rare edge case, but safe). Add 12 offline unit tests in `test_unit_lineage.py`: sim with no parent and no children (original with no forks), sim with parent (fork returns parent data), sim with 3 children (returns all 3, each with correct kind), child is unpublished (excluded from children list), parent is unpublished (parent field returns null not 403), simulate missing reproduce.json on a child (treated as parent_simulation_id=null, not counted), children sorted by created_at ascending (oldest fork first), total_children reflects only public count, 403 on unpublished requested sim, route-decorator presence, openapi drift guard, 50-child limit (51st is not returned).
2. Add `GET /api/simulation/{simulation_id}/lineage` to `openapi.yaml` under Analytics with a `SimulationLineage` schema. Drift-detection test passes. Add `getSimulationLineage(simId)` helper to `frontend/src/api/simulation.js`. In `SimulationView.vue`, fetch lineage on mount (alongside belief chart data). Render the "🌳 Lineage" section only when `parent !== null || children.length > 0` — hidden for standalone original sims with no forks (the majority of sims). Parent card: compact one-liner with scenario_preview truncated to 60 chars + "↑ Parent" label + link to `/watch/<parent_id>`. Children list: one row per child with a `[🪐 Fork]` or `[🔀 Counterfactual]` chip + scenario_preview + link to `/watch/<child_id>`. "View all N branches →" link when `total_children > 5` (shows first 5 inline). CSS: same chip style as the lineage badges in EmbedDialog (already defined for `repro-lineage-badge`).
3. In `EmbedDialog.vue`, add a "🌳 Lineage" row beneath the Reproducibility Config section. Show `total_children` as "N forks / counterfactuals" with a link to the SimulationView lineage section. If `lineage_kind === "fork"` or `"counterfactual"`, show "View parent simulation →" link. This closes the navigation loop: the embed dialog is where operators manage sharing, and lineage context belongs there alongside the reproducibility config. Add "Simulation Lineage" row to `docs/FEATURES.md` under Publish & Embed. Add the endpoint to `docs/API.md` under Analytics.

### 4. Peak-Round Belief Snapshot (inflection card export)

**Type:** Feature / Research
**Effort:** Small (hours)
**Impact:** The share card (PR #42) shows the final consensus: the verdict. The transcript (PR #57) shows every round: the full record. There is no surface for the most dramatic moment in a simulation — the round where belief divergence was highest, where the most agents switched stance, where the most interesting thing happened. A `GET /api/simulation/<id>/peak-round` endpoint identifies this moment by scanning the trajectory for maximum `|bullish_pct - bearish_pct|` (or whichever stance leads by the widest margin over the runner-up). It returns the peak round's belief split, quality snapshot, and a sample of the 3 most polarized agent posts from that round. The share card generator gets an optional `?round=<n>` parameter so the EmbedDialog can export a "Peak moment" card alongside the existing final-consensus card — same 1200×630 format, but the belief bars reflect the peak divergence rather than the closing state. The peak-moment card is the Twitter-native format: one image, maximum drama, the story in a glance rather than the verdict.
**How:**
1. Add `backend/app/services/peak_round.py` (~200 LoC, pure stdlib `json` + `os`): `STANCE_THRESHOLD = 0.2` constant (parity with all other surfaces). `_divergence_score(row)`: computes `max(bullish_pct, neutral_pct, bearish_pct) - sorted([bullish_pct, neutral_pct, bearish_pct])[-2]` — the lead of the dominant stance over the runner-up; returns `0.0` on invalid input. `find_peak_round(sim_dir)`: reads `trajectory.json`, computes divergence score per round, returns the round with maximum score (ties: prefer the later round, since late-sim divergence is more meaningful than early noise). `_sample_peak_posts(sim_dir, peak_round_num, n=3)`: reads per-round agent post data (inspect the sim_dir structure — likely in `trajectory.json` or per-round files) and returns up to 3 posts from that round whose authors had the most extreme stance. `build_peak_round_payload(sim_dir)`: assembles `{peak_round: int, total_rounds: int, bullish_pct: float, neutral_pct: float, bearish_pct: float, divergence_score: float, stance_label: str, sample_posts: [{agent_name, stance, text_excerpt}], empty: bool}` — `empty: true` when no trajectory data exists (graceful degradation). Add `GET /api/simulation/<id>/peak-round` route (publish-gated, 403 for unpublished, `Cache-Control: public, max-age=300`). Add 14 offline unit tests in `test_unit_peak_round.py`: flat trajectory (all equal) returns earliest round with `divergence_score=0.0`, maximum-divergence round selected over earlier rounds, tie-break prefers later round, missing trajectory degrades to `empty:true` (no exception), corrupt-line in trajectory skipped (same pattern as other surface readers), `STANCE_THRESHOLD` parity guard, stance_label is the dominant stance name, `total_rounds` matches trajectory length, `peak_round` is within `[1, total_rounds]` range, sample_posts capped at 3, posts are from the correct round, route-decorator presence guard, openapi drift guard, publish gate 403.
2. Extend `backend/app/services/share_card_renderer.py` (or equivalent) to accept an optional `round_override: int` parameter in `render_share_card(sim, round_override=None)`. When `round_override` is set, use that round's belief split rather than the final consensus split. Modify `GET /api/simulation/<id>/share-card.png` to accept `?round=<n>` query param: validate it's a positive integer ≤ total_rounds, pass it to the renderer. The cache key (`summary_cache_key()`) must incorporate the round parameter so `?round=15` and `?round=final` produce distinct cached images. Add `getPeakRoundCardUrl(simId)` helper to `frontend/src/api/simulation.js` that composes `/api/simulation/<id>/share-card.png?round=<peak_round>`. Add `getPeakRound(simId)` API helper.
3. In `EmbedDialog.vue`, add a "⚡ Peak moment" section beneath the share card row. Fetch `GET /api/simulation/<id>/peak-round` on dialog open. If `payload.empty` is false, show: "Round [peak_round] of [total_rounds] — [stance_label] [bullish_pct]% / [neutral_pct]% / [bearish_pct]% — divergence [divergence_score rounded to 1dp]pp". Render the peak-moment share card as a lazy-loaded `<img>` (same `tap-to-load` poster pattern as the replay GIF row). Copyable URL + "Download peak card" button. Sample posts (up to 3, truncated to 120 chars each) shown in collapsed `<details>` beneath the card. If `payload.empty` is true, show "No trajectory data yet." Add "Peak-Round Snapshot" row to `docs/FEATURES.md`. Add `?round=<n>` parameter note to the `/api/simulation/{id}/share-card.png` entry in `docs/API.md`.

### 5. Operator Profile + Attribution Layer

**Type:** DX / Community
**Effort:** Small (hours)
**Impact:** MiroShark results are circulating externally — Umia leaderboard, Grok citations, Bankr Terminal references. But every published sim is anonymous: the share page shows the scenario, the belief bars, and the quality health, but no indication of who operates the instance or what they're researching. A lightweight operator profile drawn from env vars (`OPERATOR_NAME`, `OPERATOR_BIO`, `OPERATOR_URL`) adds the first authorship layer. `GET /api/operator/profile` returns `{name, bio, url, verified: bool}` — `verified: true` when `OPERATOR_NAME` is set (the operator took the time to configure it). Share pages (`/share/<id>`) and watch pages (`/watch/<id>`) get a small "By [name]" attribution line in the footer. Gallery cards get a `{operator_name}` field; ExploreView shows it as a subtle byline beneath the scenario. This is the layer that makes "Aaron's MiroShark instance" distinct from "a MiroShark instance" — relevant when the operator's identity is part of the signal (which it is on Umia and in Grok citations). Zero runtime cost: read-only, env-configured, no database, no auth. One config, sitewide propagation.
**How:**
1. Add `backend/app/services/operator_profile.py` (~80 LoC, pure stdlib `os`): `get_operator_profile()` reads `OPERATOR_NAME`, `OPERATOR_BIO`, `OPERATOR_URL` from `os.environ`, strips whitespace, validates `OPERATOR_URL` is `http(s)://` or empty (no-op on invalid). Returns `{name: str|null, bio: str|null, url: str|null, verified: bool}` — `verified = bool(name)`. All fields optional; if `OPERATOR_NAME` is unset, the endpoint returns `{name: null, bio: null, url: null, verified: false}` (no error). Add `GET /api/operator/profile` route (new `operator_bp` blueprint with `prefix: "/api"` — add to `_BLUEPRINT_PREFIXES` so drift-detection test passes). No auth required — this is public metadata. `Cache-Control: public, max-age=3600` (profile changes only on redeploy). Add 8 offline unit tests in `test_unit_operator_profile.py`: all env vars set → all fields populated, `verified=true`, `OPERATOR_URL` validation rejects `javascript:`, unset `OPERATOR_NAME` → `name=null, verified=false`, trailing whitespace stripped from all fields, `OPERATOR_BIO` ≤ 200 chars (truncated silently), `OPERATOR_URL` missing scheme treated as null (not crash), route-decorator presence guard. Add `GET /api/operator/profile` to `openapi.yaml` under a new `Operator` tag with an `OperatorProfile` schema — drift-detection test passes.
2. Inject the operator profile into share page and watch page HTML. In `backend/app/api/share.py` (share page renderer) and `backend/app/api/watch.py`, call `operator_profile.get_operator_profile()` at render time. If `profile.verified` (i.e., `OPERATOR_NAME` is set), add a `<footer>` to the HTML with: `By <a href="{url}">{name}</a>` (linked if `url` is set, plain text otherwise) + `{bio}` on a second line (truncated to 100 chars). If not verified, footer is omitted — unbranded instances don't show a blank attribution. Add `<meta name="author" content="{name}">` to the `<head>` (improves SEO and citation tools). The gallery card assembly in `_build_gallery_card_payload()` gains an `operator_name: str|null` field drawn from `get_operator_profile().name` — computed once per gallery request and injected into every card (not per-sim). Extend the `openapi.yaml` `SimulationGalleryCard` schema with `operator_name: {type: [string, null]}` — drift-detection test passes.
3. In `ExploreView.vue`, add a subtle `operator_name` byline beneath each gallery card's scenario text — rendered only when `card.operator_name` is non-null. Style: `opacity: 0.6`, `font-size: 0.78rem`, `font-style: italic` — visually subordinate to the scenario but findable. Add `getOperatorProfile()` API helper to `frontend/src/api/simulation.js`. In the site header or footer of the SPA (wherever `App.vue` or `NavBar.vue` renders), fetch `/api/operator/profile` on mount and show `{name}` as an instance byline beneath the "MiroShark" wordmark — only when verified. Add "Operator Profile & Attribution" row to `docs/FEATURES.md` under Configuration. Add `OPERATOR_NAME`, `OPERATOR_BIO`, `OPERATOR_URL` to the environment variable reference in `README.md` and `docs/` configuration section.

## Selection Rationale

This batch targets the layer *above* the fifteen completed share surfaces: interconnection, platform integration, lineage navigation, peak-moment discovery, and authorship.

- **Trending Simulations Sort** (#1) — Surface-stats (PR #74) is currently operator-facing only. Extending the gallery sort to include `trending` turns behavioral data into a discovery signal — the first time a sim's distribution performance influences its findability. Pure stdlib, extends existing `gallery_filters.py`, zero new files.
- **oEmbed Endpoint** (#2) — The long-form writing audience (Substack, Notion, Ghost) is exactly the audience that downloads transcripts and subscribes to the Atom feed. OpenGraph covers Twitter/Discord; oEmbed covers the rest. One new route, no new deps, makes every existing share card URL instantly embeddable on a dozen platforms.
- **Simulation Lineage Navigator** (#3) — PR #75's lineage metadata is invisible in the UI. The fork/counterfactual network exists on disk but can't be traversed. A lineage endpoint turns `parent_simulation_id` into a navigable graph. The feature skill already knows the sim_dir layout from PR #75; this is a read-and-join, not a new write path.
- **Peak-Round Belief Snapshot** (#4) — The share card shows the verdict; the peak-round card shows the story. The trajectory data is already on disk (PR #66); the share card renderer already accepts belief inputs. The peak-round endpoint is a trajectory scan + renderer call, not a new data pipeline.
- **Operator Profile + Attribution** (#5) — MiroShark results are circulating on Umia and in Grok citations without authorship context. A three-field env-var profile closes this with zero runtime cost. The Umia leaderboard moment makes operator identity load-bearing for the first time.

Each idea is scoped for autonomous implementation by the `feature` skill — all data is on disk or in existing endpoints, no ambiguous design decisions, no external approvals required.
