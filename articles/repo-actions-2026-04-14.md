## Ecosystem Context

MiroShark enters its fourth week with a feature stack that few open-source simulation projects have matched: cross-platform simulation engine, graph-built personas, Wonderwall AMM prediction market, full observability tracing, article generator, prediction resolution & accuracy tracking, belief drift visualization, trace-grounded agent interview (landing imminently in PR #26), fork/branch, comparison mode, and public share permalinks. At 680 stars and ~10–15 new stars daily, growth is sustained — the 7-day token rally (+560%) suggests increasing community crossover between the GitHub/dev audience and the crypto-native audience.

The gap that now matters most is **statistical depth and external connectivity**. MiroShark runs one simulation at a time and produces one set of results. A researcher who wants to claim "agents consistently predicted X" can't do that with a single run — they need N runs with variance data. A developer who wants to trigger simulations from an external workflow can't do that without a programmatic interface. A user who runs a 15-minute simulation on a cloud instance has no way to know it finished without watching the tab. These aren't edge cases — they're the exact friction points that separate a demo tool from a production-grade research platform.

Key signals driving this batch:

- **Single-run results are not statistically defensible:** Every feature added since April (belief drift, prediction resolution, article generator, polarization index) produces a richer output from a single run. But one run's market price could be +15% in a different seed. Batch runs with variance reporting turn each simulation into a *distribution*, not a point estimate. OASIS (Park et al., Stanford/Google) — the most-cited multi-agent social simulation paper — ran every experiment as an ensemble. MiroShark needs this to be taken seriously in academic contexts.
- **Long simulations lose users:** A 50-agent, 15-round simulation on a cloud Railway/Render instance takes 4–10 minutes. Users start a run, switch tabs, and forget to return. Browser push notifications via the Web Push API (Service Worker, no external service) are a 2-hour implementation that immediately reduces abandoned-run rates for the cloud deploy audience.
- **Research pipelines run in Python, not Vue:** The article generator, belief drift chart, and prediction resolution UI are all designed for consuming results visually. But researchers who want to run their own statistical tests (t-tests, regression, network centrality on belief change) need the raw data in a format they can load into pandas or R. A Jupyter notebook export that pre-populates the DataFrames and includes starter analysis cells is the bridge.
- **No programmatic trigger path:** `aaronjmars` mentioned Aeon integrating with MiroShark. The feature skill and repo-pulse skill could trigger a simulation, wait for results, and act on them. But there's no API key + REST endpoint for this. Currently simulations can only be started from the Vue frontend. A `/api/run` endpoint with key-based auth + webhook-on-complete opens MiroShark to any external automation — Zapier, GitHub Actions, Make, n8n, or Aeon itself.
- **No discovery layer for completed simulations:** The history view is per-instance and private. But with public share permalinks, cloud deploy, and 127 forks, there are now users running MiroShark in environments with multiple participants. A "Hall of Fame" page that surfaces the most interesting completed simulations (most surprising consensus, most polarized, most accurate prediction) creates an in-product discovery layer and gives casual visitors an immediate answer to "what can this thing actually do?"

These 5 ideas are distinct from all previously generated ideas in the past 7 days (CLI runner, demographic calibration, agent decision trace viewer, multi-language input, custom OG image — all from Apr 7–10; agent interview ✓ PR #26, belief drift ✓, Polymarket seeding, community template gallery, polarization index — all from Apr 12) and from the 1 open PR (#26 trace interview).

### 1. Statistical Batch Runs with Aggregate Dashboard

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** A single simulation run produces one market outcome, one leaderboard, one belief drift curve — inherently non-generalizable. For any simulation used in a research context ("our agents predicted YES with 73% confidence"), reviewers will ask: "Is that result stable, or did it happen to come up that way once?" Batch mode runs the same configuration N times (3, 5, or 10 runs) with different random seeds, then presents an aggregate dashboard: mean market final price with ±1 standard deviation error bars, belief drift consensus round as a histogram (across N runs), leaderboard stability score (% of runs where the same agent ranked #1), and a "robustness rating" (Low/Medium/High based on SD relative to mean). This is the single feature that unlocks academic citation. OASIS paper ran 100+ ensemble experiments; even 5 runs with variance data is defensible. It also makes MiroShark useful for high-stakes practical simulation ("we ran 10 trials; 8 predicted rejection") without the user needing to manually trigger and compare runs.
**How:**
1. Add a "Run as Batch" toggle to the simulation launch button — when enabled, show a "Runs: 3 / 5 / 10" radio button group. On submit, the backend creates N simulation entries with the same config but incremented random seeds (stored as `batch_id` and `seed_offset` in each simulation's metadata). A new `POST /api/batch` endpoint creates the batch record and enqueues the N runs sequentially. Return a `batch_id` to the frontend.
2. Add a `GET /api/batch/:batch_id` endpoint that polls the N simulation statuses, and when all are `completed`, computes aggregate stats: for each simulation, extract `final_yes_price`, `belief_drift_consensus_round` (from the belief drift endpoint), and `top_agent_id` (rank 1 from influence leaderboard). Compute mean/SD for continuous values; mode for categorical. Return the aggregate object.
3. Add a `BatchResultsView.vue` component that opens as an overlay once all batch runs complete. Show: a market price distribution chart (box plot or violin: final YES price across N runs), a consensus round histogram, a leaderboard stability section (top 3 most consistently ranked agents across runs with stability %), and a "Robustness: High/Medium/Low" badge. Include a "View individual run" drill-through link for each of the N simulations. Add a "Download batch CSV" that exports all N results as a flat CSV with one row per run.

### 2. Browser Push Notification on Simulation Complete

**Type:** DX Improvement
**Effort:** Small (hours)
**Impact:** The cloud deploy path (Railway/Render) puts MiroShark in the hands of users who run simulations in background tabs. A 50-agent, 15-round run on a remote instance takes 4–10 minutes — enough time to switch tasks and forget the browser tab. Users discover their completed simulation hours later or abandon the tab entirely, losing the result. Browser Web Push notifications (via the Push API + Service Worker) require no external service and no backend infrastructure change beyond a `/api/push/subscribe` endpoint — the notification is sent peer-to-peer from the server to the user's browser. When a simulation finishes, the user receives a browser notification even if the tab is hidden: "Your simulation on '{scenario}' is complete — consensus: 73% YES, 12 rounds." Clicking the notification focuses the tab and scrolls to the result. This is a 2-4 hour implementation that directly reduces the single most common UX failure mode for long-running simulations.
**How:**
1. Register a Service Worker in `frontend/public/sw.js` that listens for `push` events and calls `self.registration.showNotification(title, { body, icon, data: { simulationUrl } })`. In `main.js`, register the service worker on app load and request notification permission when the user starts a simulation (ask once, persist the result). On permission grant, call `POST /api/push/subscribe` with the PushSubscription object (endpoint + keys from `registration.pushManager.subscribe()`). Store subscriptions in a `push_subscriptions` table in SQLite.
2. When a simulation transitions to `completed` status in the simulation service, look up all push subscriptions associated with the session (use session ID from the simulation config as the association key). Send a Web Push notification to each subscription using the `pywebpush` library (`pip install pywebpush`). The notification payload: `{ title: "Simulation Complete", body: "{scenario} — consensus {final_price}% YES after {rounds} rounds", url: "/history?highlight={simulation_id}" }`. Generate VAPID keys once at server startup and store in environment variables.
3. Add a "Notify me when done" toggle to the simulation start button — enabled by default once permission is granted, off by default before permission. Show a subtle browser notification permission prompt banner if notifications are available but not yet granted. Add a `GET /api/push/test` endpoint that fires a test notification immediately — useful for debugging and for users who want to verify the feature works before starting a long run.

### 3. Jupyter Notebook Export of Simulation Data

**Type:** DX Improvement
**Effort:** Medium (1–2 days)
**Impact:** The belief drift chart, prediction resolution, and article generator all visualize simulation results within the MiroShark UI. But researchers who want to run their own analysis — network centrality on agent interaction graphs, regression on stance-change predictors, custom visualizations — need the raw data in a Python-friendly format. A "Download as Jupyter Notebook" button generates a `.ipynb` file that contains: pre-loaded simulation data as pandas DataFrames (timeline events, agent belief states per round, market prices, interaction network edges), pre-written analysis cells (plot belief drift, compute polarization index, build interaction heatmap), and empty research cells with prompts ("Add your own analysis below"). The researcher downloads the notebook, runs `jupyter notebook`, and immediately has working Python code with the simulation's data. This is the bridge between MiroShark's visual UI and the Python/R analysis pipelines that academic users actually publish from. No other multi-agent social simulation tool offers this out-of-the-box.
**How:**
1. Add a `GET /api/simulation/:id/export/notebook` endpoint that builds the notebook JSON programmatically (using Python's `nbformat` library — `pip install nbformat`, zero external dependencies). The notebook contains 6 cells: (1) imports cell (`pandas`, `matplotlib`, `networkx`, `json`), (2) data loading cell (base64-encodes the three core JSON files — timeline.json, trajectory.json, market prices — directly into the notebook as Python dict literals so the notebook is self-contained with no file path dependencies), (3) DataFrame construction cell (builds `df_timeline`, `df_beliefs`, `df_market`), (4) belief drift plot cell (pre-written matplotlib stacked area chart), (5) interaction network cell (builds a `networkx` graph from reply/like data, computes degree centrality), (6) blank "Your analysis" cell. Return the notebook as `Content-Type: application/octet-stream` with filename `simulation-{id}-{date}.ipynb`.
2. Add a "Download Notebook" button to the simulation results view alongside the existing Share and export controls. Add it to the completed simulation history card's action menu (the three-dot menu). Show a tooltip: "Opens in Jupyter — requires Python + Jupyter Notebook or JupyterLab." If the simulation is still running, grey the button out with "Available after completion."
3. Add a `GET /api/simulation/:id/export/csv-bundle` endpoint alongside the notebook endpoint that returns a ZIP file containing: `timeline.csv` (all events with round, agent_id, platform, action, sentiment), `beliefs.csv` (agent × round belief state matrix), `market.csv` (round × yes_price time series), `agents.csv` (agent profiles with demographics). Link to this from the notebook download area as "Download raw CSVs instead." Both formats share the same data extraction logic in a new `backend/app/services/export_service.py`.

### 4. REST API with API Key Authentication and Webhook on Completion

**Type:** Integration
**Effort:** Medium (1–2 days)
**Impact:** MiroShark simulations can only be triggered from the Vue frontend — there is no programmatic path. This blocks three valuable workflows: (1) automated agents like Aeon that want to trigger simulations as part of a daily skill run and receive results; (2) researchers who want to script 20 parameter variations from a notebook without clicking through the UI; (3) external services (Zapier, Make, GitHub Actions) that want to trigger a simulation when a new document is published, a market event fires, or a PR is opened. A `POST /api/run` endpoint with API key auth + a `webhook_url` parameter closes this gap in one implementation. The simulation runs as normal; on completion, it POSTs a result payload to the webhook URL. This is the "composability unlock" — it turns MiroShark from a standalone app into a building block in any automation stack.
**How:**
1. Add API key management: a `GET /api/keys` endpoint that generates and stores a new API key (UUID, hashed in SQLite `api_keys` table). Add a "Generate API Key" button in a new "Developer" section of the Settings panel. The key is shown once and stored as its SHA-256 hash. Add a `api_key_required` decorator that checks `Authorization: Bearer <key>` header on protected endpoints, hashes the input, and compares against stored hashes.
2. Add a `POST /api/run` endpoint (requires API key) accepting a JSON body: `{ "document": "text or URL", "agent_count": 50, "rounds": 10, "scenario": "...", "webhook_url": "https://..." }`. This endpoint validates the payload, creates a simulation entry (same as the UI flow), and returns `{ "simulation_id": "...", "status": "queued" }` immediately. Optionally also accept `"config": {...}` as a pre-built config JSON (skipping the LLM config generation step for scripted use). Add `GET /api/simulation/:id/status` as a polling endpoint (also key-protected) returning `{ "status": "running|completed|failed", "result_url": "..." }`.
3. When a simulation with a `webhook_url` completes, fire an HTTP POST to that URL with a signed payload: `{ "simulation_id": "...", "status": "completed", "final_yes_price": 0.73, "top_agent": "...", "rounds_completed": 10, "share_url": "https://..." }`. Sign the payload with an HMAC-SHA256 signature using the API key as the secret — include it as `X-MiroShark-Signature` header so the receiver can verify authenticity. Retry up to 3 times on failure with exponential backoff. Add webhook delivery logs to the simulation record (`webhook_attempts`, `webhook_last_status`).

### 5. Notable Simulations Hall of Fame Page

**Type:** Community / Growth
**Effort:** Small (hours)
**Impact:** MiroShark's history view is per-user and private. But with 127 forks, cloud deploy, and public share permalinks, there are users generating interesting simulation results that no one else can discover. A `/hall-of-fame` page (or `/explore`) surfaces the most interesting completed simulations from the local instance, automatically ranked by objective quality metrics: "Most surprising consensus shift" (largest stance delta from round 1 to final), "Most polarized debate" (highest polarization index), "Most accurate prediction" (resolved simulations where the agent call matched the outcome), "Most shared" (highest share page view count). This creates an in-product discovery layer that answers the visitor's question "what can MiroShark actually do?" with real examples, not screenshots. For multi-user instances (cloud deploy with teams), it creates a metagame — "who ran the most interesting simulation this week?" Each card links to the public share page, turning the hall of fame into an acquisition funnel: a visitor who finds "EU AI Act: 83% agents predicted rejection — and they were right" is already sold. For single-user instances, it's a personal portfolio of best runs.
**How:**
1. Add a `GET /api/hall-of-fame` endpoint that queries the local simulation database and returns the top 3 simulations in each of 4 categories: (a) Most surprising — `ORDER BY ABS(final_yes_price - 0.5) DESC` (farthest from 50/50 consensus); (b) Most polarized — `ORDER BY polarization_index DESC` (requires polarization_index to be stored — compute on-the-fly if not present for older runs by approximating from belief drift variance); (c) Most accurate — `WHERE resolution IS NOT NULL ORDER BY accuracy_score DESC LIMIT 3`; (d) Most viewed — join with share_simulations table, `ORDER BY view_count DESC LIMIT 3`. Return up to 12 entries total (deduped — a simulation can only appear in one category).
2. Build a `HallOfFame.vue` page at `/explore` with 4 section tabs: "Surprising", "Polarized", "Accurate", "Popular". Each simulation card shows: scenario title, a mini belief-drift sparkline (if available), final market price badge, agent count and round count, resolution outcome if resolved (✓ Correct / ✗ Incorrect), and a "View →" button that opens the public share page. If `share_token` doesn't exist yet for a simulation, auto-generate one when it appears in the hall of fame (so all hall-of-fame entries are always shareable). Add a "Refresh rankings" button that re-computes the ranking on demand.
3. Add a "Hall of Fame" link to the main navigation (between History and Settings). On the home screen (Step 1), add a subtle "Explore past simulations →" link below the main action buttons that routes to `/explore` — this is the entry point for visitors who arrive at a cloud instance and want to understand what the tool does before running their own simulation. Add an `exclude_from_hall_of_fame` checkbox to the simulation settings modal (for users who want to run private simulations on a shared instance).

## Selection Rationale

This batch targets MiroShark's transition from single-run demo tool to production-grade simulation platform:

- **Statistical batch runs** (#1) — The single feature that makes MiroShark's outputs statistically defensible. One run is a story; N runs with variance is evidence. Unlocks academic citation and high-stakes practical simulation.
- **Push notifications** (#2) — Highest-ROI DX improvement available: 2–4 hours to implement, directly prevents the most common failure mode for cloud deploy users (abandoned long-running tabs). Service Worker + Push API requires no external service.
- **Jupyter export** (#3) — Bridges MiroShark's visual UI to Python research workflows. Researchers who want to publish need raw DataFrames, not screenshots. A self-contained notebook with pre-written analysis cells removes the last barrier to academic adoption.
- **REST API + webhook** (#4) — The composability unlock. Turns MiroShark from a standalone app into a component in any automation stack — Aeon, GitHub Actions, Zapier, researcher scripts. API key + HMAC-signed webhooks is the right trust model for an open-source self-hosted tool.
- **Hall of Fame** (#5) — Creates a discovery layer for multi-user instances and answers the "what can this do?" question with real examples rather than documentation. Every hall-of-fame card is a public share link — extends the share-permalink growth mechanic into a browsable gallery without requiring user-submitted curation.

Each idea is scoped for autonomous implementation by the `feature` skill — clear inputs/outputs, no ambiguous design decisions, no external approvals needed.
