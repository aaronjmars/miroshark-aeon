## Ecosystem Context

The surface-multiplication arc reached ten surfaces last week. The question this batch asks is: *what serves the next phase of growth — from 1K to 5K stars?*

Three signal threads drive this batch:

- **The reproducibility gap.** MiroShark is being positioned as a research substrate — six of the ten share surfaces (transcript, trajectory, thread, watch, GIF, share card) are designed to make simulation results citable and shareable. But there's no way to reproduce an exact run. PR #71's `?scenario=...` URL carries the scenario text but not the model config, agent count, rounds, or preset. A researcher who wants to validate Aaron's Aave risk call needs to guess the parameters. A `config.json` export — machine-readable, complete, importable — turns a MiroShark result into a reproducible experiment. This is what the academic and quant audiences need before they can cite the tool seriously.

- **The programmatic access gap.** The OpenAPI spec lives at `/api/docs` and passes drift-detection tests on every PR. But there's no SDK — a researcher who wants to start simulations from a Python script, poll for completion, and download the trajectory CSV has to hand-write requests against the spec. `openapi-generator-cli` can produce a Python client from the existing spec in one CI job. Published as a GitHub Release artifact, it closes the last barrier between MiroShark and the Jupyter-notebook / quant-pipeline audience.

- **The attribution blindspot.** PR #73 adds delivery visibility for the outbound webhook. But there's still no way to know how many times the share card was served, how many RSS subscribers fetched the feed, or how many people copied the thread. An operator running MiroShark for a DeFi fund has no signal on which surfaces their audience actually uses. A per-surface request counter — one atomic increment per `_serve_*` handler — closes this gap with zero new deps and zero architectural change.

Two additional ideas round out the batch:

- **The director-mode annotation gap.** Director mode can inject scenario events mid-simulation ("Liquidity Crisis at round 15"). The belief chart shows the resulting curve but not the cause — there's no visual marker at the injection point. An annotated chart (vertical dashed lines with event labels) turns the belief curve from raw data into a causal narrative. This is the "show your work" surface that director-mode has always needed.

- **The parameter sensitivity gap.** Running the same scenario twice with one variable changed — agent count, preset, research depth — is the most common question power users ask. But comparing two results requires opening two tabs and mentally diffing the belief bars. A `/compare?a=<id>&b=<id>` page with a delta view (which stance moved, by how much) makes parameter sensitivity analysis a product feature rather than a manual exercise.

### 1. Simulation Config Export + Reproducibility Badge

**Type:** DX / Research
**Effort:** Small (hours)
**Impact:** There is currently no way to reproduce an exact MiroShark run. PR #71's shareable scenario URL carries the scenario text and template slug but not the model configuration, agent count, rounds, or research depth — the parameters that determine the simulation's behavior. A researcher citing Aaron's Aave risk sim needs to guess or manually inspect to reproduce it. `GET /api/simulation/<id>/config.json` exports a complete machine-readable config blob: scenario, agent_count, total_rounds, preset_name, model list (from Wonderwall config), research_depth, template_slug (if started from a preset), and director_events (if any were injected). `POST /api/simulation/from-config` accepts this blob and starts a new simulation with identical parameters — the backend already assembles a config object at sim start; this is a round-trip of that object. The EmbedDialog gets an "Export config" section with a download button and a `curl -X POST /api/simulation/from-config -d @config.json` snippet. Sims started via `from-config` get a `{reproduced_from: <original_id>}` marker in `sim_dir/provenance.json`; the share page and watch page display a "🔁 Reproducible" badge when `provenance.json` exists. This is the primitive the academic and quant audiences need before they can cite MiroShark in a paper or thread without qualification.
**How:**
1. Add `backend/app/services/config_export.py` (~180 LoC, pure stdlib): `export_sim_config(sim)` assembles `{schema_version: "1", exported_at: ISO8601, simulation_id: str, scenario: str, agent_count: int, total_rounds: int, preset_name: str, model_config: [...], research_depth: str, template_slug: str|null, director_events: [...]|null}` from the `SimulationState` + Wonderwall config + on-disk `director-events.json` (graceful fallback to `null` when missing — not all sims use director mode). Add `import_sim_config(config_dict)` that validates the schema version, clamps numeric params to their existing min/max bounds, and returns a `SimulationConfig` dict that the existing `POST /api/simulation/new` handler can consume. Add `_validate_config_schema(config_dict)` that type-checks every field and returns `(valid: bool, errors: [str])` — gives callers a clean error surface without raising. 15 offline unit tests in `test_unit_config_export.py`: round-trip (export then import produces the same agent_count/rounds), missing director-events degrades to `null`, model_config is exported in stable key order (for diff-friendliness), schema version field present, import rejects unknown schema version, import clamps agent_count above max to max (no silent truncation — returns error), import rejects negative rounds, director_events preserved when present, `exported_at` is ISO8601 UTC, scenario text preserved verbatim (no truncation), `template_slug` is `null` for sims not started from a template, export of a reproduced sim includes the original `simulation_id` not the child's, provenance.json written on `from-config` start, route-decorator presence for GET and POST, drift-detection guard.
2. Add `GET /api/simulation/<id>/config.json` (no auth required on read — config is public for published sims; 403 for unpublished, same gate as all other share surfaces). Add `POST /api/simulation/from-config` (no auth required — anyone can start a sim, same as `POST /api/simulation/new`): accepts `Content-Type: application/json` body, calls `import_sim_config`, validates, delegates to the existing sim-creation path. Returns `{simulation_id: str}` on success, `{errors: [...]}` on validation failure. Add both paths to `openapi.yaml` under Publish & Embed — drift-detection test passes. Add `getConfigJsonUrl(simId)` + `startSimFromConfig(configBlob)` helpers to `frontend/src/api/simulation.js`.
3. In `EmbedDialog.vue`, add an "Export config" section beneath the thread row (publish-gated). Layout: a "Download config.json" button (`<a :href="configJsonUrl" download="config.json">` — no network round-trip, browser handles it), a copy-ready `curl` snippet, and a note: "Anyone with this config can reproduce this exact simulation — agent count, rounds, model preset, all parameters." Add a `<ProvenanceBadge />` component (micro-sized, inline) that reads `provenance.json` on mount and renders "🔁 Reproduced from [original_id]" when present; mount it on the share page (`/share/<id>`) and watch page (`/watch/<id>`) beneath the scenario text. Add "Config Export & Reproducibility" row to `docs/FEATURES.md` under Publish & Embed. Add `POST /api/simulation/from-config` to `docs/API.md` with the config schema reference. Update README Features table.

### 2. Python Client SDK via openapi-generator CI

**Type:** Integration / DX
**Effort:** Small (hours)
**Impact:** The OpenAPI spec at `/api/docs` passes drift-detection tests on every PR. But researchers who want to integrate MiroShark into a Python pipeline — start sims from a notebook, poll for completion, download the trajectory CSV, parse the thread JSON — have to hand-write HTTP requests against the spec. `openapi-generator-cli generate -g python` produces a fully typed Python client from the existing spec in one CI job: all endpoints, all request/response models, retry logic, authentication headers, and a `README.md` with usage examples. Generated in CI, committed to `client/python/`, published as a GitHub Release asset on every main-branch merge that touches `backend/openapi.yaml`. No new secrets required — `GITHUB_TOKEN` handles both commit and release creation. The generated client is importable immediately: `from miroshark import ApiClient, SimulationApi; api = SimulationApi(ApiClient(configuration)); api.start_simulation(body=...)`. This closes the programmatic-access gap without writing any SDK code — the spec is already doing the work; the CI job just runs the generator. With the 5K-star thesis circulating and the Bankr Terminal v2 Aave-sim citation still producing inbound, researchers who want to integrate MiroShark into a quant pipeline now have a path that doesn't require reading the raw OpenAPI YAML.
**How:**
1. Add `.github/workflows/generate-client.yml`: triggers on `push` to `main` with `paths: ['backend/openapi.yaml']` (path filter so it only runs when the spec actually changes — avoids CI noise on unrelated commits). Steps: checkout, pull `openapitools/openapi-generator-cli:latest` Docker image (no install needed), run `docker run --rm -v ${PWD}:/local openapitools/openapi-generator-cli generate -i /local/backend/openapi.yaml -g python -o /local/client/python/ --additional-properties=packageName=miroshark,projectName=miroshark-client,packageVersion=0.1.0`. Commit the result: `git add client/python/ && git commit -m "chore(sdk): regenerate python client from openapi spec" && git push`. Create a GitHub Release asset: `gh release create sdk-$(date +%Y%m%d) client/python/ --title "Python SDK $(date +%Y-%m-%d)" --generate-notes` (uses GITHUB_TOKEN, no extra secrets). Add `client/python/**` to `.gitignore` exclusion list (generated files shouldn't fail linting). Add `client/python/` to `gitattributes` with `linguist-generated=true` so GitHub doesn't count the generated code against language stats.
2. Add `client/python/README.md` (hand-authored, not generated — openapi-generator's default README is sparse): quickstart section with four code blocks: (a) install from GitHub Release (`pip install https://github.com/aaronjmars/MiroShark/releases/latest/download/miroshark-client.tar.gz`), (b) start a simulation (`from miroshark import ApiClient, SimulationApi, StartSimulationRequest; ...`), (c) poll for completion (loop on `GET /api/simulation/{id}/status` until `run_complete`), (d) download trajectory CSV (`response = api.get_trajectory_csv(sim_id); open("trajectory.csv","wb").write(response.data)`). Add a "Jupyter notebook" section with a one-cell quickstart that runs a full sim end-to-end. Add API key / base URL configuration section (the client uses `Configuration(host=..., api_key={...})` — mirrors the standard openapi-generator pattern).
3. In the main `README.md`, add a "Python client" section under the API row in the Features table: "Generated from the OpenAPI spec — `pip install` from the latest GitHub Release." In `docs/API.md`, add a "Python client" section at the top that cross-links the generated `client/python/README.md`. Add a note in the OpenAPI Swagger UI description (in `openapi.yaml`'s `info.description`) pointing to the Python client for programmatic consumers. No new backend code — the entire feature is CI configuration and documentation.

### 3. Director Event Timeline Overlay on Belief Chart

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Director mode lets an operator inject scenario events mid-simulation — a "Liquidity Crisis" at round 15, a "Regulatory Announcement" at round 30. The belief chart in SimulationView shows the resulting belief curve, but not the cause: there is no visual marker at the injection point. A researcher looking at a sharp belief shift can't tell whether it was driven by agent reasoning or an operator-injected event. Adding vertical dashed annotation lines to the belief chart — one per director event, labeled with the event name and round number — turns the chart from a raw belief curve into a causal narrative: "the bullish surge at round 15 was triggered by a Liquidity Crisis injection." The data is already on disk (director events are persisted in simulation state); this is a read-and-annotate operation, not new data generation. A new `GET /api/simulation/<id>/director-events` endpoint returns the event list for the frontend; the belief chart gains a Chart.js annotation layer using the existing `chartjs-plugin-annotation` dependency (check if already pinned — if not, a pure-SVG overlay on top of the canvas avoids any new dep). EmbedDialog gets a "Director events" panel listing events with round numbers. The director-mode use case — which Aaron highlighted in docs as a "director mode for narrative control" — finally has a distribution artifact: the annotated chart screenshot is more citation-worthy than the raw belief curve.
**How:**
1. Add `GET /api/simulation/<id>/director-events` to `backend/app/api/simulation.py` (no auth required, public for published sims; 403 for unpublished; 200 with `{events: []}` empty array when no director events were injected — graceful non-error for the majority of sims that don't use director mode). Read director events from their on-disk location (inspect `SimulationState` and `sim_dir/` for the exact field — candidates: `simulation_state.json`'s `director_events` field, or a dedicated `director-events.jsonl`). Each event in the response: `{round: int, label: str, description: str|null, injected_at: ISO8601}`. Add `GET /api/simulation/{id}/director-events` to `openapi.yaml` under Analytics with a `DirectorEvent` schema — drift-detection test passes. Add `getDirectorEvents(simId)` helper to `frontend/src/api/simulation.js`. 10 offline unit tests in `test_unit_director_events.py`: sim with no director events returns `{events: []}`, sim with 2 events returns both in round order, corrupt director-events source degrades to empty list (not 500), publish gate 403, round field is integer, label field is non-empty string, injected_at is ISO8601, route-decorator presence guard, openapi drift guard, response envelope always has `events` key even on error paths.
2. In `SimulationView.vue`, fetch `GET /api/simulation/<id>/director-events` on mount (alongside the existing belief chart data fetch). When the response contains at least one event, add annotation vertical lines to the belief chart using `chartjs-plugin-annotation` (check if already a dependency — if yes, add `annotations: {}` to the chart config with one `line` annotation per event, `xValue: event.round`, `borderColor: "#f59e0b"` amber, `borderWidth: 2`, `borderDash: [4, 4]`, `label: { content: event.label, enabled: true, position: "start" }`). If `chartjs-plugin-annotation` is not already installed, implement a pure-SVG overlay instead: an absolutely-positioned `<svg>` layer over the canvas element with `<line>` elements at the correct x positions (computed from the chart's x-axis scale — `chart.scales.x.getPixelForValue(round)`), each with a `<text>` label rotated 90°. Either approach avoids new npm dependencies if the plugin is already pinned. Annotations only render when the director-events response is non-empty — no change to sims without director events. Add an orange "⚡ Director events" indicator chip in the SimulationView header when events are present (same chip style as the existing status chips), with a click handler that scrolls to the belief chart.
3. In `EmbedDialog.vue`, add a "⚡ Director events" section (only visible when events array is non-empty, fetched with the same API call as above). Layout: a compact timeline list — `Round N: [label]` rows with a small amber ⚡ bullet — so an operator can quickly scan what was injected. Add "Director Event Timeline" row to `docs/FEATURES.md` under Simulation Controls. Add a note to `README.md` Director Mode section: "Director events are annotated on the belief chart — see exactly which injection drove which belief shift." No new deps if `chartjs-plugin-annotation` is already pinned (check `package.json`); SVG overlay fallback costs zero deps regardless.

### 4. Share Surface Usage Analytics

**Type:** DX / Feature
**Effort:** Small (hours)
**Impact:** PR #73 adds delivery visibility for the outbound webhook — operators can now see whether the webhook fired and what HTTP status it returned. But there is still no equivalent visibility for the inbound share surfaces: how many times was the share card served? How many RSS subscribers fetched the feed today? How many people copied the thread? An operator running MiroShark for a DeFi fund publishes simulation results across ten surfaces and has zero signal on which ones their audience actually uses. A per-surface request counter — one atomic increment in each `_serve_*` handler, written to `sim_dir/surface-stats.json` — costs almost nothing architecturally and closes this gap entirely. `GET /api/simulation/<id>/surface-stats` (no auth required, public for published sims) returns `{share_card: N, replay_gif: M, transcript_md: P, transcript_json: Q, trajectory_csv: R, trajectory_jsonl: S, thread_txt: T, thread_json: U, watch_page: V, rss_atom: W, total: X}`. The EmbedDialog "Distribution" panel shows the counts as a compact table — "Share card · 42 serves · Replay GIF · 17 serves · Thread · 8 copies." This is the first operator-side feedback loop on distribution: the operator learns whether to invest time in Twitter threads or in the RSS channel for their specific audience. Zero new deps, minimal code change per handler, no architectural shift.
**How:**
1. Add `backend/app/services/surface_stats.py` (~120 LoC, pure stdlib `json` + `os`): `SURFACE_KEYS = frozenset({"share_card", "replay_gif", "transcript_md", "transcript_json", "trajectory_csv", "trajectory_jsonl", "thread_txt", "thread_json", "watch_page", "rss_atom"})`. `surface_stats_path(sim_dir)` returns `<sim_dir>/surface-stats.json`. `increment_surface_stat(sim_dir_path, surface_key)` — reads current stats from disk (empty dict if file missing), increments the key (defaulting to 0), writes back via atomic rename (`os.replace`) so concurrent requests can't corrupt the file. `read_surface_stats(sim_dir_path)` returns `{key: count, ..., total: sum}` with all `SURFACE_KEYS` present (zero-defaulted for keys not yet recorded). Increment is fire-and-forget — called inside each `_serve_*` handler after the response is constructed (same position as the existing `Cache-Control` header set), so it never blocks the response path. 12 offline unit tests in `test_unit_surface_stats.py`: increment from zero creates file, increment on existing increments correctly, all SURFACE_KEYS zero-defaulted in read, atomic write (simulate concurrent increment by calling twice in the same process — both increments land), `total` field equals sum of all counts, unknown key rejected with ValueError, missing sim_dir raises gracefully (no unhandled exception — returns zero stats), corrupt JSON file resets to zeros (log warning, don't 500), `os.replace` is used (not `open(..., "w")` — verified by mock), increment returns nothing (fire-and-forget), route-decorator presence, drift guard.
2. Add `increment_surface_stat(sim_dir_path, "share_card")` call in `_serve_share_card_png()`, `"replay_gif"` in `_serve_replay_gif()`, `"transcript_md"` / `"transcript_json"` in `_serve_transcript()`, `"trajectory_csv"` / `"trajectory_jsonl"` in `_serve_trajectory()`, `"thread_txt"` / `"thread_json"` in `_serve_thread()`, `"watch_page"` in the watch endpoint, `"rss_atom"` in the feed endpoint (once per request, regardless of Atom vs RSS format — both are a "feed serve" from the distribution perspective). Each increment happens after `sim_dir_path` is resolved and confirmed to exist — no-op if `sim_dir_path` is falsy (private/missing sims don't get counted). Add `GET /api/simulation/<id>/surface-stats` route (no auth required, publish-gated — 403 for unpublished sims): calls `read_surface_stats(sim.sim_dir)`, returns the stats dict. Add path to `openapi.yaml` under Publish & Embed with a `SurfaceStats` schema — drift-detection test passes. Add `getSurfaceStats(simId)` helper to `frontend/src/api/simulation.js`.
3. In `EmbedDialog.vue`, add a "📊 Distribution" section (publish-gated, collapsed by default — same disclosure chevron pattern as other EmbedDialog sections). Fetch `GET /api/simulation/:id/surface-stats` when expanded. Render as a two-column compact table: surface name (left) + count (right). Sort by count descending so the operator's most-used surface is at the top. Show "Total serves: N" in the section header. Add a "Refresh" button (same pattern as the webhook log panel). If all counts are zero (sim just published), show "No surface serves recorded yet — share this simulation to see distribution data." Add "Surface Usage Analytics" row to `docs/FEATURES.md` under Publish & Embed. Add the endpoint to `docs/API.md`. Update README Features table.

A "Copy comparison link" button (copies `window.location.href`, same clipboard + toast pattern as other copy buttons). ## Selection Rationale

This batch targets the layer *above* the ten completed share surfaces: reproducibility, programmatic access, distribution analytics, causal annotation, and comparative analysis.

- **Simulation Config Export + Reproducibility Badge** (#1) — The researcher who wants to cite MiroShark in a paper needs to be able to reproduce the exact run. PR #71 carries scenario text; it doesn't carry model config, agent count, or rounds. A `config.json` round-trip closes this gap and makes "reproducible experiment" part of the product identity rather than a promise. Pure stdlib, follows existing `_serve_X` pattern. High leverage for the academic and quant audiences that the 5K-star thesis depends on.
- **Python Client SDK via openapi-generator CI** (#2) — The OpenAPI spec already exists and passes drift-detection tests on every PR. A CI job that runs `openapi-generator-cli` turns it into a Python package without writing SDK code. Published as a GitHub Release artifact (no new secrets), immediately usable in Jupyter notebooks. Closes the programmatic-access gap that the researcher audience hits after they read the API docs.
- **Director Event Timeline Overlay** (#3) — Director mode is one of MiroShark's differentiating features — no other simulation tool lets an operator inject narrative events mid-run. But the belief chart doesn't show *where* those injections happened. Annotating the chart with vertical markers closes the "what caused that shift?" question and makes director-mode results more citation-worthy. Small effort; zero new deps if `chartjs-plugin-annotation` is already pinned (SVG fallback otherwise).
- **Share Surface Usage Analytics** (#4) — PR #73 closes the outbound webhook observability gap. The inbound share surface observability gap has had no attention. One atomic counter increment per `_serve_*` handler, one read endpoint, one EmbedDialog panel. The first operator-side feedback loop on which surfaces their audience actually uses. Zero new deps; minimal code change per handler.
- **Comparative Run View** (#5) — The parameter sensitivity use case — "what changed when I tweaked agent count from 12 to 24?" — is the most common power-user question after seeing a result. A `/compare?a=<id>&b=<id>` page with belief bar delta requires zero new backend code (uses the existing embed-summary endpoint) and produces a shareable artifact: a URL that shows the before/after of any parameter change. Pure frontend, fully testable, zero new deps.

Each idea is scoped for autonomous implementation by the `feature` skill — all data is on disk or in existing endpoints, no ambiguous design decisions, no external approvals required.
