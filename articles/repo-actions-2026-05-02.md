### 1. 1-Click Cloud Deploy (Fly.io + Railway)

**Type:** DX / Growth
**Effort:** Small (hours)
**Impact:** 976 stars, 4 contributors — the gap between "starred" and "running" is wide. The current install path requires Python ≥3.11, Node 18, Neo4j 5.x, `.env` configuration, and an OpenRouter key before seeing the first simulation. A "Deploy to Railway" button in the README and a `fly.toml` for Fly.io close that gap to a single click for operators who just want to run the tool without reading the full install guide. Railway's zero-config container detection handles the Python backend automatically; a `railway.toml` with a `[build]` section pointing at `Dockerfile` (already present for Docker Compose installs) is all that's needed. Fly.io targets the slightly more technical audience who want a persistent deployment. Both paths benefit from a `docker-compose.prod.yml` that wires the backend, Neo4j, and a minimal frontend nginx together behind environment-variable injection. With 24 stars to go before 1K, this is the single highest-leverage friction-reduction move: every fork that turns into a live deployment is a potential contributor and community amplifier.
**How:**
1. Add `fly.toml` to the repo root: `app = "miroshark"`, `primary_region = "iad"`, `[build] dockerfile = "Dockerfile"`, `[[services]] internal_port = 5000 protocol = "tcp"`. Add `[[services.ports]] port = 443 handlers = ["tls", "http"]` for HTTPS. Add a `[env]` section with safe defaults (`FLASK_ENV = "production"`, `LOG_LEVEL = "info"`) and a comment block listing the required secrets (`flyctl secrets set OPENROUTER_API_KEY=... NEO4J_URI=... ...`). Add `[mounts] source = "miroshark_data" destination = "/app/data"` so simulation artefacts survive deploys. Update `.gitignore` to exclude `.fly/`. Add `railway.toml` with `[build] builder = "dockerfile"` and `[deploy] startCommand = "python -m gunicorn app:app --bind 0.0.0.0:$PORT"`. Both files land in the repo root alongside the existing `docker-compose.yml`.
2. Add `docker-compose.prod.yml` for operators who prefer Compose in production. Differences from the dev `docker-compose.yml`: pulls pre-built image from a public registry (or builds from the repo), Neo4j uses a named volume, backend reads secrets from a `.env.prod` file (excluded from git), Nginx reverse-proxy service (`nginx:1.27-alpine`) terminates TLS and proxies `/api/*` to the backend and `/*` to the built frontend. Add an `nginx.conf.template` that honours `$SERVER_NAME` and `$BACKEND_PORT` env vars.
3. Add a `DEPLOY.md` (one page, not a wall of text) covering: Railway (button + required env vars + Neo4j add-on from Railway's marketplace), Fly.io (three-command flow: `fly launch`, `fly secrets set`, `fly deploy`), and self-hosted Docker Compose prod. In `README.md`, replace the current "self-hosted" heading with a "Deploy in one click" section: Railway badge (`[![Deploy on Railway](https://railway.app/button.svg)](https://railway.app/new/template?...)`) + Fly.io badge + link to `DEPLOY.md`. Update `docs/INSTALL.md` to add a "Cloud deploy" section that cross-links `DEPLOY.md`. The Railway template URL is assembled from the repo's public URL + required env var names (Railway supports this in the template URL schema — no Railway account required to generate the link).

### 2. Gallery Full-Text Search & Consensus Filter

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The `/explore` gallery shows the 20 most recent public simulations in reverse-chronological order. For a researcher who wants to find all bearish-consensus sims about "ETH staking," or filter to verified predictions only, the current interface is a scroll. Adding keyword search (against scenario text and quality tags) and consensus/outcome filters turns the gallery from a stream into a research corpus — directly supporting the "simulation as research substrate" positioning MiroShark has been building toward. The existing `GET /api/simulation/public` endpoint already supports `?verified=1`; extending it with `?q=<keyword>&consensus=bullish|bearish|neutral&quality=excellent|good&sort=quality|date&page=<n>` costs almost nothing: all data is on disk as JSON, an in-memory scan of up to a few hundred sims completes in milliseconds. Frontend gets a search bar + filter chips in the ExploreView header. The result is a surface that researchers, quant analysts, and community members can bookmark and share — "all verified bearish calls on Aave" becomes a URL you can tweet.
**How:**
1. Extend `GET /api/simulation/public` with new optional query params: `q` (string, case-insensitive substring match against `scenario` in `simulation_config.json`), `consensus` (`bullish`|`bearish`|`neutral` — filter cards where `consensus_label` matches), `quality` (`excellent`|`good`|`acceptable`|`poor` — filter by quality health tier), `sort` (`date` default, `quality` sorts by quality score descending, `stars` sorts by engagement descending for future leaderboard use), `page` (int, 1-based, default 1), `limit` (int, max 50, default 20). In `_get_public_simulations()`, apply filters before pagination so `total_count` in the response reflects the filtered set. Add `{total_count, page, limit, has_next, q, consensus, quality}` to the response envelope. Graceful degradation: if `simulation_config.json` is missing for a sim, treat scenario as empty string and skip `q` filter (sim remains visible unless `q` is set). Add `?q` + `?consensus` + `?quality` to `openapi.yaml` under the `/api/simulation/public` path — drift-detection test passes. 17 offline unit tests covering: empty query returns all, `q` substring match is case-insensitive, `consensus` filter removes non-matching cards, `quality` filter excludes lower tiers, combined `q`+`consensus` intersection, pagination `page=2` returns correct slice, `total_count` reflects filtered count not total corpus, missing `simulation_config.json` skips gracefully, `sort=quality` orders descending, `sort=date` is default order, `limit` cap at 50, `has_next` is false on last page, `page=0` coerces to page 1.
2. In `ExploreView.vue`, add a search + filter bar above the sim cards. Layout: full-width search input (`placeholder="Search scenarios..."`) with a magnifying-glass icon, debounced 300ms, updates `?q=` URL param on change. Below the input: a row of filter chips — `All` (default, active chip style) / `🟢 Bullish` / `🔴 Bearish` / `⚫ Neutral` — clicking a chip appends `?consensus=` to the URL and re-fetches. A second chip row for quality: `All` / `Excellent` / `Good` / `Acceptable`. A sort dropdown: `Date ↓` / `Quality ↓`. Chips and search state are kept in URL params so a filtered view is bookmarkable and shareable. Add a "N results" count line beneath the filter bar when a filter or query is active. Existing pagination controls move to the bottom. If `total_count = 0`, show a "No simulations match your filters — try broadening the search" empty state instead of the default empty-gallery copy. The `/verified` route preserves the existing `verifiedOnly: true` behaviour and is compatible with the new filter params (i.e., `/verified?q=aave` works).
3. Add a "🔍 Search the gallery" section to `docs/FEATURES.md` under Public Gallery. Update `README.md` gallery row to mention search and filtering. Add a URL example to `docs/API.md` — `GET /api/simulation/public?q=aave&consensus=bearish&quality=excellent&sort=quality` — showing the filter composition pattern for programmatic consumers. No new dependencies (all in-memory filtering of existing on-disk JSON).

### 3. Pre-Run Cost & Time Estimator Widget

**Type:** DX / Feature
**Effort:** Small (hours)
**Impact:** The GitHub description and README both promise "$1 & under 10 min." But when a user opens the New Sim form and is about to click Run, they see no cost or time estimate — the promise is invisible at the moment of truth. A `~$0.40 · ~7 min` estimate line next to the Run button closes the loop: the marketing claim becomes a real-time widget that updates as the user tweaks rounds, agent count, and preset. The formula is straightforward (model cost-per-token × avg-tokens-per-round-per-agent × rounds × agent-count) and the required constants are already available — PR #51 (Langfuse) instruments every OpenRouter call, PR #52 documents the avg cost profile of the Cheap and Full presets. This directly reduces abandonment from users who worry about unexpected LLM bills before they've seen a result. Small effort: the estimate logic lives entirely in the frontend and reads from a static `PRICING_CONSTANTS` object — no new API endpoint needed.
**How:**
1. Add `frontend/src/data/pricing_constants.json` containing the pricing reference table: one entry per model (model ID as key; `{cost_per_million_input_tokens, cost_per_million_output_tokens, avg_tokens_per_round_per_agent}` as value). Populate from OpenRouter's public pricing page for the models in the Cheap preset (Qwen-2.5-72B, DeepSeek-R1, Grok-3-Mini) and the Full preset (Claude Sonnet, GPT-4o, Gemini Pro). The `avg_tokens_per_round_per_agent` values come from the Langfuse instrumentation logs (PR #51 already captures per-call token counts in trace metadata). Add a `RESEARCH_DEPTH_COST` multiplier (0.2 for light, 0.5 for standard, 1.0 for deep — correlates to the existing research depth setting). Formula: `est_cost = sum_over_agents(avg_tokens_per_round * rounds * token_cost) + research_depth_cost`. `est_time = rounds * avg_seconds_per_round` where `avg_seconds_per_round` is ~12s for Cheap preset, ~25s for Full preset (derived from Langfuse P50 latency). Cap display precision to 2 sig figs (`~$0.43`, never `$0.4284`). Mark as "est." to communicate approximation. Update `pricing_constants.json` when new presets are added.
2. Add a `useRunEstimate(config)` Vue composable in `frontend/src/composables/useRunEstimate.js`. Inputs: `{ preset, model_overrides, num_agents, default_rounds, research_depth }`. Output: `{ est_cost_usd, est_minutes, label }` — `label` is the formatted string `"~$X.XX · ~N min"`. The composable reads `pricing_constants.json` statically (import at build time), computes the estimate reactively whenever any input changes. If a model is not in the pricing table (e.g., a custom Wonderwall endpoint), return `{label: "Cost varies (custom endpoint)"}` and skip the estimate display. Add a `<RunEstimate />` component that renders the `label` in a muted subtext style (12px, `color: var(--text-muted)`) — visible but not louder than the Run button itself.
3. Mount `<RunEstimate />` in the New Sim form immediately below the "Run Simulation" button. Also show it in the simulation quick-start widget on the landing page. When the estimate crosses `$1.00`, show a soft amber color and a `(Full preset)` note — the "Cheap preset is under $1" vs "Full preset may cost more" distinction maps directly to the existing preset naming. Add a `<RunEstimate />` to the scenario template cards so the cost is visible before the user even opens the New Sim form. No new backend, no new dependencies. Update `docs/FEATURES.md` with a "Cost & Time Estimate" row under New Simulation.

### 4. Per-Agent Stance Sparklines (Character Arc View)

**Type:** Feature / Research
**Effort:** Small (hours)
**Impact:** PR #66 gave analysts the per-round *aggregate* stance split (bullish%, neutral%, bearish% of all agents each round). But the research question "which agents were persuaded during the simulation, and when?" requires the per-agent cut: a stance time-series for each individual agent across rounds. This is the data behind "agent 7 started bullish, flipped bearish at round 12, and held until the end" — the kind of finding that makes a MiroShark result citable in a multi-agent systems paper. The data already exists in `trajectory.json` (per-round agent posts include a stance score). One new endpoint `GET /api/simulation/:id/agent-arcs` returns the per-agent series, and a sparklines panel in SimulationView renders it as a compact table of mini belief-score charts — one row per agent, one sparkline per row. Zero new deps (sparklines rendered with SVG paths, no chart.js). Follows the `_serve_X` shared-body pattern. Adds a new dimension to the "seven surfaces / one folder" arc: the quantitative per-round aggregate (trajectory CSV) paired with the qualitative per-agent narrative (transcript).
**How:**
1. Add `GET /api/simulation/:id/agent-arcs` — publish-gated (same `is_public` gate as trajectory endpoints). Read `trajectory.json`. For each recorded round, extract the list of agent posts and their stance scores. Build a dict keyed by agent ID: `{agent_id: {name, archetype, rounds: [{round: N, stance: float, post_excerpt: str}]}}`. Stance score is the raw float from the agent's final post in that round (the same value the ±0.2 threshold thresholds into bullish/neutral/bearish). If an agent has no post in a round, insert `{round: N, stance: null}` — an explicit null so the sparkline can render a gap rather than interpolating. Sort rounds chronologically. Resolve agent names from `reddit_profiles.json` or `polymarket_profiles.json` (same resolution order as the transcript service). Return `{simulation_id, total_rounds, agents: [...]}` with `Cache-Control: public, max-age=60`. `_serve_agent_arcs(simulation_id)` shared body, single route decorator. 15 offline unit tests: stance series for a 3-round, 3-agent fixture; null-insertion for absent rounds; name resolution across profile types; corrupt trajectory graceful degradation; publish gate 403; route-decorator presence guard; `total_rounds` matches max round number in the series; empty trajectory returns empty agents list; single-agent sim; agent with no posts in any round returns full null series; stance scores are floats not strings; sorted round order even when on-disk snapshots are out-of-order; `post_excerpt` truncated to 120 chars; missing profiles degrade to `agent_id` as name; openapi drift test.
2. Add an "Agent Arcs" tab in `SimulationView.vue` alongside the existing tabs (Report, Interaction, Demographics, Observability). On tab activation, fetch `GET /api/simulation/:id/agent-arcs`. Render a compact table: one row per agent, with columns `Agent (name + archetype badge)`, `Final stance (pill)`, and `Rounds (sparkline)`. The sparkline is a 120×24px inline SVG: a polyline through `(round, stance)` points, clipped to `[-1, 1]` range, with a horizontal zero-line at y=0. Points with `stance > 0.2` render in bullish-green, `< -0.2` in bearish-red, else neutral-grey. Gap (null stance) is rendered as a break in the polyline. Hovering a sparkline shows a tooltip: "Round N: [post_excerpt]" — helps analysts read the arc without opening the full transcript. Add a sort control: sort by `Final stance` (most bullish → most bearish), `Volatility` (std dev of stance across rounds, descending), or `Name`. Add a "📥 Download arcs JSON" button that triggers a direct download of the raw `agent-arcs` response.
3. Add `GET /api/simulation/{simulation_id}/agent-arcs` to `openapi.yaml` under Analytics with a `SimulationAgentArc` schema (per-agent series shape). Drift-detection test passes. Add an "Agent Character Arcs" row to `docs/FEATURES.md` under Analytics. Add a note to `docs/API.md` explaining the per-agent stance series and its relationship to the trajectory CSV (aggregate vs. per-agent). Update README Features table with an "Agent Stance Arcs" row.

### 5. Pre-filled Scenario URL (Shareable Scenario Template Link)

**Type:** Growth / DX
**Effort:** Small (hours)
**Impact:** Aaron's "he used MiroShark" marketing pattern on X is pointing at the repo — but every link drops the viewer at the homepage with a blank New Sim form. The "try this yourself" CTA has maximum friction: you have to write the scenario from scratch. A `/new?scenario=...&preset=Cheap&rounds=30&agents=12` pre-fill URL changes the CTA from "go to the site and figure it out" to "click here and it's already filled in — just hit Run." Any blog post, tweet thread, or Discord message about a specific scenario can include a shareable URL that drops the reader directly into a pre-configured sim. The "Fork this scenario" button in PR #67's watch page does this for *finished* sims. This is the same affordance for *new* (un-run) scenarios: a URL that anyone can share to invite others to run the exact same setup. With Aaron actively promoting specific use cases (the WWII talkie demo, the Aave risk sim, the "better than deep research" framing), this is the missing link between the marketing and the product.
**How:**
1. In `frontend/src/views/SimulationRun.vue` (or wherever the New Sim form lives), add URL parameter parsing on mount. Read `scenario`, `preset`, `rounds`, `agents`, `research_depth`, and `template` (template slug) from the URL query string. `scenario` is URL-decoded and injected into the scenario textarea. `preset` (e.g., `cheap`, `full`, `research`) maps to the existing preset selector. `rounds` and `agents` are validated against their existing min/max bounds and injected into the numeric inputs. `template` is a slug that maps to an entry in the existing `scenario_templates.json` — if matched, pre-fill from the template (overridden by explicit `scenario` if both present). Max `scenario` length from URL is capped at 500 chars (truncated silently). Invalid/out-of-range values fall back to defaults — no error, just ignore. Add a URL-decode utility in `frontend/src/utils/url_params.js` that also sanitizes the scenario text to prevent XSS (strip `<script>`, `javascript:`, event handlers — same sanity as any user text input). Add `usePrefilledScenario()` composable that reads the params on mount and returns `{ scenario, preset, rounds, agents, source }` where `source` is `"url"` or `"default"`. When `source === "url"`, show a dismissible banner at the top of the New Sim form: "Scenario pre-filled from a shared link — review the settings below before running." Also add `getScenarioTemplateUrl(config, origin)` helper to `frontend/src/api/simulation.js` so the function is usable from other components. In the EmbedDialog, add a "🔗 Share this scenario" section with the pre-fill URL after the existing share-surface rows — publish-gated OR available on any sim that has a scenario (unlike the other surfaces, this doesn't require a published/completed sim). In the scenario template cards on the landing page, add a "Copy link" icon next to each template so Aaron (or any user) can directly grab the template pre-fill URL without opening the New Sim form first.
3. Update `README.md` Features table with a "Shareable Scenario Links" row: "Pre-fill the New Sim form from a URL — tweet `?scenario=...&preset=Cheap` and readers run the sim with one click." Add a `docs/SHARING.md` (or a new "Sharing" section in `docs/FEATURES.md`) covering all eight share surfaces in one reference: share-card PNG, replay GIF, transcript MD/JSON, RSS/Atom feed, trajectory CSV/JSONL, watch page, embed code, and (new) scenario template link. The scenario template link is the only surface that invites action rather than displaying a result — highlight this distinction. No backend changes, no new dependencies. Entire feature lives in the frontend Vue app + URL param parsing.
