# The 34th Surface Is the First One That Travels

For three months MiroShark's surface family has answered a single question for an integrator: *can you fetch this?* Thirty-three entries on the catalog, every one of them a `GET` (with one new `POST` last week) handing back JSON, SVG, PNG, ZIP, Atom — the deployment serves it, you store it, you render it. PR #152 merged at 12:40:13 UTC today and added the next question the catalog had never answered: *can you prove what you stored?* `GET /api/simulation/<id>/signed-result.json` wraps the canonical `signal.json` payload in an HMAC-SHA256 envelope so an integrator who archived a result months ago can verify the stored bytes match what MiroShark actually emitted — without making a live call back.

## A Verifiability Property, Not a New Data Slice

Every other surface shipped in the last seven days adds a *new way to look at* the corpus: a per-project lens, a platform health probe, a shape-vs-size companion, a many-ids-at-once primitive. Signed-result.json doesn't add anything new about the data. The fields inside its `result` block — `direction`, `confidence_pct`, `risk_tier`, `bullish_pct`, `neutral_pct`, `bearish_pct`, `quality_health`, `signal_generated_at` — are the same fields `signal.json` has been returning for months. What changes is what an integrator can *do with the bytes after the response closes*. The signature commits MiroShark, irreversibly and offline-verifiably, to the exact result it emitted; a Capacitr settlement ledger or an AntFleet leaderboard row can hash-pin the signature and prove integrity months later without phoning home.

The signing primitive is deliberately not new. HMAC-SHA256 over canonical JSON — `sort_keys=True, separators=(",",":"), ensure_ascii=True` — under the same `WEBHOOK_SECRET` the outbound webhook service already uses for the `X-MiroShark-Signature` header. An integrator who already wrote a `hmac.compare_digest` verifier for the webhook handler reuses it byte-for-byte here. Zero new config, zero new key-distribution problem, no Ed25519 / public-key surface area to operate. A future migration to a different primitive bumps the `algorithm` field and `schema_version`; today, v1 publishes only `hmac-sha256`. The envelope's `signed_at` is wall-clock and never enters the signed material — two consecutive requests for the same finished sim emit the same `signature` but different `signed_at`, so a recipient can hash-pin a single signature for a sim's outcome without the envelope's timestamp drifting it. Missing-secret deployments return `signed=false` with a human-readable error and a 200, not a 500 — the unsigned `result` block still carries the same fields as `signal.json`; the absence of a signature is the missing feature, not an API failure.

## Why It Matters

The named external integrators citing MiroShark by endpoint — Capacitr's `spec.capacitr.xyz/#miroshark` pinning `/x402/run` for settlement, AntFleet's `miroshark-bench` leaderboard, RevaultDrops's airdrop flow, research pipelines anchoring runs via `cite.bib` and DKG — were all consuming `signal.json` over HTTPS and trusting in-transit authenticity. Once the bytes were in their archive, the only thing standing between a months-old stored result and a downstream consumer was the integrator's own attestation. Signed-result.json shifts that trust anchor from "we promise this is what MiroShark returned" to "here's the signature; verify it yourself in fifty lines of stdlib." For a settlement layer where dispute resolution might surface a year after the run, that distinction is the entire bull case. For a research pipeline citing the corpus, it's the lightweight commitment that doesn't require an on-chain anchor for every row but still makes a falsified row immediately detectable.

The streak — 42 consecutive PRs without a new runtime dependency, six new catalog surfaces in seven days, the Aeon-built open-PR queue hitting zero for the first time in seventeen days — is what makes the integrity guarantee credible. A platform that ships verifiability on top of dependencies it can audit in one `pip show` is a platform whose attestation is worth signing under.
