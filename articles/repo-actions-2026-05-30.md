## Ecosystem Context

The last week split across three tracks that rarely move together: the backend finished (25th analytical surface in PR #124, on-chain wallet identity in PR #126, Railway deployment hardened in PR #125), the discovery layer landed (surfaces catalog open in PR #130, oEmbed shipped in PR #107), and the UI ran through a polishing sprint that ported the marketing-site visual language across the full application in three successive PRs.

The analytical quadrant is closed: signal gives direction, peak-round gives inflection, volatility gives turbulence, sparklines give per-agent trajectory. The catalog/discovery layer is nearly closed: oEmbed, badge, surfaces.json. What's missing now is not more analysis but access patterns — ways to get data out, share it selectively, and consume it programmatically in formats the rest of the web already speaks.

Three structural gaps emerge from the current state:

**The sharing gap.** Publish is binary — either globally public or entirely private. There is no selective share: a researcher who ran a private sim can't send it to a co-author without publishing it. Private Share Link fills this. It's the first privacy-preserving access pattern the platform has.

**The input portability gap.** Every surface returns outputs — direction, chart, badge, volatility score, agent trajectories. There is no surface that returns the *inputs*: the exact scenario configuration (scenario text, agent count, rounds, locale, model) that produced those outputs. Without it, forking a simulation requires either remembering what you entered or re-entering it manually. `GET /<id>/clone.json` fills this.

**The automation integration gap.** Gallery.json is a snapshot. There is no event stream — no way to say "notify me when a new simulation on topic X is published." RSS is the standard answer: every automation platform (Zapier, n8n, Make, Feedly) has RSS as a native input trigger. A single `GET /api/feed.rss` endpoint opens MiroShark simulations to the entire existing RSS/automation ecosystem.

Two re-eligible ideas from outside the 7-day window close the remaining gaps:

**French Locale** — Issue #95 (May 22) is the only open issue on the repo. The locale-negotiation protocol was documented in PR #123 (`?lang=` → Accept-Language header → fallback en). The infrastructure is in place; French translations aren't. Re-eligible from May-22 #2.

**Private Share Link** — Every integrator eventually needs a private demo or stakeholder preview. The platform has no selective sharing; adding it requires a token issuance endpoint, a gated view, and an expiry mechanism. Re-eligible from May-22 #1.

### 1. Private Share Link

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** The platform's sharing model is currently binary — publish globally or stay private. There is no middle path: sharing a completed simulation with a specific person (a client, a co-author, a stakeholder) without making it publicly indexed. Private Share Link fills this: a `/share/<token>` URL that exposes a simulation's results page to anyone holding the token, without triggering publication. The token is expiry-gated and revocable. Enables private demos, pre-publication review, and API-level selective access. Re-eligible from May-22 #1.

**How:**
1. Add `backend/app/services/share_link_service.py` (~180 LoC, stdlib `secrets` + `time` + `json` + `os`). `generate_token(sim_id, expires_in_days=30) -> dict`: creates a 32-char URL-safe token via `secrets.token_urlsafe(24)`, stores `{sim_id, token, expires_at: unix_timestamp, created_at: ISO-8601, revoked: false}` in a `data/share_tokens/<sim_id>/<token>.json` file (append-safe; each token is its own file). `resolve_token(token, sim_root) -> str | None`: scans `data/share_tokens/` for matching token file, returns `sim_id` if token is non-revoked and non-expired, `None` otherwise. `revoke_token(sim_id, token)`: sets `revoked: true` in the token file. `list_tokens(sim_id) -> list[dict]`: returns all non-revoked tokens for a sim (for settings panel display). Add `POST /api/simulation/<id>/share-link` (admin-auth via `X-Admin-Token`; body `{expires_in_days: int (1–365, default 30)}`; returns `{token, share_url, expires_at}`). Add `DELETE /api/simulation/<id>/share-link/<token>` (admin-auth; revokes the token; 204). Add `GET /share/<token>` — resolves the token to a sim ID, renders the simulation's results page with the same HTML as the published share view but with a "Private Preview" banner and no public indexing (`<meta name="robots" content="noindex">`). Return 404 if token invalid/expired/revoked. Add 10 offline unit tests in `test_unit_share_link.py`: valid token resolves to sim_id, expired token returns None, revoked token returns None, token file is created on generate, list_tokens excludes revoked, `expires_at` arithmetic correct (now + days × 86400), token is 32 chars URL-safe, two tokens for same sim both resolve, unknown token returns None, revoke sets revoked=true.
2. In `frontend/src/components/SettingsPanel.vue` (or wherever the per-simulation management UI lives), add a "Private Share" section (visible only for completed sims). Displays existing non-revoked tokens for this sim as a list (`created_at`, `expires_at`, revoke button). "Generate Link" button → `POST /api/simulation/<id>/share-link` → copies the resulting URL to clipboard + shows it inline. Revoke button → `DELETE /api/simulation/<id>/share-link/<token>` → removes from list. Add `createShareLink(simId, expiresInDays)` + `revokeShareLink(simId, token)` + `listShareLinks(simId)` to `frontend/src/api/simulation.js`.
3. Add `POST /api/simulation/<id>/share-link` and `DELETE /api/simulation/<id>/share-link/<token>` to `docs/API.md` under Sharing with a `ShareLinkResult` schema. Add `GET /share/<token>` to `docs/API.md` under Public Access. Document that share links bypass publish-gating but not revocation/expiry. Add "Private Share Link" to `docs/FEATURES.md` under Sharing. Zero new deps.

### 3. Simulation RSS Feed

**Type:** Integration
**Effort:** Small (hours)
**Impact:** Gallery.json gives a static snapshot of public simulations; it doesn't push notifications when new ones are published. Every automation platform (Zapier, n8n, Make.com, IFTTT) and RSS reader (Feedly, Reeder, NetNewsWire) has RSS as a native trigger. `GET /api/feed.rss` serves a standard RSS 2.0 feed of the 20 most-recently-published public simulations. An integrator building a news alert, a researcher monitoring a topic, or an automation pipeline watching for new data can subscribe with a single URL — no polling code, no webhook setup, no auth. This is the protocol adapter that makes MiroShark visible to the existing RSS/automation ecosystem, the same pattern oEmbed (PR #107) used for the publishing-platform unfurl gap. Net-new.

**How:**
1. Add `backend/app/api/feed.py` (~120 LoC, stdlib `xml.etree.ElementTree` + `json` + `os` + `datetime`). `GET /api/feed.rss` — no auth, no params. Scans the same sim_root as gallery service: filter `is_public=True` + `status="completed"`, sort by `published_at` desc, take 20. For each sim, build an RSS `<item>`: `<title>{scenario_title} — {direction} ({confidence_pct:.0f}% confidence)</title>`, `<link>https://{HOST}/share/{sim_id}</link>` (use `request.host`), `<description>{quality_health} quality · {total_rounds} rounds · Published {published_at}</description>`, `<pubDate>{RFC-2822 formatted published_at}</pubDate>`, `<guid isPermaLink="true">https://{HOST}/share/{sim_id}</guid>`. Channel-level tags: `<title>MiroShark Simulations</title>`, `<link>https://{HOST}/explore</link>`, `<description>Latest public MiroShark consensus simulations</description>`, `<lastBuildDate>` = newest published_at. Content-Type: `application/rss+xml; charset=utf-8`. `Cache-Control: public, max-age=300` (5-min cache; simulations complete in < 10 min so fresh enough). ETag: `rss-{sim_count}-{newest_published_at[:10]}`. Register blueprint in `backend/app/__init__.py`. Add 8 offline unit tests in `test_unit_feed.py`: returns valid XML, `<channel>` element present, `<item>` count ≤ 20, each item has title/link/guid/pubDate, unpublished sims excluded, incomplete sims excluded, empty sim_root returns valid RSS with 0 items, `Content-Type` header is `application/rss+xml`.
2. In the `/explore` gallery page `<head>`, add the RSS autodiscovery link: `<link rel="alternate" type="application/rss+xml" title="MiroShark Simulations" href="/api/feed.rss">`. This causes browsers and RSS readers to auto-detect the feed. In the `/explore` gallery page body, add an unobtrusive "RSS" badge/link near the filter bar (same pattern as the `[JSON ↗]` link added in May-28 idea #1's spec). Add `getFeedUrl()` returning `/api/feed.rss` to `frontend/src/api/simulation.js`.
3. Add `GET /api/feed.rss` to `docs/API.md` under Gallery & Discovery. Note: returns RSS 2.0 XML, not JSON. Document the `<title>` and `<description>` format. Add "RSS Feed" to `docs/FEATURES.md` under Gallery & Discovery. Add the autodiscovery `<link>` tag to the HEAD documentation note in `docs/API.md`. Zero new deps.

### 4. Simulation Clone JSON

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** Every surface in MiroShark returns outputs — direction, chart, badge, trajectories, volatility score. None return the *inputs*: the exact scenario configuration (scenario text, agent count, debate rounds, locale, model, style prompt) that produced those outputs. Without this, forking a simulation — run the same scenario with 30 agents instead of 20, or in French instead of English — requires manually re-entering the original parameters or storing them externally. `GET /api/simulation/<id>/clone.json` returns the full input configuration as a ready-to-POST body for `/api/run`. Pairs naturally with the Scenario Clone Button (still unbuilt in the UI) and with AntFleet's benchmark workflow (which needs reproducible variant configurations). The 28th publish-gated surface. Net-new.

**How:**
1. Add `backend/app/services/clone_service.py` (~100 LoC, stdlib `json` + `os`). `build_clone_payload(sim_dir) -> dict | None`: reads `simulation_state.json` for the sim; extracts `{scenario_text: str, agent_count: int, debate_rounds: int, locale: str (default "en"), model: str, style_prompt: str | null, reddit_config: {subreddit: str | null, post_count: int | null} | null}` — these are the fields that, when submitted to `POST /api/run`, reproduce the scenario. Returns `None` if sim is unpublished or incomplete (publish-gate). The response envelope: `{success: true, data: {sim_id: str, clone_payload: dict, example_curl: str}}` where `example_curl` is a copy-pasteable `curl -X POST https://your-host/api/run -H 'Content-Type: application/json' -d '<clone_payload_json>'`. Add `GET /api/simulation/<id>/clone.json` (publish-gated; 404 `{"error": "simulation not found or not published"}`). `Cache-Control: public, max-age=3600`. Add `"clone_json"` to `SURFACE_KEYS` + `surface_stats` (28th surface). Add 8 offline unit tests in `test_unit_clone.py`: valid published sim → returns dict with all required keys, unpublished sim → None, `example_curl` contains endpoint and sim_id, `agent_count` is int, `locale` defaults to "en" when absent, `scenario_text` is non-empty string, `surface_stats` increment called, response JSON-serialisable.
2. Add `getCloneJson(simId)` + `getCloneJsonUrl(simId)` to `frontend/src/api/simulation.js`. In `EmbedDialog.vue`, add a "🔁 Clone Configuration" section (publish-gated; positioned at the end of the Embed sections). Layout: one-line summary ("Copy the scenario configuration to fork this simulation"), `<pre>` block showing the `clone_payload` as pretty-printed JSON (truncated to 3 lines + expand toggle if > 100 chars), "Copy clone JSON URL" button, "Copy POST body" button (copies the raw `clone_payload` JSON). This is the API complement to the Scenario Clone Button; the button (when eventually built) will read this endpoint.
3. Add `GET /api/simulation/<id>/clone.json` to `docs/API.md` under Data Export & Analysis with a `ClonePayload` schema documenting each field and its meaning. Include a variant-testing curl example: get the clone payload, change `agent_count` from 20 to 40, POST to `/api/run`. Add "Simulation Clone JSON" to `docs/FEATURES.md` under Data Export & Analysis. Add to `openapi.yaml`. Zero new deps (35th straight PR since Nemotron).

### 5. Simulation Comparison UI View

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The comparison API (`GET /api/simulation/compare?a=<id>&b=<id>`) was confirmed to exist in the pre-Aeon codebase. It returns structured deltas — belief distribution differences, consensus agreement, quality comparison, rounds delta. No frontend view exposes this to human users; currently it requires curl, JSON parsing, and mental arithmetic. `CompareView.vue` at `/compare?a=<id>&b=<id>` makes the comparison human-usable in a browser: two sim cards side by side, colored delta indicators (↑ green / ↓ red / — gray) on each belief metric, a consensus agreement chip (Agree / Disagree), and a quality winner badge. Closes the compare workflow: AntFleet runs benchmarks programmatically; researchers compare variants in-browser. Net-new frontend for an existing backend. Net-new.

**How:**
1. Add `frontend/src/views/CompareView.vue`. The view accepts route params `a` and `b` (sim IDs). On mount: call `compareSimulations(a, b)` (existing or new function in `simulation.js`). Loading state: two skeleton cards. Error state: "Simulation not found or not published" (mirrors the API's 404 message). Layout: two-column grid (responsive: stacks on mobile). Left column = Sim A summary card; right column = Sim B summary card. Each card: scenario title, direction chip (Bullish/Neutral/Bearish with color), confidence pct, quality health badge, total rounds, created_at. Below each belief pct field (bullish/neutral/bearish), show the delta vs the other sim: `+{n}%` in green or `-{n}%` in red or `—` if <0.5% difference. Center row between the two cards: "Consensus Agreement" chip — green "✓ Agree ({direction_a})" if `same_direction: true`, red "✗ Disagree ({direction_a} vs {direction_b})" if false. "Quality" chip: "A wins", "B wins", or "Tied" based on `quality_comparison`. "Rounds delta" row: `A ran {n} more rounds` or "Same round count". Add the route `/compare` to `frontend/src/router/index.js` with `CompareView` as the component. Add `compareSimulations(simIdA, simIdB)` to `frontend/src/api/simulation.js` if it doesn't exist (calls `GET /api/simulation/compare?a=<idA>&b=<idB>`). Add query-param validation: if `a` or `b` is missing, show "Provide two simulation IDs to compare" state with an input form.
2. In the simulation detail view (wherever Share/Embed actions appear), add a "Compare" action. On click: opens a minimal modal with a single input ("Enter a second simulation ID to compare") and a "Compare" button. On submit: navigate to `/compare?a=<current_id>&b=<input_id>`. This closes the discovery loop — users can enter the compare view directly from any simulation they're viewing.
3. Add `GET /compare` to the client-side route documentation (no backend changes needed — the compare API already exists). Document the route in `docs/API.md` under the compare endpoint: "Interactive UI: `/compare?a=<id>&b=<id>`." Add "Simulation Comparison View" to `docs/FEATURES.md` under Analytics. Zero new deps.
