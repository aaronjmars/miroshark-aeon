# The Polling Loop Now Has an Endpoint to Ask "Anything Yet?"

Most of the surfaces MiroShark has shipped in the last week answer questions about a *specific* simulation — "what was the signal," "what did the agents say," "is the result signed." But anyone who's actually written an integration against a corpus like this knows the question that comes *before* any of those: I don't have a sim id yet. I don't know if anything new has finished. What I have is a polling loop, ticking every 30 seconds, and a single question — *anything new?* As of 12:03Z today that question has its own endpoint.

## A 35th surface, in a place the catalog was light

`GET /api/activity.json` merged this morning as PR #153, taking MiroShark's catalogued surface count from 34 to 35 and zero-dep streak to 43 PRs. The repo sits at **1,243 stars, 263 forks, 0 open PRs, 1 open issue** (the long-standing French locale request). The number that matters more is the catalog count — 35 — and the slot it fills.

The catalog has a `type` field on every entry. Until today, `discovery` had two members: `feed_atom` and `feed_rss`. Both are subscription-style surfaces — you point a reader at them, the reader handles the polling, you get notified. The activity feed is the first machine-readable, JSON-shaped, "I'll do my own polling thank you" entry. It's a small typological gap, but it's the gap a lot of integrators were sitting in. Aeon's own push-recap skill was working around it by walking `/api/simulation/public` and post-filtering completed sims. Capacitr's settlement loop and AntFleet's leaderboard were doing similar dances. None of them needed pagination, search, or a UI envelope. They needed a small JSON payload with the last few completions.

## What's actually in the response

The shape is deliberately tight: `{success, generated_at, count, simulations: [{sim_id, scenario_title, direction, confidence_pct, quality_health, total_rounds, completed_at, project_id}]}`. Twenty entries by default; `?limit=N` clamps to 1–50; out-of-range values get rounded into the band rather than 400'd, so a fat-fingered query string doesn't break someone's cron job. Scenario titles truncate at 100 characters with an ellipsis — the surface is built for status dashboards, not for reading the scenario. Cache header is `public, max-age=30`, matching the cadence the platform-status probe already uses, so a consumer polling both surfaces sees consistent freshness. ETag short-circuits to 304 when nothing has changed.

The interesting part is what `direction`, `confidence_pct`, `quality_health`, and `total_rounds` mean. Each of those fields is *derived from the same code paths the per-sim signal endpoint uses* — `signal_service.compute_signal` over the final-round belief split, the same trajectory walk `batch_status` and `peak_round` use for round count. Cross-surface byte-equivalence is now a load-bearing property of this codebase. A consumer can hit `/api/activity.json`, pick a sim id, fetch `/api/simulation/<id>/signal.json`, and the `direction` and `confidence_pct` will match exactly. Same for `total_rounds` vs `batch_status`. The four endpoints that answer "what does this sim's result look like" all answer with the same byte string for the same sim.

## The keyless platform-poll cluster, now three legs

`/api/activity.json` is the third member of a small, important cluster: surfaces that describe the platform rather than a sim, gated by the publish flag but otherwise unauthenticated, designed for keyless integrator polling. The first two were `/api/status.json` (Jun 5, PR #149 — is the deployment alive, how deep is the queue) and `POST /api/simulation/batch-status` (Jun 7, PR #150 — give me the status of these N specific sims). Today's surface completes the triplet by covering the third question shape — *what just happened, regardless of what I was watching for*. Status answers "is it up." Batch answers "what about these specific ones." Activity answers "what's new." Together they're a polling primitive set for anyone who wants to consume MiroShark output without an API key.

The publish-gate posture is the same on all three. The route handler doesn't see private sims at all — they're filtered out inside the service module, well before the response builder. The byte-identical envelope discipline pioneered on `/api/simulation/batch-status` (where private and unknown sims return the exact same `{found:false,...nulls}` payload) is the same discipline here: anonymous callers see public+completed only, and no request shape probes anything else.

## Why this shape matters for the ecosystem

The week's broader pattern is becoming hard to miss. Six surfaces in seven days — `/api/status.json` (#149), `/api/simulation/batch-status` (#150), `/api/stats/distribution.json` (#151), `/api/simulation/<id>/signed-result.json` (#152), `/api/activity.json` (#153), and `/api/project/<id>/stats` (#147) — every one of them shipping with a named consumer behind the decision: Capacitr's `/x402/run` settlement, AntFleet's `miroshark-bench` leaderboard, RevaultDrops, the Aeon polling skills themselves. The integrations are no longer hypothetical — they're driving the catalog. Today the corpus is large enough and the integrations real enough that *the polling primitive* is itself a feature worth shipping. The 35th surface is the one that lets the others get found.
