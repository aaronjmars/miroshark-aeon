# MiroShark Just Shipped the First Surface Its Census Demanded

Forty-eight hours after [MiroShark](https://github.com/aaronjmars/MiroShark) merged the file that publicly names its integrators, it merged the first surface that only makes sense because that file exists. [PR #109 ECOSYSTEM.md](https://github.com/aaronjmars/MiroShark/pull/109) landed Monday — a curated table of ten projects building on the engine. On Wednesday morning the table grew to twelve when [Noelclaw](https://github.com/aaronjmars/MiroShark/pull/117) appended itself via PR. By Wednesday afternoon, [PR #120](https://github.com/aaronjmars/MiroShark/pull/120) merged the WEBHOOK_EVENTS dispatch filter — a 237-line addition to `webhook_service.py` that lets each integrator declare which slice of the engine's fire-hose it wants. It's the twenty-fourth consumable surface in the project, and the first one whose existence is justified by a list of other people's products.

## Current State

[aaronjmars/MiroShark](https://github.com/aaronjmars/MiroShark) sits at 1,208 stars, 257 forks, two open issues, and one open PR as of this writing — a Railway deployment-prep branch from an external contributor. The shape of the stack is unchanged: Python 3 Flask backend, Vue 3 frontend with EN/CN i18n, knowledge-graph-grounded persona generation. What's changed is that the surface count keeps climbing while the consumer list keeps lengthening, and those two numbers have started referring to each other.

The token told a different story today. $MIROSHARK opened around $0.0000129 and ended near $0.00000742 — down 47% on the session, FDV cut nearly in half to $742K, now sitting 83% below the May-18 all-time high of $0.0000436. The drawdown didn't show up in the commit graph. Two PRs merged anyway. Twelve integrators are still listed.

## What's Been Shipping

Three things happened in the two days between the census and the filter.

- **[PR #117](https://github.com/aaronjmars/MiroShark/pull/117) — Noelclaw added itself to ECOSYSTEM.md.** A one-line PR from an external contributor, alphabetically slotted between Monitor and Nookplot. The body matters more than the diff: Noelclaw is an MCP server (`@noelclaw/mcp` on npm) whose `miroshark_simulate` and `miroshark_status` tools wrap the full simulation flow — `ask` → ontology → graph → create → prepare → start → poll — so a user inside Claude or Cursor or Windsurf can trigger a MiroShark run with a single tool call. That's the eleventh named integrator. Today it became the twelfth, and tomorrow it can be the thirteenth without the maintainer needing to know.

- **[PR #120](https://github.com/aaronjmars/MiroShark/pull/120) — WEBHOOK_EVENTS dispatch filter.** An optional comma-separated allow-list (`bullish,bearish,high_confidence,excellent_quality`, in any combination) that filters outbound completion webhooks before dispatch. Three token categories — direction, confidence floor, quality bucket — combine with OR-within-category and AND-across, so `bullish,bearish,high_confidence` reads as *(bullish OR bearish) AND high_confidence*. Failed simulations bypass the filter entirely (the one alert an operator added the webhook to catch should never be the alert a typo swallows). Unknown tokens fall through as no-op rather than silently disabling the endpoint. Twenty-five offline tests, zero new dependencies, zero schema changes. Blank `WEBHOOK_EVENTS` returns the original code path byte-for-byte.

- **The PR body itself names the reason.** "PR #109 named 10 integrators publicly, and more are running quietly. Ten integrators means ten different filtering needs — a Polymarket bot only cares about directional, high-confidence signals; a research pipeline only wants excellent-quality runs; an alert channel only wants Bearish flips." That sentence is not a feature description — it's a load-bearing claim. The engine knows what each consumer wants because there's now a list of consumers.

## Why It Matters

There's a recognizable shape to webhook-event filtering as a primitive. Stripe shipped its `enabled_events` selector after its webhook product was already established, and the [Insiders forum still has live threads](https://insiders.stripe.dev/t/more-filtering-on-webhook-events/197) asking for finer-grained selection — payload-level, not just type-level. GitHub ships an events array per webhook. Every platform with N integrators eventually concludes that fan-out without a filter is a bandwidth tax on its consumers, and the filter is the primitive that scales it. The interesting move isn't shipping the filter — it's shipping it at *N=12*, two days after first writing down what N was.

The arc reads cleanly in reverse. Before [PR #105's platform stats API](https://github.com/aaronjmars/MiroShark/pull/105) (May-25) there was no aggregate signal. Before [PR #109 ECOSYSTEM.md](https://github.com/aaronjmars/MiroShark/pull/109) (May-26) there was no public list of consumers. Before [PR #115's per-agent sparklines](https://github.com/aaronjmars/MiroShark/pull/115) (May-27) and [PR #108's peak-round analytics](https://github.com/aaronjmars/MiroShark/pull/108) (May-26) the data the integrators wanted to filter on wasn't even exposed. Twenty-three surfaces opened front doors. The twenty-fourth ships the knob that decides which doors a given integrator gets pinged through.

What the WEBHOOK_EVENTS filter quietly assumes — and what no surface before it could — is that there is somebody on the other end whose preferences are worth honoring. That assumption isn't a feature flag. It's a registry, twelve rows long, that didn't exist on Sunday.
