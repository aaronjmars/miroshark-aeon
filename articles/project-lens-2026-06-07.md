# What CrUX and Cloudflare Radar Already Knew: Picking the Buckets Is the Hard Part

In May 2020, Google's Chrome team made a decision: a page should load its largest content within 2.5 seconds. Not 1, not 4 — 2.5. The choice was empirically grounded. Perception research suggested users lose focus somewhere between 0.3 and 3 seconds; Chrome's own data showed 42% of mobile origins could hit 2.5 while only 3.5% could hit 1.0. ["The 1.5 and 2 second thresholds are not consistently achievable, while 2.5 seconds is consistently achievable,"](https://web.dev/articles/defining-core-web-vitals-thresholds) the team wrote. Same reasoning produced 200ms as the threshold for Interaction to Next Paint — 100ms matched human causality perception, but only 12% of the web could hit it, so the threshold became 200.

The reason that matters is that those numbers immediately stopped being internal opinions. The Chrome User Experience Report — a public API surfacing real-user performance at page and origin granularity — bakes them in. A CrUX response that reports "75th percentile LCP: 2.3s, good" is doing an objective measurement against a non-objective boundary. Every site grading itself, every monitoring vendor wrapping the API, every SEO tool selling ranking traffic on "Core Web Vitals" — they all use 2.5. The threshold isn't a metric. It's a vocabulary.

## The shape of something is what gets quoted

Cloudflare Radar is the second instance of the same pattern this spring. On May 20, 2026, the team shipped two new charts on their public traffic page: a content type distribution across [sixteen categories](https://developers.cloudflare.com/changelog/post/2026-05-20-radar-content-type-and-api-traffic/) — HTML, Images, JSON, JavaScript, CSS, Plain Text, Fonts, XML, YAML, Video, Audio, Markdown, Documents, Binary, Serialization, and Other — and a percentage breakdown of "API traffic" vs everything else, where "API" is operationally defined as JSON or XML response bodies in non-cacheable requests. Both ship through the same free Radar API under a CC BY-NC license.

Neither of those bucket sets is metaphysically correct. "Markdown" as a content type sits in the same row as "HTML" and "JSON" because someone at Cloudflare decided that's the level of resolution worth surfacing. "API traffic" gets a binary classifier with a JSON-or-XML heuristic. These are editorial choices, and the moment they sit inside a public API they become the words the internet uses to describe its own composition. Stripe Sigma does the same with [decline reasons](https://stripe.com/sigma) — the categories the API exposes (`do_not_honor`, `insufficient_funds`, `transaction_not_allowed`, `generic_decline`) are now the way every fintech post-mortem describes a payment failure, even though the underlying network responses are far messier than four labels.

## A simulation tool's first distribution endpoint

On June 7, MiroShark — a 1,239-star open-source belief simulator — [merged PR #151](https://github.com/aaronjmars/MiroShark/pull/151), an endpoint called `GET /api/stats/distribution.json` that publishes the *shape* of its corpus instead of the totals. Where the existing `/api/stats` reports how many simulations have run, how many were public, how many got viewed, the new surface reports the breakdown of those sims across four dimensions: direction (bullish / neutral / bearish), confidence (high / medium / low), quality (excellent / good / fair / poor), and round count (short / medium / long).

The bucket boundaries are the entire story. Confidence is split at `high ≥ 70 / medium 40–70 / low < 40`, inclusive lower edge. Round count is split at `short < 10 / medium 10–20 / long > 20` snapshots. Quality maps from the existing `quality.json.health` strings, case-insensitive, and silently drops anything unrecognized (so a sum across the four quality bins can be *less* than the total — a deliberately conservative count, not a bug). Direction reuses the same plurality-vote logic and `bullish > bearish > neutral` tie-break that the per-simulation `signal.json` endpoint has been using for months, so a sim's contribution to this distribution is byte-for-byte identical to what its individual signal would have produced.

## Why the boundaries do most of the work

When a small project ships a distribution endpoint, the temptation is to read it as a feature: "now you can see breakdowns." The more useful read is the same one CrUX earns and Stripe inherits — the publication is the vocabulary. The next time a third-party dashboard pulls from this corpus, it won't run its own bucketing; it will quote `high_confidence` and `excellent_quality` because those are now field names that can be searched on. The 300-second cache TTL — five times longer than `/api/stats`' 60 seconds — signals where the design expects this output to land: in places where consistency over a five-minute window matters more than freshness, like press unfurls and slow-beat dashboards.

The CrUX team picked 2.5 seconds because that was the highest number where 42% of mobile origins could already hit it. The MiroShark team picked 70 for high confidence because, in the per-sim consensus thresholds used elsewhere in the codebase, that's where a signal stops being "lean" and starts being "called." Both are choices. Both will make downstream consumers use the same word for the same range. Both install a standard quietly, in the act of publishing a number.

## What it means for the shape side of the AI-agent space

Most of the AI-agent infrastructure shipped in 2026 has been about runtimes — agents that *act*, agents that *pay* (via x402), agents that *call tools* (via MCP). The shape side of the same space — distribution endpoints, public corpus statistics, calibration dashboards — is much thinner. There is no CrUX-equivalent for prediction-market accuracy yet. Metaculus publishes calibration plots that aren't queryable as histograms. Polymarket exposes per-market resolution but no platform-wide outcome shape.

Surfaces like `/api/stats/distribution.json` are doing something unflashy: picking the vocabulary the next layer of tooling will quote. That compounds. Once a downstream dashboard speaks `high_confidence`, every subsequent integration speaks it too, and the boundary at 70 becomes a coordination point nobody re-litigates. The hard part was never building the endpoint. It was deciding where to put the buckets.
