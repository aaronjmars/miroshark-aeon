# MiroShark Built Its First Mirror

For ten weeks the project shipped surfaces that all pointed the same direction: outward, from one simulation. Chart SVG, share-card PNG, signal.json, polymarket.json, badge.svg, cite.bib, archive.zip, frame-metadata, dkg-citation, waybackclaw-record — fourteen distinct routes, all rooted at `/api/simulation/<id>/...`, all answering some flavour of *"what does this one run look like to the outside world?"* On Sunday 2026-05-24 at 11:33 UTC, the fifteenth surface opened as PR #105. It is the first one that doesn't take a simulation id at all. `GET /api/stats` and `GET /api/stats/badge.svg` describe the platform itself.

## Current State

[aaronjmars/MiroShark](https://github.com/aaronjmars/MiroShark) sits at 1,194 stars, 247 forks, 4 open issues, 2 open PRs as of this writing. The repo crossed 1,000 stars on 2026-05-03 — the hyperstition target of 1,000 by 2026-04-30 missed by three days, the next threshold (1,500) is projected for late July at current pace. The framework runs Python 3 backend (Flask blueprints, pure-stdlib services where it can), a Vue 3 frontend with i18n, and an LLM-driven WonderwallProfileGenerator that now takes graph context, web context, and — as of yesterday's PR #103 — Nemotron demographic anchors.

The $MIROSHARK token tells a separate story: -68.8% from the 2026-05-18 ATH of $0.0000436 as of yesterday's close, then a sharp +25.3% bounce today to $0.0000175 (FDV $1.75M; volume $342K; buy/sell ratio 2.63×, the strongest buy-side dominance in several sessions). The framework has decoupled from the chart in both directions now: it kept shipping during the five-day drawdown and it kept shipping when the price rebounded. The merge cadence does not consult the candles.

## What's Been Shipping

A merge wave landed in 95 minutes on 2026-05-23, four hours after the daily push-recap cutoff:

- **PR #99** (Polymarket-ready prediction JSON) — the fifteenth publish-gated surface and the first *integrator-shaped* one. `GET /<id>/polymarket.json` reshapes `signal.json` into a YES/NO market envelope with direction-aware `yes_probability` and a four-bucket `confidence_tier`. Stricter publish gate than its sibling: only `status == "completed"` emits.
- **PR #98** (project-id path-traversal fix, external — AntFleet) and **PR #100** (launcher Aura support, external — voidfreud) — merged five seconds apart at 17:37 UTC. Third external contributor in ten days after teifurin (PR #89). AntFleet runs MiroShark as a *public security-benchmark target* at [AntFleet/miroshark-bench](https://github.com/AntFleet/miroshark-bench/pull/1), which is the first integrator-product feedback loop the project has had.
- **PR #103** (Nemotron demographic grounding) — the streak-breaker. From PR #72 through PR #102, thirty-one consecutive PRs added zero new dependencies. PR #103 explicitly adds `duckdb >= 1.0.0` and `huggingface_hub >= 0.23.0` to both `pyproject.toml` and `requirements.txt`. The dependencies are framed as "optional, degrades to graph-only" in the code, but they are unconditional in the package manifest, so `pip install` always pulls them. The cost is paid where the value cannot be derived from what's already in the tree — a real columnar query over Nemotron parquets is not something `xml.etree` can fake. A new streak starts at PR #104.

Then today's PR #105 — coupled May-22 batch ideas #4 and #5, built as one PR because the badge route reuses the platform-stats scan. The endpoint walks `WONDERWALL_SIMULATION_DATA_DIR` once, filters to `is_public=true AND status="completed"`, derives stance via the same `compute_signal` helper the per-sim `signal.json` uses (so platform-level consensus distribution matches the sum of per-sim signals byte-for-byte), sums every `surface-stats.json` counter, and deduplicates by `project_id`. A 60-second module cache makes a polling consumer pay roughly the cost of one HEAD request per minute. ETag derives from `total_sims + newest_sim_id`. The badge is a [Shields.io](https://shields.io/)-compatible flat pill — `MiroShark | N simulations` in platform-blue `#0ea5e9`, distinct from the three stance colours — and it never 404s. Zero-sim deployments render `MiroShark | 0 simulations`.

## Why Platform-Level Is Different

The fourteen per-sim surfaces are pull points for *a specific run*: cite this paper, embed this chart, archive this provenance, render this badge. A consumer needs a sim id and a reason. Platform-level surfaces are pull points for *the framework itself*: how many simulations have run, what the consensus distribution looks like across the whole catalogue, whether the platform is alive at all. The first one ships press-kit copy ("MiroShark has run N simulations"), live README badges for every operator's portfolio, and LLM-agent health checks ("is this endpoint warm?") that didn't exist before. The badge is the second-order amplifier — per-sim badges (PR #94) make any one simulation a pull point for itself; the platform badge makes any blog post mentioning MiroShark a live signal that the framework is still running.

## Why It Matters

Two architectural shifts are now visible in the same week. PR #103 ends the zero-deps streak by paying a real infrastructure cost where the feature actually requires it — first opt-in remote dataset (HuggingFace Hub), first `SimulationState` field that's neither config knob nor output channel. PR #105 opens a new surface class — platform-level rather than simulation-level — that the project never had a slot for before. Both changes admit that the framework has grown past the size where everything can be described in terms of one sim and pure stdlib. Outside builders agree: in the same ten days, three independent contributors landed five PRs, and one of them is using MiroShark as the benchmark target for their own product. The architecture is now meeting consumers it didn't design for, on its own surfaces, under a token chart that does not care.
