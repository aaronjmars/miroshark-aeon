## Ecosystem Context

MiroShark now carries the deepest feature stack in open-source LLM social simulation: graph-built personas, multi-platform cross-pollination engine (Twitter / Reddit / Polymarket), full observability, fork/branch, comparison mode, history search, prediction resolution, belief drift visualization, one-click article generation, trace-grounded agent interview, browser push notifications, Director Mode (mid-run event injection), simulation replay/playback, quality diagnostics, agent interaction network graph, embeddable widget, and agent demographic breakdown. Ten major features shipped in eleven days. At 720 stars — up from 563 on April 7 — growing at roughly 12–15/day with a strong academic/developer signal.

The project's analysis layer is now genuinely research-grade. What gaps remain are subtler but consequential: **setup friction**, **longitudinal research**, **counterfactual depth**, **artifact portability**, and **developer ecosystem health**.

Key signals driving this batch:

- **New users face a blank-page problem at setup:** A user who lands on MiroShark, uploads a document or URL, and arrives at the scenario field faces a cold-start question: "What should I ask?" The simulation engine is powerful — but its output is only as good as the scenario framing. First-time users often write generic scenarios ("What will happen?") that produce weaker simulations than a well-framed bull/bear/neutral split. The document is already in hand; a single LLM call against it can generate three concrete scenario suggestions (with expected market direction) in under 2 seconds. This turns scenario authoring from a blank text field into a "pick one and refine" flow — dramatically reducing the setup burden for users who aren't prediction market researchers by background. Nothing in the current codebase does this; it's the last major friction point before first-value.

- **Longitudinal simulation is impossible:** Every simulation run is one-shot. Users can fork, compare, and replay — but there's no way to say "run this same 30-agent setup on today's headlines about EU AI regulation every Monday and show me how consensus evolves over time." Researchers studying opinion dynamics over a multi-week news cycle have no path. The simulation engine, queue system, and history view already exist; a "recurring run" scheduling UI (CRON-style, config stored in a `watch_config` record) would wire them together. History view already shows a series of completed simulations — a watch-mode filter would render them as a temporal series. This is the primitive that enables longitudinal social simulation as a research methodology.

- **Trajectory data supports counterfactuals that don't require rerunning:** The `trajectory.json` captures every agent's belief state, every post, every interaction — per round. The influence leaderboard and quality diagnostics already derive insights from this data client-side. But the most shareable research insight is: "What would have happened if Agent X — the top influencer — hadn't been in this simulation?" Every user who sees a dominant hub agent on the network graph will immediately wonder this. A counterfactual toggle that excludes selected agents from the belief drift computation (refiltering the existing trajectory, no rerun needed) answers this in milliseconds. A 30-second interaction that produces a publishable figure.

- **Simulation outputs are browser-only:** The belief drift chart, network graph, quality diagnostics, and demographic breakdown all exist as interactive Vue components inside a browser tab. Researchers writing a paper, a report, or a grant application need a self-contained, shareable artifact — one file they can attach to an email or paste into a Notion page. A downloadable PDF report that captures: scenario/title, quality health, belief drift chart (as SVG-to-PNG), top 5 agents, echo chamber score, top demographic divergence, and any Director Mode injection timeline. All data is already rendered in the browser; a server-side PDF renderer (or `html2canvas` + `jsPDF` client-side) turns "print screen" into "export report." Every researcher who currently screenshots the UI would use this.

- **Contributor onboarding requires local setup:** Contributing to MiroShark requires Python (backend), Node.js 18+ (frontend), Neo4j (graph), and several env vars. The README has setup instructions, but local setup is still a 10-15 minute process with multiple failure points (Python version mismatch, Neo4j not running, missing `.env`). A `.devcontainer/devcontainer.json` config makes the entire environment reproducible in GitHub Codespaces or VS Code Dev Containers in a single click — no local install required. For academic users and short-term contributors who want to patch one thing without committing to a full local setup, this is the difference between "I'll contribute later" and "I'm contributing now."

### 1. Scenario Auto-Suggest from Document

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** MiroShark's setup flow has one blank field that blocks new users: the scenario text ("What do you want the agents to debate?"). A researcher who uploads a Substack essay about the EU AI Act knows roughly what they want — but articulating a crisp prediction-market-style scenario is a non-trivial skill. Scenario Auto-Suggest triggers the moment a document or URL is pasted into the input: a backend call sends the first 2,000 characters of the document to the configured LLM and returns three pre-filled scenario cards within 1–2 seconds. Each card shows a scenario question ("Will the EU AI Act pass with its stricter biometrics provisions intact by June 2026?"), a framing label (Bull / Bear / Neutral), and an expected market direction with confidence band ("Expected initial YES: 60–70%"). The user clicks any card to auto-fill the scenario field — or ignores all three and types their own. This eliminates the blank-page problem for every new user and makes MiroShark immediately more productive for users who understand the domain but aren't fluent in prediction market framing. It also trains users over time in what good simulation scenarios look like.
**How:**
1. After the document text or URL resolves in the Step 1 setup component, debounce 800ms and fire `POST /api/suggest-scenarios` with `{ document_preview: text.slice(0, 2000) }`. The backend calls the active LLM provider with a compact prompt: "Given this document excerpt, generate 3 prediction-market-style simulation scenarios. For each: a YES/NO question framed around a concrete outcome, a label (Bull/Bear/Neutral), and an expected initial YES probability range. Return as JSON array with keys: question, label, expected_yes_range." Cache the response in `<sim_dir>/.auto_suggest.json` keyed by a SHA-256 of the document preview so rerenders don't re-call.
2. Render the 3 cards below the document input field as horizontal chips (or a compact card row on mobile). Each card shows the scenario question, label badge (colored: Bull = teal, Bear = coral, Neutral = slate), and the probability range in muted text. A "Use this" button fills the scenario field and dismisses the cards. An "X" button dismisses all suggestions without filling. Cards fade in via CSS transition so they don't feel intrusive. If the LLM call fails or times out (>3s), cards silently don't appear — the UX is unchanged for the fallback path.
3. Add telemetry: log `{ suggestion_used: boolean, suggestion_rank: 0|1|2|null, document_length: N }` to the simulation record on submit. After 50 simulations, compute the pick rate per suggestion position — this data informs future prompt tuning. Include the selected (or user-written) scenario in the article generator context unchanged. Add a "Suggestions powered by {LLM provider}" tooltip on the card row so users know which model generated them. Surface the auto-suggest feature in the README under a "Smart Setup" heading with a GIF showing the cards appearing.

### 2. Recurring Simulation Watch (Scheduled Auto-Run)

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** Every simulation run in MiroShark is one-shot: configure, run, read results. But the most research-valuable questions are longitudinal — "how does the same 30-agent pool's consensus on EU AI regulation shift as the news cycle evolves over three weeks?" or "does the same document produce different outcomes when run Monday vs. Friday?" A Watch mode lets users configure a recurring simulation setup: agent count, LLM provider, round count, and either a fixed document or a URL that's re-fetched at each run time. The watch schedule is set via a simple picker: daily (at a configured hour), weekly (day + hour), or on-demand manual trigger. When the scheduled time arrives, MiroShark auto-creates a new simulation from the stored config and runs it exactly as if the user had clicked "Run." All runs in a watch series appear grouped in the history view as a "Watch Series" section — a mini timeline showing how consensus, market price, and quality score evolved across runs. This is the primitive that makes MiroShark useful for tracking opinion dynamics over real-world news cycles, not just analyzing one document at one moment.
**How:**
1. Add a `watches` table (SQLite) with columns: `id`, `name`, `config_json` (full simulation config: agent count, rounds, LLM provider, document_text OR document_url), `schedule` (cron string: `"0 9 * * 1"` = Mondays at 9am), `next_run_at` (UTC ISO8601), `created_at`, `last_run_at`, `run_count`, `active` (boolean). Add a `watch_id` foreign key to the `simulations` table so each auto-run links back to its parent watch. Add `POST /api/watches`, `GET /api/watches`, `DELETE /api/watches/:id`, `POST /api/watches/:id/trigger` (manual run) endpoints.
2. Add a background scheduler in `app/scheduler.py` using APScheduler (already a common FastAPI dependency). On startup, load all active watches, register their cron schedules, and trigger `_run_watch(watch_id)` at the scheduled time. `_run_watch` re-fetches the URL (if `document_url` is set), creates a new simulation entry with `watch_id` set, and enqueues it through the normal simulation runner. On completion, update `last_run_at` and `run_count`. Handle errors gracefully: if a URL fetch fails, log the error on the watch record and skip the run without deactivating the watch.
3. Add a "Watch Series" section to the history view that groups simulations by `watch_id`. Each watch shows: watch name, schedule description ("Every Monday at 9am"), run count, and a mini sparkline of final YES prices across all runs (so the temporal trend is visible at a glance). Expanding a watch series shows all individual runs in chronological order. Add a "Create Watch" button to the simulation setup flow: after configuration is complete (but before running), a "Save as Watch" option opens a lightweight modal with a name field and schedule picker — the user can save the config for recurrence without running it immediately. Add a "Watches" tab to the history view navigation.

### 3. Agent Counterfactual Explorer ("What If?")

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The Agent Interaction Network Graph shows who the hub agents are — the nodes that every other agent responds to. The influence leaderboard ranks them. But the question every researcher asks after seeing a dominant hub is: "What would have happened if Agent X wasn't in this simulation?" The Counterfactual Explorer answers this without rerunning anything. Since `trajectory.json` records every agent action and stance per round, belief drift and consensus can be recomputed from a filtered subset of agents — excluding any combination the user selects. The user opens the Counterfactual panel, checks off 1–3 agents to "remove," and the belief drift chart instantly re-renders using only the remaining agents' trajectories. The final market price and consensus round recompute in the same pass. A side-by-side view shows "With Agent X (original)" vs. "Without Agent X (counterfactual)" — two belief drift lines on the same axes. For researchers, this is a publishable figure: "Removing the top hub agent shifted consensus from 74% YES to 51% YES — a 23-point swing." It's the structural insight the network graph suggests but can't quantify on its own.
**How:**
1. Add a `GET /api/simulation/:id/counterfactual?exclude_agents=id1,id2` endpoint. Reads `trajectory.json` and recomputes: `participation_rate` (excluding removed agents), per-round stance distribution (same entropy calculation as quality diagnostics, but over the filtered agent set), consensus round (first round where one stance bucket > 60% of filtered agents), and final YES price approximation (mean bullish fraction × 100 over the filtered set in the final round). Return as `{ filtered_drift: [...], filtered_consensus_round: N, filtered_final_yes: P, original_drift: [...], original_consensus_round: M, original_final_yes: Q, delta_final_yes: P-Q }`. Pure data-transform — no LLM call, no re-simulation.
2. Add a "What If?" toggle to the simulation results view (alongside Quality, Demographics, Network). Opening it shows the influence leaderboard with checkboxes beside each agent row. Selecting 1–3 agents enables a "Recompute" button. On click, call the counterfactual endpoint and render a split belief drift chart: the original curve (muted, dashed) and the counterfactual curve (solid, highlighted) on the same axes. Below the chart, show a "Counterfactual Impact" summary: final YES with vs. without the selected agents, consensus round shift, participation rate change. A badge shows "Strong influence" (>15% YES delta), "Moderate" (5–15%), or "Minimal" (<5%).
3. Add a "Share counterfactual" button that generates a URL with `?counterfactual=id1,id2` appended to the share permalink. On the share page, if the counterfactual parameter is present, render the counterfactual split chart as the default view (with a toggle to switch back to the original). Include the counterfactual finding in the article generator context when the "What If?" panel is open: inject as "Key counterfactual: Removing {agent_names} shifted consensus from {original_yes}% to {cf_yes}% YES." Export the counterfactual chart as PNG using the same mechanism as the belief drift and network graph export buttons.

### 4. Downloadable PDF Simulation Report

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** All of MiroShark's analytical outputs live inside a browser tab. Researchers writing papers, filing grant applications, or briefing stakeholders need a self-contained shareable artifact — one file that captures the full simulation result without requiring the recipient to navigate a running MiroShark instance. A "Download Report" button on completed simulation cards generates a formatted PDF containing: scenario title and document URL, simulation metadata (agent count, rounds, LLM model, run time), quality health badge with the four diagnostic metrics, the belief drift chart as an embedded PNG, top 5 agents with influence scores and final stances, echo chamber score from the interaction network, top demographic divergence finding, and any Director Mode event injections with their round stamps. The PDF is generated server-side (no browser print dialog, no layout unpredictability) and available for download within 2–3 seconds. For researchers, this replaces "screenshot + paste into Word" with a branded, reproducible report that includes all the quantitative findings MiroShark produces. Every downloaded report carries the MiroShark project URL as a footer — passive distribution to anyone who receives the report.
**How:**
1. Add a `GET /api/simulation/:id/report.pdf` endpoint. Use `weasyprint` (Python, pure CSS-to-PDF) or `reportlab` to generate the PDF server-side. Assemble the data: pull from the existing `/quality`, `/interaction-network`, `/demographics`, and `/embed-summary` endpoints (all already built and cached) to avoid re-computation. For charts, call the belief drift endpoint and render a PNG via `matplotlib` (already a likely dependency) or inline SVG converted to PNG using `cairosvg`. The PDF template is a single HTML file rendered by weasyprint: MiroShark header, scenario block, metadata table, chart image, findings sections. Return as `Content-Type: application/pdf` with `Content-Disposition: attachment; filename=miroshark-{sim_id[:8]}-report.pdf`.
2. Add a "Download Report" button to the simulation history modal and the simulation results header (alongside the existing Share and Fork buttons). Show a brief loading state ("Generating PDF…") while the server-side render completes. On success, trigger the browser download. Add a size note on hover: "~150KB PDF with all charts and findings." For simulations where quality.json or network.json haven't been computed yet, compute them inline during report generation (with a slightly longer generation time — show a progress bar with "Computing quality metrics…" / "Building network…" / "Rendering charts…" steps).
3. Add a report footer with: "Generated by MiroShark · https://github.com/aaronjmars/MiroShark · {timestamp}" and a QR code linking to the share permalink (so the PDF recipient can click through to the interactive version). Add a `reports_generated` counter to the instance admin view (if one exists) or a simple log entry. Update the README with a "Export" section showing a screenshot of the PDF report layout — a strong visual signal for researchers evaluating the tool.

### 5. Dev Container / GitHub Codespaces Setup

**Type:** DX / Community
**Effort:** Small (hours)
**Impact:** Contributing to MiroShark requires Python 3.11+, Node.js 18+, a running Neo4j instance, and several env vars configured in `.env` files across both `backend/` and `frontend/` directories. The README has setup instructions, but the process still takes 10–15 minutes with multiple potential failure points: wrong Python version, missing system dependencies (libssl, libpq), Neo4j not running, or conflicting global Node versions. A `.devcontainer/devcontainer.json` config makes the entire dev environment reproducible in GitHub Codespaces with a single click — the contributor clicks "Code → Open in Codespaces" and gets a fully configured environment (Python 3.11, Node 18, Neo4j via Docker Compose service, all deps installed, `.env.example` auto-copied) in under 3 minutes, no local setup required. For the academic/research segment — which is disproportionately on university machines they can't fully configure — this is the difference between "I'd like to contribute" and "I contributed this morning." At 138 forks, this benefits everyone who currently has a half-working local clone. It also makes demo and workshop scenarios feasible: a live audience can have MiroShark running in Codespaces in 3 minutes, which no local setup path can match.
**How:**
1. Create `.devcontainer/devcontainer.json`. Use the `python:3.11` base devcontainer image as the primary container. Add a `docker-compose.yml` service for Neo4j (`neo4j:5.13-community` with `NEO4J_AUTH=neo4j/miroshark-dev`). The `devcontainer.json` `postCreateCommand` runs: `pip install -r backend/requirements.txt && cd frontend && npm install && cp backend/.env.example backend/.env && echo "NEO4J_URI=bolt://neo4j:7687\nNEO4J_USER=neo4j\nNEO4J_PASSWORD=miroshark-dev" >> backend/.env`. Add VS Code extension recommendations: `ms-python.python`, `ms-python.vscode-pylance`, `Vue.volar`, `bradlc.vscode-tailwindcss`. Set `forwardPorts: [8000, 5173, 7474, 7687]` with display names ("MiroShark Backend", "MiroShark Frontend", "Neo4j Browser", "Neo4j Bolt").
2. Add a `devcontainer.json` `"tasks"` section with two pre-wired run tasks: "Start Backend" (`cd backend && uvicorn app.main:app --reload --host 0.0.0.0`) and "Start Frontend" (`cd frontend && npm run dev -- --host`). These appear in VS Code's task runner so contributors can launch both services from the Command Palette without memorizing the commands. Add a `.devcontainer/README.md` (shown by Codespaces on first launch) with the one-paragraph orientation: "This devcontainer includes Python 3.11, Node 18, and a pre-configured Neo4j instance. Run the 'Start Backend' and 'Start Frontend' tasks to get MiroShark running. Use the Forwarded Ports panel to open the UI."
3. Add an "Open in Codespaces" badge to the top of the README (GitHub's official badge: `[![Open in GitHub Codespaces](https://github.com/codespaces/badge.svg)](https://codespaces.new/aaronjmars/MiroShark)`). Update the "Installation" section of the README to lead with "Option A: GitHub Codespaces (zero local setup)" above the existing local setup instructions. This makes Codespaces the recommended default path for first-time contributors and evaluators — a signal that aligns with MiroShark's cloud-deploy-first positioning.

## Selection Rationale

This batch targets MiroShark's remaining friction points after the analytics layer matured this week:

- **Scenario Auto-Suggest** (#1) — Removes the blank-page barrier at the most critical moment: first value. The scenario field is the last setup friction point; LLM-generated suggestions reduce it to a one-click choice. Every new user benefits immediately.
- **Recurring Watch** (#2) — The longitudinal primitive. One-shot simulations are demos; recurring runs over a news cycle are research. This is the feature that converts MiroShark from "I ran it once and it was cool" to "I've been tracking this topic for three weeks."
- **Counterfactual Explorer** (#3) — Zero new data collection. Client-side recomputation of belief drift with selected agents excluded. Turns the top hub finding on the network graph into a quantifiable research claim: "Removing Agent X shifted consensus by 23 points." The most demo-able insight in the entire analytics suite.
- **PDF Report** (#4) — Closes the "browser-only" gap. All analytical outputs (quality, network, demographics, belief drift) captured in one portable file. Passive distribution: every shared PDF links back to the project.
- **Dev Container / Codespaces** (#5) — Infrastructure investment for the contributor funnel. At 138 forks and growing academic interest, reducing the local setup barrier from 15 minutes to 3 minutes compounds. The "Open in Codespaces" badge in the README signals maturity.

Each idea is scoped for autonomous implementation by the `feature` skill — clear inputs/outputs, no ambiguous design decisions, no external approvals needed.
