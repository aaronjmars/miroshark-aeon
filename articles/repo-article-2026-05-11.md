# The First Surface MiroShark Doesn't Own

Every surface MiroShark has shipped so far runs on Aaron's instance. The gallery, the share links, the embed, the lineage tree, the reproduce.json — they all live behind one origin, and the trust model is implicit: you trust the URL because you trust the domain. PR #79, opened this morning, is the first surface whose verification runs somewhere else. A recipient on the other side of the internet can now hold a few bytes in their hand and decide, with no help from MiroShark, whether a webhook payload actually came from a MiroShark instance. That's a different kind of primitive than anything that came before.

PR #79 sits outside that arc. It doesn't touch `sim_dir/`. It doesn't add a surface to the gallery. It modifies one file in the dispatch path — `backend/app/services/webhook_service.py` — and ships three small functions: `compute_signature`, `verify_signature`, `_resolve_webhook_secret`. The diff is +692 / -6 across ten files, with the largest single addition being 312 lines of offline tests. Zero new dependencies. That's the eighteenth consecutive substantive PR (running from #57) that has not added a single line to `requirements.txt`. The streak has become a discipline.

## Technical Depth

The scheme is the boring one — boring being the highest praise for a security primitive. When `WEBHOOK_SECRET` is set in the environment, every dispatched payload gets an `X-MiroShark-Signature: sha256=<hex>` header, where the digest is HMAC-SHA-256 over the raw body bytes. It's the same scheme Stripe and GitHub use for their outbound webhooks. The recipient verifies in three lines: read the header, recompute the digest with `hmac.new`, compare with `hmac.compare_digest` for constant-time equality. If the secret is unset or blank, the header is omitted entirely and every existing Slack/Discord/Zapier/Make/n8n integration keeps working untouched. That's the backward-compatibility surface, and it's load-bearing — Aaron has a non-trivial integration tail already, and any change that breaks the wire format is a change that breaks revenue-adjacent partners.

The structural twin is `reproduce.json`. PR #75 made a simulation's state cryptographically pinned: serialize the spec with `sort_keys=True, indent=2`, take SHA-256, and you have a citation key that survives transport, mirror sites, and screenshot misquotation. PR #79 does the same trick for the *delivery*: sign the raw body bytes, and you have a digest that survives a hostile recipient, a man-in-the-middle, or — most realistically — a curious external party who just wants to know whether the payload they're staring at is a real MiroShark event or someone's `curl` impression of one. Reproducibility was the citation primitive for the *content*. HMAC signing is the citation primitive for the *transport*.

## Why It Matters

The integration tail matters more than the star count right now. @TheGodfath13541's thread this week named Revault and CancerHawk as live integrations and flagged that x402/API has been running ATH usage on OpenRouter. Lorimer Ventures reportedly followed Aaron. Every new external system that pulls a webhook from MiroShark is one more piece of someone else's stack quietly trusting the domain in the `From` field. Without signing, that trust was based on TLS termination and the goodwill of intermediate hops. With signing, it's based on a shared secret that the operator and the recipient hold in independent vaults. The verification check runs on the recipient's hardware. MiroShark doesn't get a vote.

That inversion — the first surface where MiroShark hands over the verification step instead of serving it — is the structural moment. The eleven-surface arc made MiroShark legible. The signature header makes MiroShark *cryptographically detachable* from its own outputs. For an autonomous-simulation engine that wants to be cited, syndicated, and bought against, that's the next floor.
