## Ecosystem Context

The surface-multiplication arc is complete. MiroShark now has nine distinct surfaces over a single `sim_dir/` folder: gallery card, share card PNG, replay GIF, transcript MD+JSON, RSS/Atom feed, trajectory CSV/JSONL, watch page, gallery search, and shareable scenario pre-fill URL. Each additional surface took less work than the last because the substrate never changed. The question for this batch isn't "what's the next surface?" — it's "what's missing from the surfaces that already exist?"

Three signal threads drive this batch:

- **The embed gap.** The share card PNG unfurls in X/Discord. The replay GIF auto-plays in motion-capable surfaces. The transcript Markdown lands in Notion. But none of these can be *embedded live* in a Substack post, a Ghost article, or a Discord pinned message that stays current as a sim runs. An `<iframe src="/embed/<id>">` endpoint — a self-contained HTML widget with real-time belief bars — closes the gap between "sharing a link" and "embedding a live result." The watch page already did the engineering hard part (vanilla-JS 30s poller, belief bar rendering from the embed-summary endpoint). The iframe variant is a constrained version of the same page: minimal chrome, no navigation, designed to be dropped into any embedding surface.

- **The operational blindspot after PR #46.** The outbound webhook fires on simulation completion. But there is currently no way for an operator to verify it fired, debug a 5xx from their endpoint, or see how long the delivery took. Every Zapier/Make/n8n integration that relies on the webhook is flying blind on delivery status. A `webhook-log.jsonl` append in `sim_dir/` on each dispatch attempt — timestamp, masked URL, HTTP status, latency — costs nothing architecturally and closes the "did it fire?" question that every production integration eventually asks.

- **The open issue that's been sitting since this morning.** Issue #70, filed May 4, requests a "Private Impact mode" — the ability to share a simulation privately before publishing it publicly. Right now the binary is: fully private (no sharing) or published (in the public gallery). There's no "share with a client before going public" mode. A per-sim secret token, appended as `?t=<uuid>` to any share URL, unlocks all nine surfaces for the token holder without indexing the sim in the public gallery. Directly resolves the issue. Small scope: one POST endpoint, UUID storage in `sim_dir/`, a `?t=` check in the existing `is_public` gate.

Two additional ideas round out the batch:

- **The last distribution format that's missing.** The share surfaces cover visual (PNG), motion (GIF), prose (transcript), data (CSV/JSONL), and live (watch). But Aaron's primary distribution channel — X/Twitter — is a *text* channel. Copy-pasting the share card link gets a preview; what Aaron actually needs is a pre-formatted tweet thread he can paste directly: intro tweet, one tweet per major belief flip, closing tweet with the final verdict and link. The data is already in `trajectory.json` (round-by-round stance splits) and the transcript (agent-authored posts). A `GET /api/simulation/:id/thread.txt` endpoint assembles it.

- **The browsability gap in the gallery.** Gallery search (PR #69) covers keyword + consensus + quality + sort. But operators running MiroShark for clients want to curate *named collections*: "all our DeFi risk sims," "Q2 2026 macro calls," "verified-correct ETH calls." Tags are the primitive: operator-applied, stored in `sim_dir/`, filterable in the gallery, visible on cards. It's the "editorial layer" PR #69 didn't provide — search finds what you're looking for; tags surface what an operator has deliberately grouped.

### 1. Embeddable Live Belief Widget (iFrame Endpoint)

**Type:** Feature
**Effort:** Small (hours)
**Impact:** The watch page (`/watch/<id>`) shows a full-viewport live spectator view. But embedding a simulation result in a Substack post, Ghost article, Notion page, or Discord pinned message requires an `<iframe>` — a compact, chrome-free widget that fits in a content column. Today that use case has no clean path: pasting the `/share/<id>` URL gets an OG preview image (static), pasting `/watch/<id>` gets a full-page experience that breaks a blog layout, and the replay GIF is motion-only with no live data. A `GET /embed/<sim_id>` endpoint (no `/api` prefix, mirrors the `/share` and `/watch` blueprint) returns a self-contained HTML page designed for embedding: 100% width, 120px height, dark background, three belief bars (bullish/neutral/bearish with proportional widths and percentage labels), a round counter, and a live pulsing badge that disappears when the sim finishes. A 30s vanilla-JS poller hits the existing `/api/simulation/:id/embed-summary` endpoint — exactly what the watch page uses — and updates the DOM in place. No cross-origin issues (served from the same origin), no iframe-busting scripts, `X-Frame-Options: SAMEORIGIN` allows embedding from the same host. EmbedDialog gains an "Embed on your site" section with a copy-ready `<iframe src="/embed/<id>" width="100%" height="120" frameborder="0"></iframe>` snippet and a 120×120 live preview. The result: a researcher can drop one line into their Substack and their readers see the live belief distribution as it updates.
**How:**
1. Add `backend/app/services/embed_renderer.py` (~200 LoC, pure stdlib `html` + `json`): `STANCE_THRESHOLD = 0.2` (parity constant), `_build_belief_bars_html(bullish, neutral, bearish)` renders three `<div class="bar bar-{stance}">` elements with inline `width` set to the percentage value and a `<span class="pct">N%</span>` label, zero-width safety check (bars never render negative). `_build_poller_js(sim_id)` emits the inline `<script>` block: `setInterval` at 30000ms calling `fetch('/api/simulation/<id>/embed-summary')`, updates the three bar widths + percentage labels + round counter via `innerHTML`, stops polling when `run_complete === true`. `render_embed_html(summary, sim_id)` assembles the full document: `<meta name="robots" content="noindex">` (prevent the widget URL from appearing in search), `Content-Security-Policy: frame-ancestors 'self'` header, 200 lines of inline CSS (dark background `#0d0f14`, bar colors matching the watch page palette: `#22c55e` bullish / `#6b7280` neutral / `#ef4444` bearish), the belief bars div, a `<div class="meta">Round N of M · <span class="live-badge">● Live</span></div>` (live badge hidden via JS when `run_complete` is true), and the poller script. Keep total HTML under 8KB so it loads fast in embedded contexts.
2. Add `backend/app/api/embed.py` (~80 LoC): `_build_summary_for_embed(simulation_id)` mirrors the pattern from `watch.py` — calls `SimulationManager` for the sim object, checks `is_public` (403 if not public; the embed endpoint does NOT support the `?t=` private token — private sims aren't embeddable), reads `final_consensus` from `trajectory.json`, falls back gracefully on missing data, returns `{bullish, neutral, bearish, current_round, total_rounds, run_complete, scenario}`. Route handler: `GET /embed/<simulation_id>` — no `/api` prefix (same as `/share` and `/watch`), `Cache-Control: public, max-age=30` (short TTL so the widget refreshes for in-flight sims), calls `_build_summary_for_embed` → `render_embed_html`. Register `embed_bp` with prefix `""` in `_BLUEPRINT_PREFIXES`. 15 offline unit tests in `test_unit_embed.py`: correct HTML structure for in-flight sim (live badge present, poller script present, bars render with proportional widths), completed sim (live badge absent, poller `clearInterval` present), zero-belief bars (all zero-width, no divide-by-zero), private sim → 403, missing sim → 404, `STANCE_THRESHOLD == 0.2` parity guard, `<meta name="robots">` present, `Cache-Control` header short, `render_embed_html` output is valid HTML (does not raise `html.parser.HTMLParser`), route-decorator presence guard.
3. In `EmbedDialog.vue`, add an "Embed on your site" section beneath the "Watch live" section. Show the section only when `isPublished` is true (embed requires a public sim — consistent with all other share surfaces). Layout: a `<iframe>` live preview (same dimensions as the widget: `width="100%" height="120"`), the copy-ready snippet (`<iframe src="<embed_url>" width="100%" height="120" frameborder="0"></iframe>`), and a "Copy embed code" button (same clipboard + toast pattern as the existing copy buttons). Add `getEmbedUrl(simId)` helper to `frontend/src/api/simulation.js`. Add `openapi.yaml` entry for `/embed/{simulation_id}` under the Publish & Embed tag with `text/html` response — drift-detection test passes. Add "Embeddable Widget" row to `docs/FEATURES.md` under Publish & Embed. Add a note to `README.md` Features table.

### 2. Webhook Delivery Log

**Type:** DX / Feature
**Effort:** Small (hours)
**Impact:** PR #46 shipped the outbound completion webhook. Operators configure a URL and trust it fires when a simulation ends. But there is currently no feedback loop: if the endpoint returns a 503, if the request times out, or if the URL was misconfigured, the operator never finds out unless they check server logs manually. Every Zapier/Make/n8n integration built on this webhook is flying blind. A delivery log — `<sim_dir>/webhook-log.jsonl`, one line per dispatch attempt — costs almost nothing architecturally (the dispatch code already runs synchronously before fire-and-forget). Each line records `{attempt, timestamp, url_masked, status_code, latency_ms, error}` with `url_masked` following the existing `scheme://host/***` masking from the webhook code (so the Slack/Discord secret never persists to disk). `GET /api/simulation/:id/webhook-log` (admin-token protected, same `require_admin_token` decorator as `/publish`, `/resolve`, `/outcome`) returns the last 10 entries as JSON. In the EmbedDialog "Webhook" section, a "Delivery history" panel shows the last 3 deliveries as status chips: `✓ 200 · 234ms` in green, `✗ 503 · 1.2s` in red, `✗ timeout` in amber. A "Retry" button re-dispatches the webhook for a completed sim (POST to a new `/api/simulation/:id/webhook-retry` endpoint that re-reads the existing `outcome_payload` and fires it again). Closes the production observability gap that every webhook-dependent integration eventually hits.
**How:**
1. Modify `backend/app/services/webhook.py` (PR #46's dispatch logic): after the HTTP call returns (or times out / raises), append a JSON line to `<sim_dir>/webhook-log.jsonl`. Fields: `attempt` (monotonically incrementing integer — read current count from log before writing), `timestamp` (ISO 8601 UTC), `url_masked` (the same `scheme://host/***` masking already in the codebase), `status_code` (integer or `null` on network error), `latency_ms` (integer, wall-clock time of the HTTP call), `error` (string from exception message, `null` on success). Keep the log bounded: if the file has ≥50 lines on write, truncate to the last 49 before appending (read-modify-write with a temp file, atomic rename). The existing fire-and-forget daemon thread handles the latency; the log write happens inside that thread so the main response path is unaffected.
2. Add two new routes to `backend/app/api/simulation.py` (or a new `webhook.py` blueprint): `GET /api/simulation/<id>/webhook-log` — admin-token protected via `require_admin_token`, reads `webhook-log.jsonl`, returns `{entries: [...last 10...], total_attempts: N}`. `POST /api/simulation/<id>/webhook-retry` — admin-token protected, checks that the sim is in a completed state (same status check as the existing webhook trigger), re-calls the webhook dispatch function with the same `outcome_payload` that was originally sent (reconstructed from `simulation_state.json`), appends a new log entry. Return `{queued: true}` immediately (dispatch is async). Add both paths to `openapi.yaml` under Publish & Embed — drift test passes. 12 offline unit tests in `test_unit_webhook_log.py`: log append on success (200), log append on failure (503), log append on timeout/exception, url masking persists to disk, 50-line truncation, last-10 truncation in GET response, `total_attempts` reflects full history, admin-token gate on GET, admin-token gate on retry, retry re-dispatches with correct payload, missing log file → empty entries, `attempt` field increments monotonically.
3. In `EmbedDialog.vue`, add a "Delivery history" row inside the existing Webhook config section (only visible when `WEBHOOK_URL` is configured and the sim is completed). On expand, fetch `GET /api/simulation/:id/webhook-log` with the admin token in the header (same pattern as the existing "Mark outcome" panel which already handles admin-token UI). Render the last 3 entries as status chips: green checkmark + status code + latency for 2xx, red X + status code for 4xx/5xx, amber clock icon for timeout/network error. Show "N total attempts" count. Add a "Retry delivery" button that calls `POST /api/simulation/:id/webhook-retry` and shows a "Queued!" toast on success. If no deliveries recorded yet, show "No deliveries yet — webhook fires on simulation completion." Keep the panel collapsed by default (same disclosure chevron pattern as the existing EmbedDialog sections).

### 3. Twitter/X Thread Formatter

**Type:** Growth / Content
**Effort:** Small (hours)
**Impact:** Aaron's primary distribution channel is X/Twitter. His "he used MiroShark" marketing pattern builds social proof by posting simulation results as tweet threads with narrative context. Today's toolchain for that thread: manually read the replay GIF, manually read the transcript, manually write 8–12 tweets fitting within 280 chars each. A `GET /api/simulation/:id/thread.txt` endpoint generates the thread automatically from data already on disk: one intro tweet (scenario summary + consensus label), one tweet per *belief inflection point* (rounds where the dominant stance crossed the ±0.2 threshold in either direction — the moments of drama), and a closing tweet (final verdict, quality health, share link, and `/watch/<id>` for readers who want the live replay). The thread format is specifically designed for X/Twitter: each tweet ≤280 chars, thread numbering in the intro tweet (`1/`), tweet separators in the `.txt` form, a `.json` form with one element per tweet for programmatic use. EmbedDialog gains a "Thread" section with full-thread copy (one click) and per-tweet copy buttons. Complements the share card (visual hook), replay GIF (motion narrative), and transcript Markdown (long-form prose) with the *short-form text thread* format — the last major distribution format the share-surfaces arc has not yet covered. With Aaron actively running "try this yourself" marketing at >15 sims per week, a 1-click thread copy reduces the friction between running a sim and posting about it from ~10 minutes to ~10 seconds.
**How:**
1. Add `backend/app/services/thread_formatter.py` (~250 LoC, pure stdlib): `STANCE_THRESHOLD = 0.2` constant (parity with all other surfaces), `MAX_TWEET_CHARS = 280`, `MAX_THREAD_TWEETS = 15`. `_find_inflection_points(rounds)` takes the per-round trajectory (list of `{round, bullish_pct, neutral_pct, bearish_pct}`) and returns indices where the dominant stance (per `dominant_stance()` logic — same ±0.2 threshold) changes. `_format_stance_line(bullish, neutral, bearish)` returns `"↑ Bullish N% · → Neutral N% · ↓ Bearish N%"` (ASCII arrows, no emoji — maximizes compatibility). `_truncate_scenario(scenario, max_chars)` trims at the last word boundary before `max_chars` and appends `…`. `build_thread(sim_id, trajectory_data, scenario, consensus_label, quality_health, total_rounds, share_base_url)`: intro tweet = `"{truncated_scenario}\n\nRunning: {total_rounds} rounds · {total_agents} agents\nConsensus: {consensus_label} 🦈\n1/"` (≤280); per-inflection tweet = `"Round {N}: stance shifted to {new_label}\n{stance_line}"` (≤280); close tweet = `"Final: {consensus_label} consensus · Quality: {quality_health}\n\nWatch the replay: {watch_url}\nRun this scenario: {share_url}"` (≤280). If inflection count would push thread over `MAX_THREAD_TWEETS`, keep only the first 3 and last 3 inflections with a `"… {N} more flips …"` bridge tweet. Return `{tweets: [str, ...], total: N}`. `render_thread_txt(tweets)` joins with `\n---\n` separators for the `.txt` form.
2. Add `_serve_thread(simulation_id, fmt)` to `backend/app/api/simulation.py` (mirrors `_serve_transcript` pattern): reads `trajectory.json` + `simulation_config.json` (for scenario) + `quality.json` + `resolution.json` (for consensus label), calls `build_thread(...)`, returns either the `.txt` form (`Content-Type: text/plain; charset=utf-8`, `Content-Disposition: inline`) or the `.json` form (`Content-Type: application/json`). Both routes are publish-gated (`is_public` check, 403 otherwise). `Cache-Control: public, max-age=300`. Add route decorators `GET /api/simulation/<id>/thread.txt` and `GET /api/simulation/<id>/thread.json` sharing the body. Add both paths to `openapi.yaml` under Publish & Embed with a `SimulationThread` schema (`{tweets: [string], total: integer}`) — drift-detection test passes. `getThreadTxtUrl` / `getThreadJsonUrl` helpers in `frontend/src/api/simulation.js`. 18 offline unit tests in `test_unit_thread.py`: `STANCE_THRESHOLD == 0.2` parity, inflection detection on 3-phase trajectory (bullish → neutral → bearish), no inflections (single-stance sim returns intro + close only), `MAX_THREAD_TWEETS` truncation with bridge tweet, each tweet ≤280 chars, long scenario truncated with `…`, route-decorator presence for `.txt` and `.json`, `.txt` separator format, `.json` round-trip via `json.loads`, publish gate 403, missing trajectory → empty inflections + close-only thread, corrupt trajectory graceful degradation.
3. In `EmbedDialog.vue`, add a "🧵 Thread" section beneath the "Export transcript" row (publish-gated). Layout: a "Copy full thread" button (copies the full `.txt` content via clipboard, same 2.2s "✓ Copied" toast pattern); beneath it, a collapsible "Individual tweets" panel that renders each tweet in a monospace blockquote with a small copy icon at the top-right (copies just that tweet's text). Add tweet count badge in the section header: "🧵 Thread (N tweets)". On first render, fetch `/api/simulation/<id>/thread.json` (not `.txt`) so the individual-tweet copy buttons have per-tweet text. ### 4. Private Share Link (Unlisted Sims via Secret Token)

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Issue #70 (filed May 4) requests a "Private Impact mode" — the ability to share a simulation privately before deciding to publish it publicly. Right now the binary is stark: a sim is either fully private (no sharing at all) or published to the public gallery (visible in `/explore`, RSS feeds, and searchable via the new gallery filters). There is no "share with a specific audience" mode — no "send this to a client for review before going public." A per-sim secret token, a random UUID generated on demand and stored in `<sim_dir>/share-token.json`, enables all nine share surfaces (share card, replay GIF, transcript, trajectory, watch page, thread, embed widget) to serve an unpublished sim when the token is present in the URL as `?t=<token>`. The operator shares `/share/<id>?t=abc123` and the recipient sees a full result; the sim never appears in the public gallery, the RSS feed, or the embed widget (which stays public-only). Directly resolves the open issue. Small scope: one POST endpoint, one UUID file, a `?t=` check wired into the existing `is_public` gate in every share route. No new dependencies — `secrets.token_urlsafe` is stdlib.
**How:**
1. Add `POST /api/simulation/<id>/share-token` (admin-token protected via `require_admin_token` decorator): generates a 32-byte URL-safe random token via `secrets.token_urlsafe(32)`, writes `{token: str, created_at: ISO8601}` to `<sim_dir>/share-token.json` (overwrites if a token already exists — rotation is a feature: the old link goes dead). Returns `{token: str, share_url: str}` where `share_url` is the base `/share/<id>?t=<token>` URL. Add `DELETE /api/simulation/<id>/share-token` (admin-token protected) to revoke the token: deletes `share-token.json`. Returns `{revoked: true}`. Both paths to `openapi.yaml` under Publish & Embed — drift test passes. Add a helper function `_check_share_token(simulation_id, request_token)` to `simulation.py`: reads `<sim_dir>/share-token.json`, returns `True` if `request_token` matches the stored token using `hmac.compare_digest` (constant-time comparison, same pattern as the existing admin-token check), `False` on any error (missing file, corrupt JSON, token mismatch). 12 offline unit tests in `test_unit_share_token.py`: generate stores file + returns token, generate overwrites existing, revoke deletes file, revoke missing file → no-op, token check passes on correct token, token check fails on wrong token, token check fails on missing file, constant-time comparison (no timing oracle), admin-token gate on generate, admin-token gate on revoke, `secrets.token_urlsafe(32)` produces 43-char URL-safe string, round-trip through `json.loads`.
2. In every share-route handler that currently checks `is_public` — `GET /share/<id>`, `GET /watch/<id>`, `GET /api/simulation/<id>/share-card.png`, `GET /api/simulation/<id>/replay.gif`, `GET /api/simulation/<id>/transcript.md`, `GET /api/simulation/<id>/transcript.json`, `GET /api/simulation/<id>/trajectory.csv`, `GET /api/simulation/<id>/trajectory.jsonl`, `GET /api/simulation/<id>/thread.txt`, `GET /api/simulation/<id>/thread.json` — extend the gate: if `is_public` is false, call `_check_share_token(simulation_id, request.args.get('t'))`. If the token matches, treat the sim as accessible for this request (do not add it to any public listing). If both `is_public` is false and the token check fails, return the existing 403/404 response. The embed endpoint (`GET /embed/<id>`) is explicitly excluded: the embed widget is a public-facing live surface and should not be accessible via private token (embedding an unlisted sim in a public blog post would leak the content indirectly). Note this exclusion in a comment in `embed.py`.
3. In `EmbedDialog.vue`, add a "🔒 Private share link" section that is visible when the sim is NOT published (and hidden when published, since published sims don't need a private link). Layout: a "Generate private link" button that calls `POST /api/simulation/:id/share-token` with the admin token; on success, shows the generated URL in a copy-ready input + a "Copy link" button. A "Revoke link" button (calls `DELETE`) with a confirmation step ("This will break the existing link — are you sure?"). A note: "This link gives access to all share surfaces (card, GIF, transcript, trajectory, watch page) without publishing to the public gallery. Revoke it any time." If a token already exists on mount (fetch `HEAD /api/simulation/<id>/share-token` to check existence without revealing the token), show "A private link is active — regenerate or revoke." Add `getShareTokenUrl` + `revokeShareToken` helpers to `frontend/src/api/simulation.js`. Add "Private Share Link" row to `docs/FEATURES.md` under Publish & Embed, referencing issue #70. Add a note to `README.md`.

### 5. Simulation Tagging System

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Gallery search (PR #69) handles discovery by keyword and consensus. But there's a use case search can't serve: *curated groupings*. An operator running MiroShark for institutional clients wants to publish "all our DeFi risk sims" as a named page they can link to, not a keyword search they have to explain. Researchers want to tag sims by methodology ("multi-round", "polymarket-seeded") and filter to a clean corpus before running analysis. Tags are the editorial primitive: operator-applied at publish time (or any time after), stored in `<sim_dir>/tags.json`, surfered in the gallery and on share pages. `GET /api/simulation/public?tag=defi` extends the filter system from PR #69 exactly as `?consensus=bearish` did — same `gallery_filters.py` module, same composition logic. A `/explore/tags` route lists all tags with sim counts (a navigable tag cloud). The share page and watch page show the tag list beneath the scenario so viewers know what category this result lives in. Tags are admin-token-protected on write (only the operator can tag their own sims) but publicly visible on read — the same read/write split as `/outcome`. Zero new dependencies; all filtering is in-memory over existing on-disk JSON.
**How:**

Add `GET /api/simulation/<id>/tags` (no auth required): reads and returns `{tags: [...]}`. Extend `_build_gallery_card_payload()` to include `tags: []` from `tags.json` (empty list if file missing). Add `tags` to the `GalleryCard` schema in `openapi.yaml`. Extend `gallery_filters.py` `filter_cards()` to support `tag` parameter: a card passes if the requested tag is in its `tags` list (case-insensitive, already normalised). Add `_normalise_tag()` validator. Add `tag` to `select_filtered_cards()` signature and to `GET /api/simulation/public` query params. Add a new `GET /api/simulation/public/tags` endpoint: iterates all public sim cards (re-uses the same card-assembly loop), aggregates tag counts, returns `{tags: [{tag: str, count: int}, ...]}` sorted by count descending. 15 offline unit tests: PUT validates format (rejects uppercase, rejects >30 chars, rejects >10 tags, rejects invalid chars), normalizes to lowercase, writes file, GET reads file, `_build_gallery_card_payload` includes tags, `filter_cards` by single tag, tag filter is case-insensitive, filter with unknown tag returns empty, `tags` endpoint aggregation, tags absent → `[]` in card, route-decorator presence for PUT/GET/tags-endpoint, drift test guard.

When a tag is active, show a clear icon on the chip (same "Reset" button logic). The `/explore/tags` route renders a standalone tag cloud page: a heading "Browse by tag", each tag as a larger chip showing the tag name + sim count, clicking navigates to `/explore?tag=<slug>`. Add a "Browse by tag →" link in the ExploreView header (visible only when at least one tag exists in the system). ## Selection Rationale

The surface-multiplication arc closed last week. This batch targets the layer *above* the surfaces: operational visibility, access control, distribution mechanics, and editorial curation.

- **Embeddable Live Belief Widget** (#1) — The share card unfurls, the GIF auto-plays, but there's still no way to *embed a live result* in a Substack post or Notion page. The watch page already did all the engineering; the iFrame variant is a constrained wrapper around the same poller. High distribution leverage; small effort.
- **Webhook Delivery Log** (#2) — PR #46 shipped the webhook; there's been no feedback loop since. Every Zapier/Make/n8n integration is flying blind. A jsonl append per dispatch + a read endpoint + EmbedDialog delivery chips is the operational closure that a production feature needs. Small effort; zero new deps.
- **Thread Formatter** (#3) — Aaron distributes on X. Copy-pasting the share card link takes one click; writing the tweet thread still takes 10 minutes. The data for the thread (trajectory + transcript) is already on disk. A `thread.txt` endpoint generates the narrative automatically. Closes the last major distribution format gap. Small effort; zero new deps.
- **Private Share Link** (#4) — Directly resolves open issue #70. The binary publish/private toggle has no "share with a client" middle state. A UUID token unlocks all nine share surfaces for the token holder without touching the public gallery. One POST endpoint, one stdlib `secrets.token_urlsafe` call, constant-time comparison. Small effort; closes a real open issue.
- **Simulation Tagging** (#5) — Gallery search finds what you're looking for; tags surface what an operator has deliberately grouped. Named collections ("DeFi risk", "Q2 macro", "verified ETH") are the editorial layer PR #69 didn't provide. Extends `gallery_filters.py` exactly as the existing filters do. Small effort; zero new deps; composes with all existing filter params.

Each idea is scoped for autonomous implementation by the `feature` skill — all data is on disk or in existing endpoints, no ambiguous design decisions, no external approvals required.
