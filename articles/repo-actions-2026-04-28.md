## Ecosystem Context

The past two days completed two distinct arcs simultaneously. The *distribution arc* landed the animated GIF (PR #50) and admin-token auth (PR #48/#49) — every published simulation is now a motion asset on X/Discord, and mutation surfaces are locked. The *observability arc* landed Langfuse-tagged traces (PR #51) and a Langfuse-surfaced cost investigation that produced the retry-loop fix and context cap (PR #52) — 30-40% cheaper sims, bounded loop blast radius, and a Langfuse session per simulation ready to visualize.

These two arcs converge on a gap: the cost improvements are invisible to users, and the observability infrastructure isn't surfaced in the product. The "$1 & under 10 min" tagline can now be *verified* inside the app — but there's no panel that shows it. Meanwhile, the transcript that agents produce (now clean post-PR #52) has no export path, the public gallery has no passive subscription surface, and first-time users still face a blank textarea when they open the simulation form.

Four signal threads drive this batch:

- **Cost transparency:** The "$1 & under 10 min" claim is now backed by Langfuse traces with per-phase cost attribution. Surfacing that in the UI is the difference between a tagline and a verifiable guarantee. The quant/researcher audience evaluating production costs will immediately look for this.
- **Research portability:** The GIF covers motion. The share card covers OG previews. Nothing covers text — the actual agent reasoning per round. A transcript export makes sim output quotable in research papers, Substack posts, and Discord threads without a click-through.
- **Passive discovery:** The public gallery is pull-only (you visit it). An RSS feed converts it to push — researchers subscribe in Feedly/Readwise the same way they subscribe to AI newsletters. Zero maintenance, zero ops, pure stdlib.
- **First-sim conversion:** The blank scenario textarea is the primary barrier for first-time users. A curated template library (DeFi vulnerability, token launch, regulatory action, governance vote) eliminates the blank-page problem at the point of highest friction.

Previously suggested ideas excluded from this batch (last 7 days): Share as Thread Formatter (#3, Apr 26), Python Client SDK (#4, Apr 26), Director Event Overlay on Belief Chart (#5, Apr 26), SSE Streaming (#1, Apr 24), Simulation Engagement Leaderboard (#2, Apr 24), Discord/Slack Post Button (#4, Apr 24), History Search & Tags (#4, Apr 22). ### 1. Langfuse Cost Breakdown Panel

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** PR #51 tags every OpenRouter call with `sessionId = sim_id`, `sim_phase` (setup/round/research), `round`, `agent_id`, and `prompt_type`. The cost data is in Langfuse — it just isn't surfaced in the product. A `GET /api/simulation/:id/cost` endpoint calls Langfuse's trace-filter API with the session ID and returns `{total_usd, total_tokens, breakdown: {setup, rounds, research}}`. In the SimulationView "Metrics" card (or a new "Cost" row alongside quality/round count), show "Total: $0.47 — setup 9%, rounds 74%, research 17%". For the Settings panel, a "Current session cost" meter during a live run. The "$1 & under 10 min" value prop stops being a tagline and becomes a live counter. The quant/researcher audience evaluating MiroShark as production infrastructure will look for exactly this — it's the difference between "claims to be cheap" and "shows its receipts."
**How:**
1. Add `GET /api/simulation/:id/cost` — reads `LANGFUSE_SECRET_KEY` / `LANGFUSE_PUBLIC_KEY` / `LANGFUSE_HOST` from env. Calls `GET /api/sessions/:sim_id/observations` (or trace-filter endpoint), aggregates `usage.input_cost + usage.output_cost` across all observations, groups by tag `sim_phase`. Returns `{total_usd, total_tokens, breakdown: {setup: {usd, pct}, rounds: {usd, pct}, research: {usd, pct}}}`. Returns `{available: false, reason: "langfuse_not_configured"}` when env vars are absent — never 500s. Add `LANGFUSE_HOST`, `LANGFUSE_SECRET_KEY`, `LANGFUSE_PUBLIC_KEY` to `.env.example` with a comment "optional — enables in-app cost breakdown."
2. In `SimulationView.vue`: after simulation completes, fire a secondary `GET /api/simulation/:id/cost` fetch. If `available: true`, add a "Cost" row to the metrics card: total USD (2 decimal places) + expandable breakdown bars (setup/rounds/research, proportional width, muted colors). Show a "—" placeholder while loading; silently skip if unavailable. In the Settings modal's Langfuse section (or add one if absent), add a "Cost tracking" status badge: green "Active — cost breakdown enabled" when keys are set, grey "Not configured" otherwise, with a one-line note "Set LANGFUSE_* env vars to enable per-simulation cost breakdown."
3. Add `cost` to `GET /api/simulation/:id` response — a nested object or `null`. Expose it in `openapi.yaml` under the `SimulationStatus` schema so the drift-detection test catches any rename. Add a "Cost breakdown" row to `docs/FEATURES.md` under Observability. Update the README to note that the "$1 & under 10 min" claim is verifiable via in-app cost tracking when Langfuse is configured. This changes the narrative from "we claim cheap" to "we show you the receipt."

### 2. Simulation Transcript Export

**Type:** Feature / Community
**Effort:** Small (hours)
**Impact:** Every completed simulation produces a belief chart, a share card, and a GIF. None of these are quotable — you can't paste a bar chart into a research paper or cite a GIF in a Substack post. A per-round agent transcript (`GET /api/simulation/:id/transcript.md`) is the text companion: scenario header, round-by-round agent stance + post + key reasoning excerpt, final consensus summary. PR #52 cleaned the agent conversation history (`prune_tool_calls_from_memory=True`) so the transcript is now readable, not a wall of tool-call boilerplate. Download as Markdown or JSON from the EmbedDialog. Makes MiroShark outputs citable — the academic/newsletter audience wants to quote "Agent #3 (bearish, risk-analyst archetype) posted in round 7: 'The TVL drawdown pattern matches the Euler Finance exploit...' and shifted the sentiment 12% bearish" — that's a direct pull quote from the transcript. Pairs with the GIF as a dual-format share: post the GIF for socials, link the transcript for depth.
**How:**
1. Add `GET /api/simulation/:id/transcript.md` and `GET /api/simulation/:id/transcript.json`. Both read `trajectory.json` (per-round agent states, posts, stances), `simulation_config.json` (scenario), and `quality.json` (final scores). The Markdown format: YAML front-matter block (sim_id, scenario, agents, rounds, consensus), then one `## Round N` section per round with a `### Agent Name (stance)` subsection containing the agent's last post (truncated at 400 chars with `…` if longer) and a one-line stance attribution. Final `## Consensus` section with the belief split and quality health. The JSON format: `{sim_id, scenario, rounds: [{round, agents: [{name, stance, post_excerpt, stance_delta}]}], consensus}`. Publish-gated (same 403 as share card). Add `GET /api/simulation/:id/transcript.md` and `.json` to `openapi.yaml` under Publish & Embed — drift-detection test will catch any route rename.
2. In `EmbedDialog.vue`, add a "📄 Export transcript" section below the "Belief replay (animated)" block. Two buttons: "Download .md" (fetches transcript.md, triggers download via `<a href=blob>`) and "Download .json". Include a copyable URL for the `.md` endpoint — Notion, Obsidian, and Bear can import Markdown from URL. Add a preview card showing the first round's top agent post excerpt (one line, truncated) so users know what they're downloading before they click.
3. Add `getTranscriptUrl()` helper to `frontend/src/api/simulation.js`. Add a "Transcript export" row to `docs/FEATURES.md` under Sharing. Add the endpoint pair to the README Features table under Export. Log transcript download events (increment a `transcript_download_count` counter in `<sim_dir>/state.json`) alongside the existing share-view and fork counts — this feeds into the engagement scoring for the eventual leaderboard (still unbuilt from Apr 24).

### 3. RSS Feed for Public Gallery

**Type:** Integration / Growth
**Effort:** Small (hours)
**Impact:** The `/explore` gallery is a pull surface — you visit it. Researchers who follow AI/DeFi tools via RSS subscriptions (Feedly, Readwise, Inoreader, Obsidian RSS) have no way to follow MiroShark's public simulations without manually checking the gallery. A `GET /api/feed.rss` endpoint serving the 20 most recent public simulations as an Atom 1.0 feed converts the gallery to push. Each entry: scenario (title), consensus split (subtitle), quality health + agent count (summary), share URL (link), GIF URL (`<media:content>`). Zero new Python dependencies — `xml.etree.ElementTree` is stdlib. One listing on Feedly's developer directory and a "Subscribe via RSS" badge on the `/explore` page creates a passive subscription channel that compounds with every new public simulation. The MiroShark-adjacent audience already reads AI/crypto research via RSS; this meets them in their existing toolchain.
**How:**
1. Add `GET /api/feed.rss` and `GET /api/feed.atom` (same generator, different format flag). Calls `_build_gallery_card_payload()` for the 20 most recent public simulations (reuses existing gallery query). Builds Atom 1.0 XML using `xml.etree.ElementTree` (stdlib): feed-level `<title>`, `<link rel="self">`, `<updated>` (most recent sim timestamp). Per-entry: `<id>` (share URL), `<title>` (scenario truncated to 80 chars), `<updated>` (sim created_at), `<summary>` ("🔵 X% Bullish · ⚪ Y% Neutral · 🔴 Z% Bearish | Quality: excellent | Agents: N"), `<link href=share_url>`, `<media:content url=replay_gif_url type="image/gif" medium="image" />` (if GIF exists). Returns `Content-Type: application/atom+xml`. Add a 5-minute `Cache-Control: public, max-age=300` header so CDN caches don't hammer the gallery query. `?limit=20` hardcoded; `?verified=1` pass-through for a Verified-only feed.
2. Add a "Subscribe via RSS" link to `ExploreView.vue` — a small feed icon (📡 or SVG) in the explore page header next to the filter chips. Links to `/api/feed.rss` with `target="_blank"`. Add a second chip "Verified only" RSS variant linking to `/api/feed.rss?verified=1`. In `EmbedDialog.vue`, add a one-liner "📡 Follow this gallery via RSS → /api/feed.rss" in the Sharing section so users who find a sim through the share card can subscribe to the author's public sims flow.
3. Add `GET /api/feed.rss` and `GET /api/feed.atom` to `openapi.yaml` under Publish & Embed with `text/xml` response. Add a `<link rel="alternate" type="application/atom+xml" href="/api/feed.rss">` meta tag to the app's main `index.html` so browsers auto-discover the feed (the globe icon in the address bar). Submit to [Feedly's source list](https://feedly.com/i/discover) and [OSRS.io](https://osrs.io) once live. This is a zero-maintenance discovery channel — every new simulation a user publishes appears in the feeds of every subscriber automatically.

### 4. Scenario Template Library

**Type:** DX / Community
**Effort:** Small (hours)
**Impact:** The blank scenario textarea on the New Simulation form is the primary conversion barrier for first-time users. `suggest_scenarios` is LLM-powered and takes seconds to generate; users who want to start immediately can't. A built-in library of 12–15 curated scenario templates — DeFi protocol vulnerability, token launch, regulatory crackdown, governance vote, CEX listing, earnings announcement, geopolitical shock, protocol upgrade, stablecoin depeg, oracle manipulation, AI model release, ETF approval/rejection — available instantly from a "Browse templates" picker eliminates the blank-page problem at the point of maximum friction. Each template: pre-written scenario text (2–3 sentences, deployable as-is), a suggested agent count range, a recommended preset (Cheap/Smart/Research), and a category tag. Static JSON file, zero new dependencies, zero new endpoints. The onboarding arc for new users becomes: land → pick template → adjust agent count → run — 3 clicks to first result.
**How:**
1. Create `frontend/src/data/scenario_templates.json` — a static array of 12–15 template objects: `{id, category, title, scenario, suggested_agents, preset, tags}`. Categories: DeFi Protocol, Token Event, Regulatory, Macro/Geo, Tech Release. Write the scenario text to be deployable as-is (2–3 sentences, specific enough to trigger differentiated agent responses, generic enough to not require a specific ticker). Example: `{category: "DeFi Protocol", title: "Protocol Liquidity Crisis", scenario: "A major lending protocol has paused withdrawals following anomalous outflows exceeding $200M in 4 hours. Preliminary analysis points to either an oracle manipulation attack or a governance exploit. No official statement has been issued.", suggested_agents: 12, preset: "Smart", tags: ["defi", "security", "liquidity"]}`. Include 2–3 per category for variety.
2. Add a "Browse templates →" text button adjacent to the scenario textarea in `SimulationRun.vue` (or equivalent new-sim form component). On click: open a `TemplateLibraryModal.vue` — a compact dialog with category tabs (DeFi / Token / Regulatory / Macro / Tech) and template cards. Each card: title, 1-sentence scenario preview (first sentence only), suggested_agents badge, preset badge, "Use template" button. On "Use template": close modal, populate scenario textarea with full text, set agent count to suggested_agents if user hasn't changed the default, set preset to recommended. Add a "Randomize template" shortcut (🎲) that picks a random template and applies it — for users who want to explore without browsing.
3. Add 3 template titles to the README hero as concrete examples under the "Simulate anything" headline: "Protocol liquidity crisis · Governance vote · Token launch — pick a template or describe your own scenario in plain language." This surfaces the template concept at the top of the funnel for new visitors reading the README. Add `GET /api/simulation/templates` as a read-only endpoint serving the same JSON so external integrations (Claude Desktop via MCP, Python SDK) can list available templates. Add to `openapi.yaml` under Setup & Discovery.

### 5. Comparative Run View

**Type:** Feature / Research
**Effort:** Medium (1-2 days)
**Impact:** The fork button copies a sim config. The Cheap preset lets users run the same scenario at 3× lower cost with different models. But there's no way to compare two runs of the same scenario side by side — no diff view showing "Smart preset: 67% bearish consensus / Cheap preset: 52% bearish consensus / delta: -15%." A `GET /compare?a=<sim_id>&b=<sim_id>` view renders the two sims in a locked-scroll panel: same scenario header confirming they share a scenario (or a "different scenario" warning), side-by-side belief charts with a delta line, consensus diff pill ("+15% more bearish on Smart"), agent count/preset/cost comparison row, quality health badges, GIF for each side. Enables the "model comparison" research format the quant/academic audience wants to publish: "We ran the Aave depeg scenario across 3 model configurations and found that..." Makes the fork-then-compare workflow a first-class feature instead of a workaround. Shareable via `/compare?a=x&b=y` URL.
**How:**
1. Add `GET /api/simulation/compare?a=<id>&b=<id>` — reads both sim dirs, returns `{a: {sim_id, scenario_excerpt, preset, agent_count, consensus, quality, cost_usd (if Langfuse configured)}, b: {...same fields...}, delta: {bullish: float, bearish: float, neutral: float}, scenario_match: bool (first 100 chars of scenario match)}`. Returns 404 if either sim doesn't exist, 403 if either isn't public (unless admin token present). Cap at scenarios that are at least 70% similar (cosine over trigrams) to make `scenario_match: true` — otherwise show a "different scenarios" advisory but don't block. Add `GET /api/simulation/compare` to `openapi.yaml` under Analytics.
2. Add a `CompareView.vue` route at `/compare` — reads `a` and `b` from URL query params. Calls `GET /api/simulation/compare`. If `scenario_match: false`, shows a yellow advisory banner. Main layout: locked-scroll two-column grid. Left column: scenario header (shared), Sim A metadata row (preset, agent count, cost, date), belief chart (small, reuses existing `BeliefChart.vue`), consensus split bar, quality badge, GIF thumbnail, "View full →" link. Right column: same for Sim B. Center strip between columns: delta pills (🔺 +15% Bearish, 🔻 -8% Bullish) and a "Which is more confident?" one-liner (higher quality score). Responsive: stacks vertically on mobile. Add a "Compare" action to the EmbedDialog — available only when the sim is public. Opens a "Compare with another simulation" dialog: paste or search for a second sim ID/URL, hit "Open comparison →" which navigates to `/compare?a=current&b=other`.
3. Add a "Fork & Compare" button alongside the existing "Fork this →" button on `/explore` gallery cards and the `/share/:id` landing page. Click: forks the sim (calls existing `POST /fork`), redirects to `/compare?a=original_id&b=new_fork_id`. The comparison immediately shows an identical baseline (delta all 0) with a "Now change the preset or agent count and re-run →" CTA — making the fork→modify→compare loop self-explanatory. Add `/compare` to the README Features table and `docs/FEATURES.md`.

## Selection Rationale

Today's batch targets four gaps the observability arc (PRs #51/#52) and distribution arc (PRs #47–#50) left open:

- **Langfuse Cost Panel** (#1) — The Langfuse wiring is done; the cost data exists but is invisible. Surfaces the "$1 & under 10 min" claim as a verifiable in-app metric — the exact signal the quant/researcher audience looks for when evaluating production infrastructure vs. a demo. Purely additive over existing TraceContext work.
- **Transcript Export** (#2) — The GIF covers motion; the transcript covers text. PR #52 cleaned the agent conversation history so the output is readable for the first time. Enables research quotability — the format academic and newsletter authors need to cite simulation outputs without linking to the app.
- **RSS Feed** (#3) — Zero new dependencies, one stdlib XML file, passive subscription channel for the research audience that already follows AI/DeFi content via RSS. Compounds with every new public simulation — no maintenance required after shipping.
- **Scenario Templates** (#4) — The blank textarea is the #1 first-sim conversion barrier. A static JSON library of 12–15 templates eliminates it. Three clicks to first result: pick template → set agent count → run. Zero new endpoints required (static data + optional read endpoint). Directly measurable in terms of first-sim conversion rate.
- **Comparative Run View** (#5) — Fork-then-compare is the research workflow the model-comparison narrative needs. Makes the Cheap preset useful as a scientific control variable, not just a cost option. Shareable `/compare?a=x&b=y` URLs are a new distribution format — a "comparison thread" is a different content format from a single sim share.

Each idea is scoped for autonomous implementation by the `feature` skill — all required data is already in the codebase, no ambiguous design decisions, no external approvals needed.
