## Ecosystem Context

MiroShark now carries one of the deepest feature stacks in open-source social simulation: graph-built personas, multi-platform engine, AMM prediction markets, full observability, fork/branch, history search, comparison mode, LLM provider selector, cloud deploy, influence leaderboard, belief drift chart, prediction resolution, article generator, trace-grounded agent interview, browser push notifications, and mid-simulation event injection. Seven major features shipped in eight days; run summary reporting and multi-model routing landed today. At 698 stars (+7 overnight), 132 forks, and a post-ATH consolidation phase that is cooling gracefully, the project is in a maturation window — not a buildout sprint, but a deepening phase.

The gaps that now matter most are about **discoverability of insights** and **durability of outputs**. MiroShark's analytical layer (belief drift, leaderboard, article generator, trace interview, Director Mode) is increasingly rich — but results are only discoverable in-session. A user who ran a compelling simulation last week can't replay it step by step for an audience; can't share an interactive visualization without sending a link to a private instance; can't tell whether their simulation produced high-quality agent behavior or a degenerate run where half the agents were silent. And every completed simulation is still a closed graph — there's no way to ask "how does this agent pool respond differently to this alternative document?"

Key signals driving this batch:

- **Trajectory data goes unused after completion:** Every completed simulation writes a full `trajectory.json` containing per-round agent actions, belief states, market prices, and interaction events. The belief drift chart and leaderboard consume slices of this data, but the temporal sequence — who said what in round 3 before the consensus formed in round 7 — is invisible. A playback scrubber that lets users step through a completed simulation round-by-round turns trajectory.json into a presentation-ready replay. Researchers can pause at the injection round from Director Mode and walk through the downstream effect. Conference presenters can do a live walkthrough without re-running a 10-minute simulation. Requires no backend changes — trajectory.json is already complete.

- **Agent interactions are recorded but never visualized as a network:** The observability system tracks every reply, amplification, and cross-platform mention. But the resulting interaction graph is never rendered. Which agents are hubs (high in-degree — everyone responds to them)? Which are bridges (connecting the Twitter cluster to the Reddit cluster)? Where are the echo chambers (densely connected clusters with no cross-cluster ties)? A force-directed D3.js network graph, added as a "Network" tab alongside Belief Drift and Leaderboard, answers these questions without any new data collection. For researchers studying information diffusion, it's the structural counterpart to the temporal belief drift chart.

- **The comparison mode compares runs, not documents:** PR #13 added a comparison mode that shows divergence between two previously completed simulation runs. But the research question researchers actually ask most often isn't "how did two runs of the same setup differ?" — it's "how does the same agent pool respond to Document A versus Document B?" Bull case vs. bear case. "Before the news broke" vs. "after." The EU original regulation vs. the amended version. Multi-Document Mode runs the same agent population on two inputs in sequence and auto-generates a side-by-side divergence analysis. This is the controlled experiment primitive that the Director Mode (same-run injection) and Comparison Mode (different runs, same doc) don't cover.

- **No signal on simulation quality:** A 50-agent, 8-round simulation can technically complete while producing degenerate output: 40 of 50 agents silent (participation failure), all agents agreeing from round 1 (no debate), or the belief drift flat-lining immediately (zero dynamics). Currently the only signal is "completed" — no indication of whether the run produced meaningful agent behavior. A quality diagnostics badge computed post-completion (participation rate, stance entropy, cross-platform interaction rate, convergence speed) would tell users: "This simulation had low agent participation (62%). Consider fewer agents or more rounds." It would also flag which parameter combinations tend to produce high-quality output — a feedback loop that makes users better at configuring simulations over time.

- **Completed simulations can't be embedded:** The share permalink is MiroShark's distribution mechanism — a public URL that shows the simulation result. But a link requires a click; an embedded widget requires nothing. Researchers who write in Notion, Substack, or GitHub READMEs want to paste a belief drift chart and market price directly into their writing. An `<iframe>` embed code generated from the share permalink, rendering a minimal read-only visualization (belief drift sparkline, final market price, top 3 agents, scenario title), turns every published simulation into inline content. It's the difference between "here's a link to my simulation" and "here's my simulation."

### 1. Simulation Replay / Playback Mode

**Type:** Feature
**Effort:** Small (hours)
**Impact:** MiroShark's `trajectory.json` contains a complete record of every agent action, belief state, market tick, and interaction event — ordered by round. But this temporal sequence is currently only accessible as a static feed or the belief drift chart's aggregate view. Playback Mode adds a timeline scrubber to completed simulations: a Play/Pause button, a round-by-round stepper, and a progress bar showing "Round 4 of 10." When the user scrubs to round 4, the simulation feed displays only actions from rounds 1–4; the belief drift chart animates to round 4; the leaderboard shows influence rankings up to round 4. Pressing Play advances through remaining rounds at 1-second intervals, reproducing the simulation as it happened. This transforms every completed simulation into a presentation artifact — researchers can walk an audience through a simulation without re-running it, stop at a Director Mode injection point to discuss the perturbation effect, and rewind to compare agent behavior before and after consensus formed.
**How:**
1. Add a `playback` state slice to `Step3Simulation.vue`: `{ mode: 'live' | 'replay', maxRound: N, activeRound: N, playing: false }`. A "Replay" button appears on all completed simulation cards in the history view (alongside Share, Fork, Article). On click, opens the simulation in `replay` mode with `activeRound = 1`. Render a scrubber bar at the top of the simulation view: a progress bar with round labels, a back-one-round button (⏮), a play/pause button (▶/⏸), and a forward-one-round button (⏭). Speed selector: 0.5x / 1x / 2x (controls the auto-advance interval in replay mode).
2. Filter all simulation feed components to only render events where `round_number <= activeRound`. The belief drift chart already accepts data slices — pass `trajectory.slice(0, activeRound)` instead of the full array. The influence leaderboard recalculates from the filtered interaction set for rounds 1 through `activeRound`. Director Mode event banners appear in the feed only when `activeRound` passes the injection round. All filtering is client-side — no new API calls.
3. When `playing = true`, advance `activeRound` by 1 every `(1000 / speed)` ms using `setInterval`. Pause at `maxRound`. Add a "Jump to Director Event" button (visible only for simulations with injected events) that scrubs directly to the round of the first injection. Add a "Reset" button that returns to `activeRound = 1`. Include a keyboard shortcut overlay: Space = play/pause, ← → = step backward/forward. Export the current playback frame as a PNG screenshot (same mechanism as the belief drift chart's existing export button).

### 2. Agent Interaction Network Graph

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** The observability system records every agent-to-agent interaction: Twitter replies, Reddit comment threads, cross-platform amplifications, Polymarket trading reactions. This interaction data lives in `trajectory.json` as a list of `{source_agent, target_agent, type, round}` events — captured but never rendered as a graph. An Agent Interaction Network tab (alongside Belief Drift, Leaderboard, Network) renders a force-directed graph where nodes are agents, edges are interactions, and node size encodes influence rank. The graph answers questions the leaderboard can't: "Are there two distinct clusters of agents that don't interact?" (echo chamber detection), "Which agents bridge the Twitter and Reddit communities?" (bridge centrality), "Did the #1 influence agent achieve dominance by organic reach or by being amplified by a few high-activity followers?" (in-degree vs. out-degree decomposition). For researchers publishing on social simulation, the network graph is a standard figure — and one that MiroShark can generate automatically from data it already has.
**How:**
1. Add a `GET /api/simulation/:id/interaction-network` endpoint that extracts `{source, target, type, round}` interaction events from `trajectory.json`, deduplicates to `{source, target, weight}` (weight = number of interactions), and returns: `{ nodes: [{id, name, stance, influence_rank, platform_primary, type}], edges: [{source, target, weight}] }`. Compute graph metrics server-side using pure Python (no networkx required): degree centrality (total connections / max possible), clustering coefficient approximation (neighbors-of-neighbors overlap), echo chamber score (ratio of same-platform edges to cross-platform edges). Store the result in `<sim_dir>/network.json` for caching.
2. Add a "Network" tab to the simulation results view. Render the graph using `@antv/g6` (lightweight, Vue-compatible, no D3 learning curve) or `vis-network` (zero-config force layout). Node color: teal = bullish, slate = neutral, coral = bearish (matching belief drift palette). Node size: log-scaled by influence rank. Edge thickness: proportional to interaction weight. Edge color: Twitter-only (blue), Reddit-only (orange), cross-platform (purple). On node hover: tooltip showing agent name, stance, platform, top interaction partner. On edge hover: interaction count and type breakdown.
3. Add a sidebar panel to the Network tab with 3 computed insights: (1) **Top Hub**: "Agent X received 23 interactions — more than any other agent." (2) **Top Bridge**: "Agent Y had the highest cross-platform interaction rate (6 Twitter ↔ Reddit edges)." (3) **Echo Chamber Score**: "Twitter agents interacted primarily within their cluster (82% same-platform). Reddit agents were more cross-platform (54%)." Add filter checkboxes: show all agents / Twitter only / Reddit only / Polymarket traders only. Add a "Download as PNG" button (same pattern as belief drift export). Include the network graph in the article generator context — inject the top hub and bridge agent names as "Key network dynamics:" bullets in the generated article.

### 3. Multi-Document Comparative Mode

**Type:** Feature
**Effort:** Medium (1–2 days)
**Impact:** The existing comparison mode (PR #13) surfaces divergence between two separately completed simulation runs. But what researchers actually want to ask is: "How does the same agent population respond differently to Document A versus Document B?" — the bull case versus the bear case, the original press release versus the corrected version, the proposed regulation versus the final text. Multi-Document Mode runs the same agent pool on two inputs in a single workflow: the user uploads Document A, then Document B (or pastes both into a split text input), confirms the agent configuration, and launches. The backend runs both simulations with identical agent IDs and configs. The results view opens directly in comparison mode, pre-loaded with both runs. This is the controlled experiment the Director Mode doesn't cover (Director Mode perturbs a single run mid-stream; Multi-Document compares responses to entirely different stimuli). It answers the question: "Would these agents have reacted differently if the document had said X instead of Y?"
**How:**
1. Add a "Compare two documents" toggle to the Step 1 setup UI, below the document input. When enabled, the input area splits into two panels: "Document A" and "Document B" (each with its own text/URL input and scenario field). A shared config section below both panels covers agent count, rounds, and LLM settings — these apply to both runs. The "Run Simulation" button becomes "Run Comparison." On submit, the frontend sends a `POST /api/multi-doc` request with `{ doc_a: {...}, doc_b: {...}, shared_config: {...} }`.
2. Add a `POST /api/multi-doc` endpoint that creates two simulation entries with a shared `comparison_id` field and an agent pool lock: the same `agent_ids` array is used for both simulations (generated once, reused). After config generation for doc A, clone the agent roster to doc B's config (replacing only the scenario context and document content — all agent identities, demographics, and belief states are identical at t=0). Run both simulations sequentially (doc A first, then doc B when A completes). Poll status via `GET /api/comparison/:comparison_id` which returns `{ sim_a_id, sim_b_id, status_a, status_b }`.
3. When both simulations complete, open the results in a dedicated `MultiDocComparison.vue` layout: two side-by-side belief drift charts (one per document), a merged influence leaderboard showing each agent's final stance in Run A and Run B side-by-side (with a ↑↓ delta indicator), and a "Divergence Score" computed as the mean absolute difference in final stance between the two runs across all agents. Add a "Which document generated stronger consensus?" callout: "Document A reached 71% YES in 6 rounds; Document B reached 43% YES in 10 rounds." Export both simulation configs and a divergence report as a single ZIP. Include multi-doc comparison in the article generator — pass both scenarios and the divergence score as structured input.

### 4. Embeddable Simulation Widget

**Type:** Integration / Growth
**Effort:** Small (hours)
**Impact:** The share permalink is MiroShark's current distribution mechanism — a public URL that renders simulation results. But links require clicks; embedded content requires none. Researchers publishing in Notion, Substack, Medium, or GitHub READMEs want to show their simulation's belief drift chart and market outcome inline — not linked, but rendered. An embeddable widget generates an `<iframe>` embed code from any share token that renders a minimal, read-only visualization: the belief drift sparkline, the final market price badge (YES: 73%), scenario title (truncated to 80 chars), round count, and agent count. The iframe is self-contained — served directly from the MiroShark instance at `/embed/:share_token` — and requires no JavaScript on the host page. For every user who currently shares a MiroShark link in a blog post or paper, this replaces the link with a live embedded chart. It turns the 700+ forks and growing cloud deploy audience into a distribution network: every embedded simulation is a MiroShark logo and link pointing back to the project.
**How:**
1. Add a `/embed/:share_token` route that renders a minimal HTML page (no Vue app shell, no navigation, no settings): just the belief drift chart, a final market price badge, the scenario title, and a small "Powered by MiroShark" footer link. Style it for embedding: transparent background, `border-radius: 8px`, fixed height of 320px. The page accepts query parameters: `?theme=light|dark` (default: light), `?chart_only=true` (hides all text except the chart). Serve with appropriate `X-Frame-Options: SAMEORIGIN` relaxed to `ALLOWALL` for public share tokens, and `Content-Security-Policy` headers preventing script injection.
2. Add an "Embed" button to the share panel (alongside the existing Copy Link and Download buttons on completed simulation cards and the share page). On click, open a small overlay showing: (a) a live preview of the embed at 100% width, (b) the `<iframe>` code snippet in a syntax-highlighted code block with a Copy button, (c) a Markdown image badge: `[![Simulation](...)](/embed/...)` for GitHub README embedding, (d) a Notion-compatible link (Notion auto-embeds URLs that return `text/html`). Include size presets: Compact (480×240), Standard (640×320), Wide (800×400).
3. Add embed analytics to the share tracking: log each time the `/embed/:share_token` route is loaded with a `referer` header (so the instance owner can see which blog posts or READMEs are embedding their simulations). Add `<link rel="alternate" type="application/json+oembed">` to all share pages for oEmbed autodiscovery — this makes Notion, Ghost, and Substack auto-embed MiroShark share URLs when pasted as links. Update the README with an "Embed your simulation" section showing the iframe code pattern and a screenshot of an embedded widget.

### 5. Simulation Quality Diagnostics

**Type:** Feature / DX
**Effort:** Small (hours)
**Impact:** MiroShark simulations complete with a single status: `completed`. But completion doesn't guarantee meaningful output. A 50-agent simulation where 30 agents produced zero posts is technically complete; so is one where all agents reached consensus by round 2 with no debate. Currently there's no signal distinguishing a high-quality run from a degenerate one. A quality diagnostics report, computed once when a simulation transitions to `completed`, measures four dimensions: (1) **Participation Rate** — what fraction of agents generated at least one post/action; (2) **Stance Entropy** — how diverse were final stances (high entropy = meaningful debate; low = echo chamber); (3) **Convergence Speed** — how many rounds before consensus formed (fast convergence suggests weak agent independence); (4) **Cross-Platform Interaction Rate** — ratio of Twitter↔Reddit interactions to total interactions (low = siloed platforms, not a social simulation). These four scores roll up into a single **Health Badge**: Excellent (all four in healthy range), Good (minor issues), or Low (at least one critical failure). Displayed next to the simulation title in history view and in the results header. A tooltip explains the breakdown and suggests parameter adjustments: "Participation rate was 58%. Try reducing agent count to 30 for this document complexity." This closes the feedback loop that helps users configure better simulations over time.
**How:**
1. Add a `compute_quality_diagnostics(sim_dir)` function in `simulation_runner.py` that runs after `status = "completed"` is written. Reads `trajectory.json` and computes: `participation_rate = active_agents / total_agents` (active = at least 1 post/action); `stance_entropy = -sum(p_i * log(p_i))` across the 3 stance buckets (bullish/neutral/bearish) at simulation end (normalized to 0–1 by dividing by log(3)); `convergence_round = first round where one stance bucket > 60%` (or `null` if no convergence); `cross_platform_rate = cross_platform_interactions / total_interactions`. Store as `<sim_dir>/quality.json`. Compute an overall health score: `Excellent` if participation ≥ 80%, entropy ≥ 0.5, no convergence before round 4, and cross_platform ≥ 0.2; `Low` if participation < 60% or entropy < 0.2; `Good` otherwise.
2. Add a `GET /api/simulation/:id/quality` endpoint returning the `quality.json` contents. In the simulation history view, add a small colored dot next to each completed simulation title: green (Excellent), yellow (Good), red (Low). On hover, show a tooltip: "Simulation Health: Good · Participation 74% · Stance diversity: medium · Consensus at round 7." In the simulation results header (Step 3), add a "Quality" chip below the round count showing the health label. Clicking the chip expands a breakdown panel with the 4 metric bars and the plain-text suggestion.
3. Add a parameter suggestion engine to the quality report: if `participation_rate < 0.6`, suggest "Try reducing agent count by 30% for this document length." If `convergence_round < 3`, suggest "Consider increasing rounds to 12+ to allow more debate development." If `cross_platform_rate < 0.1`, suggest "Enable the cross-platform digest option to increase inter-platform agent awareness." Display these as actionable "Try for next run:" chips below the health breakdown. Include the quality score in the article generator context: inject it as a "Simulation quality" note at the top of the generated article brief. Log quality scores to the simulation database for aggregate analysis: `SELECT AVG(participation_rate), AVG(stance_entropy) FROM simulations WHERE completed_at > ?` can inform future default parameter choices.

## Selection Rationale

This batch targets MiroShark's output layer — making completed simulation data more discoverable, more shareable, and more self-diagnosing:

- **Playback Mode** (#1) — Turns trajectory data into a presentation tool. Every completed simulation becomes replayable round-by-round, without re-running it. The most direct path from "research result" to "conference walkthrough."
- **Network Graph** (#2) — The structural counterpart to the belief drift chart. Interaction data already exists; the network visualization is the missing analytical lens that social science researchers expect. Hub agents, bridge agents, echo chamber scores.
- **Multi-Document Mode** (#3) — The controlled experiment primitive. Director Mode perturbs a single run; comparison mode compares different runs on the same doc; Multi-Document Mode runs the same agents on different documents. The "bull vs. bear," "before vs. after" research pattern.
- **Embeddable Widget** (#4) — Distribution lever. Turns every share permalink into an inline embeddable chart. Researchers embed in Notion/Substack; the MiroShark attribution link compounds with every embed. oEmbed autodiscovery enables paste-to-embed in major platforms.
- **Quality Diagnostics** (#5) — The feedback loop that makes users better at configuring simulations. Flags degenerate runs (low participation, instant consensus, siloed platforms) and suggests corrective parameter changes. Closes the gap between "completed" and "meaningful."

Each idea is scoped for autonomous implementation by the `feature` skill — clear inputs/outputs, no ambiguous design decisions, no external approvals needed.
