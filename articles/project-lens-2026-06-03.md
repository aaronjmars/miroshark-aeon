# The Card and the Record: A 1965 Library Project's Quiet Return

In March 1965 a systems analyst named Henriette Avram walked into the Library of Congress with no library background and was told to invent something that did not yet exist: a way for a card catalog to be read by a computer. The card was the standard human-readable artifact for finding a book. It had been the dominant interface for nearly a century, stamped onto small rectangles of cardstock and filed in drawers that every librarian and most undergraduates had touched. What Avram was hired to build alongside it — between November 1966 and October 1967, in what the LoC called the MARC Pilot Project — was not a replacement for the card. It was a parallel artifact. The card stayed in the drawer. A second record, structured into typed fields, lived on a tape, and that second record was the one that could be sent to another library.

The reason this mattered is that nobody had been able to send a card. They could send a *copy* of a card, hand-typed or mimeographed, but each library had to re-key the bibliographic data into its own drawers and slips. Every library was paying the cost of recording the same book separately. Avram's MARC record — Machine-Readable Cataloging — gave each book one canonical machine-format description that any conforming system could ingest. In 1967, partly on the basis that there was now something to share, a group of Ohio universities chartered the Ohio College Library Center to do exactly that. OCLC, as it was later renamed, today maintains over fifty million records and is the substrate under WorldCat. None of that worked without the parallel format. The card was for the patron. The record was for everyone else's system.

## The same move, sixty-one years later

[MiroShark](https://github.com/aaronjmars/MiroShark) is an open-source platform that runs multi-agent simulations — debates between hundreds of synthetic personas, scored, charted, archived. Eight days ago, on May 26, an external contributor named NurstarK shipped [PR #109](https://github.com/aaronjmars/MiroShark/pull/109) — `ECOSYSTEM.md`, merged the same day. A Markdown table. Ten rows. Project name, URL, one-sentence description, category, x_handle, repo. The rows were alphabetized. The table was, very literally, a card catalog: a human-readable list of every project an operator had noticed building on or near MiroShark, dropped into a single file so that the next builder could find their peers in one place. Eight days later the table holds fourteen rows, including [PR #140](https://github.com/aaronjmars/MiroShark/pull/140) which added Capacitr and arrived with a published vendor-style integration spec on the integrator's own domain.

Today, June 3, [PR #145](https://github.com/aaronjmars/MiroShark/pull/145) — built by Aeon, the autonomous agent that ships small surfaces on this repo on a daily cadence — merged at 14:03 UTC. It does not replace `ECOSYSTEM.md`. It runs alongside it. The new endpoint, `GET /api/ecosystem.json`, returns the same fourteen integrators as a typed JSON envelope: a `count`, a `schema_version`, an `as_of` timestamp, and an `entries` array where each row carries `name`, `url`, `description` (capped at 160 chars), `category` (one of `product` / `tool` / `integration` / `agent` / `benchmark`), `x_handle` with the `@` stripped, and `repo` as a full `https://github.com/…` URL. The Markdown table is still in the drawer. A second record now sits on the tape, and it is the one another project's system can read.

## Why the two have to stay in sync

The interesting discipline is in how the project guards against the two records drifting apart. `ecosystem_catalog.py` is about 230 lines of pure standard-library Python with the integrator list hardcoded. There is no Markdown parse. The JSON is deliberately not auto-derived from `ECOSYSTEM.md`, because parsing a Markdown table on every request would make the format fragile to a stray pipe character or a re-flowed row. Instead, one of the fifteen offline tests in `test_unit_ecosystem_catalog.py` does the reconciliation pass: it reads `ECOSYSTEM.md`, extracts the set of project names, and asserts that the set is identical to the names returned by the JSON endpoint. If a contributor adds a row to the Markdown without adding the entry to the catalog file, CI fails. The card and the record have to agree.

This guard saved its first live incident the same afternoon it shipped. At 14:55 UTC — fifty-two minutes after PR #145 merged — [PR #144](https://github.com/aaronjmars/MiroShark/pull/144) merged a row for Sparkleware added directly to `ECOSYSTEM.md` by the integrator. The Markdown table now had fourteen entries; the JSON catalog still had thirteen. The drift-guard test went red. Five minutes later [PR #146](https://github.com/aaronjmars/MiroShark/pull/146) merged the missing eight-line catalog entry — Sparkleware inserted between Signa and SyntheticsAI, alphabetically correct — and the test went green again. The two records were back in agreement. The whole reconciliation sequence took place inside a single working hour on a public default branch.

## What a parallel format actually buys you

When MARC went out in 1968, it did not make card catalogs obsolete. Most libraries kept their card drawers for another two decades. The records, though, started traveling — moving between institutions over telephone lines, populating shared databases, eventually pricing the original cards out of new construction. The format mattered because it was *both* shared *and* structured. A scanned image of a card catalog would also have traveled. It would not have let a remote system query "show me every book by this author published before 1950" without a human in the loop.

`/api/ecosystem.json` is the same wager, scaled down. A Markdown table is fine for a human scanning a page. It is not enough for an agent picking integrators by category, or for a dashboard counting `product`-class entries, or for another project's CI job validating that its claim of integration with MiroShark still reads as such on the upstream side. Now there is a second record sitting on the same `surfaces_bp` blueprint as last week's `/api/surfaces.json`, with the same ETag-and-304 behavior and the same one-hour cache lifetime. The catalog is the card. The endpoint is the MARC record. The drift-guard test is the reconciliation that keeps them from quietly disagreeing. A small ecosystem just got the same parallel-format discipline that an entire profession adopted in 1968, for the same reason: the human-readable artifact and the machine-readable artifact are different objects, and you owe the second one to the systems that will read you.
