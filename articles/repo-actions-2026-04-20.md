## Ecosystem Context

MiroShark shipped its deepest single-day merge wave: four PRs in under 10 minutes, bringing the total analytics surface to eleven overlays (Belief Drift, Quality Diagnostics, Interaction Network, Demographic Breakdown, Counterfactual Explorer, Director Mode, Trace Interview, Scenario Auto-Suggest, Embeddable Widget, Article Generator, Prediction Resolution). The clean PR queue is an invitation to look at the second-order gaps — what the tool still can't do for users who are already inside a simulation, working collaboratively, or trying to share what they found.

Three signal threads driving this batch:

- **The temporal dimension is locked.** Every analytical panel (network graph, demographics, quality diagnostics) shows the *final* state of a simulation. But the most interesting research moment is usually not the end — it's the round where consensus formed, or the round where a hub agent changed the trajectory. The belief drift chart shows the *aggregate* arc; there's no way to "stop" the simulation at round 12 and see which agents held which stances, what the network looked like, or how the demographic split had evolved at that moment. All the data is in `trajectory.json`. The temporal dimension just hasn't been exposed in the UI yet.

- **Sharing is a URL, not a story.** MiroShark's share permalink works, but when users post it on Twitter/X or Telegram, the link preview is a generic GitHub-style card. Researchers and token-community members who want to share a striking simulation result — "The Director Mode scenario I ran Friday literally predicted the Kelp/Aave cascade" — have no visual story to attach. A properly rendered share card (scenario headline, consensus gauge, quality badge) turns a URL into a post that gets clicked. With the Kelp/Aave coincidence driving organic attention today, this gap is costing reach right now.

- **The first simulation is still blank canvas.** Scenario Auto-Suggest (PR #39) solved the "I have a document, what should I ask?" problem. But a meaningful share of users come to MiroShark without a specific document — they know they want to simulate *something* about AI regulation or crypto markets but don't have an article to paste. A "What's Trending" panel that surfaces 5 current headlines from a configurable RSS feed gives these users a starting point in one click, while also connecting MiroShark to the real-time news cycle that makes its prediction-market framing most relevant.

### 1. Round Scrubber / Temporal Explorer

**Type:** Feature
**Effort:** Small (hours)
**Impact:** Every analytical overlay in MiroShark (network graph, demographic breakdown, quality diagnostics) renders the *final* state of the simulation. But the most publishable finding is often a single turning-point round: the moment a hub agent shifted, the round where consensus crystallized, the injection point in a Director Mode run. A Round Scrubber adds an interactive time-slider below the belief drift chart. Dragging it to any round syncs all open panels — the network graph redraws with that round's interaction edges, the demographic panel shows round-N stance distributions, the agent table highlights each agent's stance at that moment. The full temporal arc becomes explorable: researchers can pause at round 12, see who was talking to whom, then scrub to round 20 to see how the hub's influence propagated. For Director Mode simulations, this makes the event injection timeline directly navigable. All the data already lives in `trajectory.json`; this is a UI-layer feature with no backend changes required.
**How:**
1. Add a `currentRound` reactive ref to the simulation results page (default: final round). Render a range input (`<input type="range" min="1" :max="totalRounds" v-model="currentRound">`) below the belief drift chart, styled as a scrubber with round labels and a "Live" badge at the final position. Subscribe to scrubber changes with a debounce (50ms) to avoid janky redraws on fast drags. Add keyboard arrow-key navigation (left/right) when the scrubber is focused. Display the current round prominently as "Round N / M" next to the scrubber.
2. Filter `trajectory.json` data by `currentRound` and pass the filtered slice as a prop to each panel. For the network graph: use only interaction edges where `round <= currentRound` (cumulative) and agent stances from the *latest* round in that slice. For the demographic breakdown: recompute stance buckets using each agent's stance at `currentRound` (read from `trajectory[currentRound][agent_id].stance`). For the quality diagnostics: recompute participation rate, entropy, and convergence up to `currentRound` (mark the health badge as "In Progress" vs "Final" so users know it's a snapshot). For the belief drift chart itself: highlight the scrubber position with a vertical rule and bold data point.
3. Add a "Snapshot" button at each scrubber position that exports the current panel state as a PNG (using the same export mechanism as the belief drift and network graph export buttons). Label the exported file `miroshark-{sim_id[:8]}-round{N}.png`. When Director Mode events exist, render event markers on the scrubber track (colored diamonds at each injected round) so users can jump directly to injection points. Include the round number in the share URL as a `?round=N` query param so that sharing a link preserves the temporal context — opening the share link lands at round N with all panels synced.

### 2. Social Share Card Generator

**Type:** Growth
**Effort:** Small (hours)
**Impact:** MiroShark's share permalink exists, but when researchers or $MIROSHARK community members post the URL on Twitter/X, Telegram, or LinkedIn, the link unfurls as a generic page preview — no visual story, no hook. Today's Kelp/Aave news cycle is driving organic mentions of MiroShark's Director Mode coincidence, but every share is text-only. A "Share Card" button on simulation results generates a polished 1200×630px PNG suitable for Twitter/X preview sizing: the simulation scenario headline (truncated to ~120 chars), a half-circle consensus gauge (YES % with colored fill: teal for high consensus, amber for divided, coral for inverted), a quality badge (Excellent / Good / Low), the top 3 agents with name, stance icon, and influence rank, and a MiroShark footer strip with the project URL. The card is generated server-side (Pillow or canvas-to-PNG) and served at `GET /api/simulation/:id/share-card.png`. A corresponding `<meta property="og:image">` tag on the share permalink URL auto-embeds it for link unfurling. For the 1K-star target, every researcher who shares a simulation result becomes a distribution channel — a compelling visual card multiplies click-through.
**How:**
1. Add `GET /api/simulation/:id/share-card.png` endpoint. Load the simulation record and pull: `scenario` (title field), `final_yes_price` (0–100), `quality_health` from `quality.json`, top 3 agents from the influence leaderboard. Generate the card using Pillow: 1200×630px white canvas, dark MiroShark header band (#0f172a), scenario text in large bold sans-serif, consensus gauge drawn as a filled arc (using `ImageDraw.arc`), quality badge as a rounded-rectangle label, three agent rows with a colored stance circle (teal=bullish, coral=bearish, slate=neutral), footer with `github.com/aaronjmars/MiroShark` in muted text. Cache the generated PNG in `.share-cards/{sim_id}.png` and invalidate on simulation update. Add `Cache-Control: public, max-age=3600` to the response header.
2. Update the simulation share permalink route to inject `<meta property="og:image" content="/api/simulation/{sim_id}/share-card.png">` and `<meta name="twitter:card" content="summary_large_image">` in the HTML `<head>` when `?share=1` is present (or always for share-target routes). This makes all existing share links auto-unfurl with the card on Twitter/X, Discord, and any platform that reads Open Graph tags. No frontend routing changes required beyond adding the meta tags to the server-rendered share view.
3. Add a "Copy Share Card" button to the simulation results header (alongside Share and Fork). Clicking it calls `/api/simulation/:id/share-card.png`, triggers a browser download as `miroshark-{scenario-slug}.png`, and copies the share URL to clipboard simultaneously (with a "Copied + image downloaded" toast). Add a "Preview Card" modal that shows the card image before downloading. Log `share_card_generated: true` on the simulation record for analytics. Surface the share card feature in the README under a new "Sharing" section with an example card image — demonstrates the visual output to potential users evaluating the tool.

### 3. Trending Topics Auto-Discovery

**Type:** Feature / Integration
**Effort:** Small (hours)
**Impact:** Scenario Auto-Suggest (PR #39) eliminated the blank-page problem for users who already have a document. But a substantial share of MiroShark's audience arrives without a specific article in mind — they want to simulate something about AI, crypto, or geopolitics but don't know where to start. A "What's Trending" panel on the simulation setup home page fetches 5 current headlines from a configurable RSS/Atom feed (defaulting to a curated list: Reuters Tech, CoinDesk, AI news) and presents them as clickable topic cards. Each card shows the headline, source, and published time. Clicking one pre-fills the document URL field with the article link and immediately fires the Scenario Auto-Suggest pipeline — the user goes from "blank page" to "three scenario options" in one click. For $MIROSHARK holders and the broader crypto community, this connects MiroShark directly to the breaking news cycle that prediction markets are built around. The Kelp/Aave exploit going viral today — while MiroShark's Director Mode demo had run that exact scenario 24 hours earlier — is the perfect illustration of what this feature unlocks: real-time current-events simulation.
**How:**
1. Add `GET /api/trending` endpoint. Accepts optional `?feeds=url1,url2` query param; falls back to a hardcoded default feed list in config (`TRENDING_FEEDS` env var, comma-separated RSS/Atom URLs). Parse feeds using `feedparser` (Python). Return the 5 most recent items across all feeds: `[{title, url, source_name, published_at}]`. Cache the response in memory for 15 minutes (keyed by feed list hash) to avoid re-fetching on every page load. If a feed fails to parse, skip it silently and fill from the remaining feeds. No authentication required — purely public RSS.
2. Add a `TrendingTopics.vue` component to the simulation setup `Home.vue`. On mount, call `/api/trending` and render 5 cards in a horizontal scroll row (mobile-friendly) or a 2-column grid (desktop). Each card shows: headline (2 lines max, truncated with ellipsis), source name in muted text, time-ago label ("3h ago"). A "Simulate →" button on each card fires two actions: (a) sets `formData.documentUrls[0] = item.url` and (b) triggers the Scenario Auto-Suggest pipeline immediately (same debounced call as when a URL is manually pasted). Show a "Refreshed 15 min ago" label and a manual refresh icon. If the API returns empty (all feeds failed), hide the component silently.
3. Add a `TRENDING_FEEDS` env var to `.env.example` with 3 default public RSS feeds (Reuters tech feed, CoinTelegraph, The Verge AI tag). Add a `Configure Feeds` link in the component footer that opens a settings modal where the operator can add/remove/reorder feeds (persisted to a `trending_feeds` config record in the DB). Include feed configuration in the README under a "Trending Topics" section. Add a `?topic=headline-slug` query param support to the home route so that external links can deep-link to MiroShark pre-loaded with a specific trending topic — useful for social posts ("Simulate this →").

### 4. Collaborative Review Comments

**Type:** Community
**Effort:** Small (hours)
**Impact:** Research teams using MiroShark to evaluate scenarios don't work alone — one person runs the simulation and then five colleagues need to weigh in on the results. Currently their only option is to copy the share URL and start a Slack thread, losing the simulation context entirely. A threaded comment system built directly into the simulation result page lets team members leave contextual notes: "This agent's behavior in round 8 seems anomalous — check the injection event," "This consensus level is suspiciously high given the agent population," "Great starting point but the bull framing is too aggressive." Comments are stored on the simulation record (no separate user accounts required — just a display name on submit). Optional: comment markers appear as vertical annotations on the belief drift chart (a small comment icon at the round the user was viewing when they left the note, if the Round Scrubber is in use). For research groups running MiroShark as a shared analysis tool, this replaces "screenshot + Slack thread" with a persistent, contextualized record of the team's analysis.
**How:**
1. Add a `comments` table: `id`, `simulation_id` (FK), `display_name` (max 50 chars), `body` (max 1000 chars), `round_context` (nullable int — the scrubber round when the comment was left), `created_at`. Add `POST /api/simulation/:id/comments` (body: `{display_name, body, round_context?}`) and `GET /api/simulation/:id/comments` (returns paginated array, newest-first, max 100). No authentication — display name is self-asserted. Add basic rate limiting: max 10 comments per IP per simulation per hour. Sanitize `body` on input (strip HTML, enforce max length). No moderation layer for now — trust the research audience.
2. Add a `SimulationComments.vue` component rendered below the existing analytical panels on the simulation result page. Show a compact "Leave a note" form: a display name field (persisted to `localStorage` so it pre-fills on return visits), a body textarea (with live character count), and a "Comment" submit button. Render existing comments in a simple thread: avatar initial circle (generated from display name), name, time-ago, body text, round context badge ("Left at Round 12" — clickable to jump the scrubber to that round if it's visible). Empty state: "No comments yet. Be the first to leave a note."
3. If the Round Scrubber (idea #1) is present and active: capture `currentRound` at the moment the comment textarea is focused and set it as the default `round_context` for that comment submission. Render comment-count markers on the scrubber track at rounds that have associated comments (a small speech-bubble icon). On the belief drift chart, show comment markers as vertical dashed lines at `round_context` positions (same layering as Director Mode event markers). Include comment count in the simulation history card: "3 comments" badge. Add the comment thread to the share URL (comments load automatically when the share link is opened), making collaborative annotation portable.

### 5. Simulation Config Export / Import

**Type:** DX / Research
**Effort:** Small (hours)
**Impact:** Researchers who publish MiroShark simulation results in papers or reports need a way to share the exact reproduction config — not just the output, but the setup that generated it. A "Export Config" button on the simulation setup form (and on completed simulation history cards) downloads a `miroshark-config.json` file capturing the full simulation parameters: agent count, round count, LLM provider and model, document URLs and text preview (first 500 chars), scenario text, platform distribution weights, any Director Mode events (if pre-configured). An "Import Config" button on the home page accepts a `miroshark-config.json` drop-and-fills all fields instantly. Power users can maintain a personal library of proven setups ("my EU AI Act config", "my DeFi stress test config") and reload them in one drag-and-drop. Research reproducibility improves because the exact simulation parameters are portable — a journal reviewer can load the config and reproduce the run. As MiroShark's community grows (143 forks, 4 contributors), community-shared config packs become a natural next ecosystem artifact.
**How:**
1. Define the `miroshark-config.json` schema (version-tagged: `{"schema": "miroshark/config/v1", ...}`): top-level keys `simulation` (agent_count, round_count, topic, scenario_text, document_urls[], document_preview[]), `model` (provider: "openrouter"|"ollama"|..., model_id, temperature), `agents` (platform_weights: {twitter: float, reddit: float, ...}, agent_personas: [] if custom, seed_agent_count), `director` (events: [{round, description, sentiment_delta}] if any are pre-set), `meta` (exported_at, miroshark_version, simulation_id if from a completed run). Add `GET /api/simulation/:id/config` that serializes a completed simulation's params into this schema (excluding trajectory data — output only, not input params). Add `POST /api/config/validate` that parses and validates an uploaded config JSON, returning a structured object ready to hydrate the form.
2. Add an "Export Config" button to: (a) the simulation setup form header ("Save this config"), (b) the simulation history card dropdown menu ("Export Config"). Clicking downloads `miroshark-config-{sim_id[:8]}-{date}.json` via `URL.createObjectURL(new Blob([JSON.stringify(config, null, 2)], {type: 'application/json'}))`. Add an "Import Config" drop zone on the `Home.vue` setup form — a dashed-border area ("Drop a config file or click to browse") that accepts `.json` files via file input or drag-and-drop. On file selection, call `/api/config/validate`, receive the parsed config, and hydrate all form fields (agent count slider, round count, scenario text, document URLs, model picker) in a single reactive update. Show a "Config loaded from miroshark-config-abc123.json" banner with an "X" to clear.
3. Add a `templates/` directory to the MiroShark repo with 3 sample config files: `eu-ai-act-bull-bear.json`, `defi-stress-test.json`, `us-election-opinion.json`. Each is a fully filled-out config that produces a meaningful simulation. Link to them from the README under a new "Simulation Templates" section ("Start with a proven setup — drop any of these into the Import Config zone"). Add a `CONTRIBUTING.md` section inviting community members to submit their own configs as PRs to `templates/`. This creates a community curation layer without requiring MiroShark to host a backend gallery.

## Selection Rationale

Today's batch targets the gaps that appear *after* users are already running simulations — temporal exploration, sharing, discovery, collaboration, and reproducibility:

- **Round Scrubber** (#1) — All trajectory data exists; the temporal dimension just isn't exposed. This is the highest-leverage UX addition available right now — it turns every existing simulation into a richer research artifact without any backend changes.
- **Social Share Card** (#2) — The Kelp/Aave coincidence is driving organic attention today. A visual share card is a direct growth lever for the 1K-star target: every shared result becomes a distribution channel.
- **Trending Topics** (#3) — Closes the remaining onboarding gap. Scenario Auto-Suggest (PR #39) handled document-paste users; Trending Topics handles users who arrive without a document.
- **Collaborative Comments** (#4) — MiroShark is increasingly used by research teams (143 forks, academic signal). The collaboration gap is real and the implementation is bounded — no auth, no moderation, just a comment thread attached to a simulation.
- **Config Export/Import** (#5) — Reproducibility is table-stakes for research credibility. At 143 forks, community-shared configs are the natural next artifact layer. The `templates/` directory is also a low-effort SEO and discovery play.

Each idea is scoped for autonomous implementation by the `feature` skill — clear inputs/outputs, all data already available, no ambiguous design decisions, no external approvals needed.
