*Feature Built — 2026-05-01*

Belief Trajectory CSV / JSONL Export
MiroShark simulations now have a one-click export of the per-round belief trajectory as RFC 4180 CSV or newline-delimited JSON. An analyst can drop the `/api/simulation/<id>/trajectory.csv` URL into `pandas.read_csv()`, Excel "Get Data → From Web", Tableau, R `read.csv()`, or Observable `d3.csv()` and get a typed DataFrame back without an intermediate conversion step. JSONL is the same shape one object per line for DuckDB, Pandas `read_json(lines=True)`, and stream-processing pipelines.

Why this matters:
The four previous share surfaces (share card preview, replay GIF motion, transcript Markdown prose, transcript JSON for SDKs) all cover the *qualitative* read of a simulation. None of them give the quant audience the *numbers* their default toolkit consumes. Pandas and Excel and R are how researchers actually compare two simulation runs, compute variance across replicates, or build comparison plots — and until today the only path to those tools was a manual JSON-to-DataFrame pivot the analyst had to write themselves. Trajectory CSV / JSONL is the surface that turns a MiroShark URL into a 10-second `pd.read_csv()` quickstart, which is the single biggest evaluation moment for a quant evaluating MiroShark as a research data source.

What was built:
- backend/app/services/trajectory_export.py: New pure-stdlib renderer (csv + io + json, zero new deps). `compute_stance_split` helper with strict-inequality bucketing at the ±0.2 boundary so an agent at exactly 0.2 stays neutral. `build_rows` reads `trajectory.json` + `quality.json`, defensively re-sorts snapshots by round so out-of-order on-disk writes still emit chronologically, and degrades gracefully on missing or corrupt files (the route can never 500). `participating_agents` is computed from distinct `viral_posts` user_ids and falls back to `active_agent_count` for quiet rounds.
- backend/app/api/simulation.py: `_serve_trajectory()` shared body and two route decorators (`/<id>/trajectory.csv` + `.jsonl`), mirroring the `_serve_transcript` pattern shipped in PR #57. Same publish gate, same 60s cache, same disk reads.
- backend/openapi.yaml + new SimulationTrajectoryRow schema: both paths land under the Publish & Embed tag, drift-detection test passes against the new routes.
- frontend/src/components/EmbedDialog.vue: "📊 Export trajectory data" section beneath the transcript row with Download .csv / .jsonl buttons (publish-gated), copyable CSV URL, and a `pd.read_csv("<url>")` quickstart snippet styled as a one-line code block.
- backend/tests/test_unit_trajectory.py: 17 offline unit tests covering ±0.2 stance threshold parity, boundary bucketing, empty/corrupt/unparseable input degradation, chronological reordering, missing-quality.json fallback, participating-agents fallback, locked CSV column order, csv.DictReader round-trip, QUOTE_MINIMAL preserves dtype inference, JSONL field-key order matches CSV_COLUMNS, empty-input zero-byte JSONL contract, route-decorator presence guard.
- README + docs/FEATURES.md + docs/API.md: Features row, dedicated section, and an Analyst quickstart with pandas.read_csv() + DuckDB read_json_auto() snippets.

How it works:
The CSV writer uses `csv.DictWriter` with `QUOTE_MINIMAL` so numeric columns stay unquoted in the output — pandas dtype inference then types `bullish_pct` as float64 directly without a `converters=` argument, which is exactly what the quickstart snippet expects. The JSONL writer filters output through the canonical `CSV_COLUMNS` field tuple so an upstream change to the row builder can't accidentally leak unstable keys into the JSONL stream and break downstream pipelines that key on field names. The same ±0.2 stance threshold runs through every other surface (gallery, share card, replay GIF, transcript, webhook, RSS / Atom feeds) so a `bullish_pct` value in the CSV matches what the gallery card and share card display for the same round — six surfaces / one threshold / one folder. Header row is emitted on empty trajectories so consumers don't have to special-case zero-row files; JSONL is zero bytes on empty input because well-formed JSONL has no header concept and a stray newline would parse as an empty object somewhere downstream.
