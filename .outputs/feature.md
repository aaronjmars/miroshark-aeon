*Feature Built — 2026-06-08*

Signed Simulation Result
MiroShark simulations now ship with a cryptographic proof of authenticity. The new `/api/simulation/<id>/signed-result.json` endpoint returns the same trading-signal payload as `signal.json` (direction, confidence, risk tier, belief percentages) but wrapped in an HMAC-SHA256-signed envelope. An integrator who stores the result locally — Capacitr's settlement ledger, a research archive, an ML pipeline's provenance row — can later prove the stored bytes match what MiroShark actually emitted, without ever calling back to the API.

Why this matters:
HTTPS guarantees in-transit authenticity, but once a signal lives in someone else's database there is no way today to prove it wasn't tampered with — short of re-fetching the sim live, which defeats the point of having stored it. This is the first endpoint that gives MiroShark a true offline-verifiability primitive, and it does it without introducing a new secret: the signing key is the existing WEBHOOK_SECRET that already powers outbound webhook delivery. Integrators who already verify webhook signatures (Zapier, Capacitr, AntFleet, custom Cloud Run listeners) reuse the exact same `hmac.compare_digest` / `crypto.timingSafeEqual` primitive against the signed payload — zero new config, zero new key-distribution problem.

What was built:
- backend/app/services/signed_result.py — Pure-stdlib service (~180 LoC, no new deps). `build_signed_result()` returns the signed envelope; `canonical_json()` exposes the deterministic JSON encoder (`sort_keys=True, separators=(",",":"), ensure_ascii=True`) so a recipient on any stdlib produces byte-identical signing material; `compute_signature()` is the bare HMAC primitive. Missing/empty/whitespace WEBHOOK_SECRET returns `signed=false` + human-readable error — never raises, never 500s.
- backend/app/api/simulation.py — New route handler `get_signed_result_json` on `simulation_bp`. Inherits signal.json's publish gate (403 private / 404 not-ready) with byte-identical envelopes so an anonymous caller can't probe for private state. Cached 5 minutes. Auth-posture comment in the docstring documents the explicit private-inherits-signal.json decision.
- backend/app/services/surface_stats.py + surfaces_catalog.py — `signed_result` registered as the 34th catalogued surface (integration type). `_PER_SIM_TRACKED_KEYS` extended in lock-step; the drift test enforces sync.
- backend/openapi.yaml — Path entry + `SignedSimulationResult` schema with Python and Node.js verification recipes inline in the description.
- backend/tests/test_unit_signed_result.py — 25 offline unit tests: envelope shape, hmac.compare_digest verification, wrong-secret rejection, sort_keys-stable canonical JSON, ASCII unicode-escape encoding (`"café"` → `café`), determinism, signature tampering invalidation, sim-id binding, unsigned-path safety, signing_key_hint shape, surface_stats + catalog drift guards, openapi drift, ISO-8601 timestamps, 64-char lower-hex signature format, auth-posture comment presence.

How it works:
The signature is HMAC-SHA256 over the canonical JSON encoding of the inner `result` block — not the outer envelope. `json.dumps(result, sort_keys=True, separators=(",", ":"), ensure_ascii=True)` produces a byte string a recipient can reconstruct exactly from their parsed copy of `result` regardless of stdlib version, OS, or JSON library, then `hmac.new(WEBHOOK_SECRET, canonical, sha256).hexdigest()` is compared against `envelope.signature` via `hmac.compare_digest`. The envelope's `signed_at` is wall-clock and deliberately NOT signed, so two consecutive requests for the same finished sim emit the same `signature` but different `signed_at` — letting recipients hash-pin a signature to a sim's outcome without timestamp drift. The simulation_id is part of the signed material, so a forwarded signature for sim_A cannot validate against sim_B. Deployments without WEBHOOK_SECRET configured still return 200 with `signed=false` + the unsigned result — the absence of a signature is the missing feature, not an API failure.
