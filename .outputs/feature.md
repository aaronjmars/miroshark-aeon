*Feature Built — 2026-06-09*

Simulation Activity Feed
MiroShark now has a lightweight "what just completed?" polling endpoint. `GET /api/activity.json` returns the 20 most recently completed public simulations in reverse-chronological order, each with the small payload a polling consumer needs (scenario title, direction, confidence, quality, total rounds, completion timestamp, project id). Public-without-auth so integrators can wire it without a key.

Why this matters:
The catalog had three discovery surfaces but a hole between them — `/api/simulation/public` is the heavy filterable gallery for the UI, `/api/feed.rss` + `/api/feed.atom` are XML subscription feeds for RSS readers, `/api/status.json` is an aggregate liveness probe with no per-sim payload. None served the "poll every 30 seconds, render a recent-runs strip, react to completions in code" use case integrators (Capacitr, AntFleet, social bots, status dashboards) actually need. Aeon's push-recap skill needed it too — "did anything new complete since the last run?" was previously a full-gallery fetch. Picked from the Jun-08 repo-actions batch as the highest-impact unbuilt idea and the second discovery primitive after the RSS / Atom feeds.

What was built:
- backend/app/services/activity_feed.py: New ~330 LoC pure-data service. `build_activity_feed(sim_root, *, limit)` walks the corpus, filters to is_public + completed, derives direction / confidence / quality from the same `signal_service.compute_signal` pipeline `signal.json` and `batch_status` use (so analytics match byte-for-byte). `clamp_limit(value)` accepts loose inputs — non-numeric → 20, out-of-range → clamped into [1, 50], booleans rejected. `feed_etag(payload)` derives a short `"activity-{count}-{newest_completed_at}"` tag so an `If-None-Match` GET short-circuits to 304.
- backend/app/api/activity.py: New ~145 LoC route handler on `activity_bp`. One endpoint, `Cache-Control: public, max-age=30` (matches `/api/status.json` cadence), ETag handling with 304 short-circuit. Leads with the `Auth posture: public — built for keyless integrator polling.` line per the PR #149 / aeon PR #53 convention.
- backend/app/__init__.py: Registers `activity_bp` at `/api`, adds `/api/activity.json` to the `internal_auth_guard` allow-list alongside `/api/status.json` and `/api/simulation/batch-status`, with a comment explaining the public-by-design rationale.
- backend/app/services/surfaces_catalog.py: New `activity_feed` entry (type=`discovery`). Catalog 34 → 35. Discovery category now has 3 entries (feed_atom, feed_rss, activity_feed) — was the joint-most-underserved category alongside embed.
- backend/openapi.yaml: Full `/api/activity.json` path entry under the Platform tag + new `ActivityFeed` + `ActivityFeedEntry` schemas with locked field set.
- backend/tests/test_unit_activity_feed.py: 39 offline unit tests across 18 properties — publish gate, ordering (descending by completed_at, ties → sim_id desc for determinism), limit clamping, scenario truncation, signal-pipeline equivalence, total_rounds matches BatchStatusEntry byte-for-byte, degraded-trajectory tolerance, updated_at → created_at fallback, corrupt-folder skip, ETag format + change detection, JSON-serialisability, catalog discoverability, OpenAPI spec coverage, blueprint registration, auth-guard allow-list, route file drift guards.
- frontend/src/api/simulation.js: `getActivityFeedUrl(limit?, origin?)` + `getActivityFeed(limit?)` helpers (same shape as the existing `getPlatformStatus` / `batchSimulationStatus` stubs).
- docs/API.md: Platform-section table row with deltas vs the four sibling surfaces.

How it works:
The service walks `WONDERWALL_SIMULATION_DATA_DIR`, applies the standard publish gate (`is_public AND status == "completed"`), reads each sim's `simulation_config.json` for the scenario title (truncated to 100 chars with trailing ellipsis), and derives signal fields by running the same final-round belief split through `signal_service.compute_signal` that the per-sim `signal.json` surface uses — so a sim's activity-feed entry matches its per-sim signal byte-for-byte. `total_rounds` mirrors the `batch_status._final_belief_from_trajectory` algorithm exactly (count rounds where stance extraction succeeded), so `ActivityFeedEntry.total_rounds == BatchStatusEntry.total_rounds` for the same sim. Auth posture was decided UPFRONT per the PR #149 lesson — three questions answered, exempt picked, allow-list and OpenAPI updated in the same commit rather than across a mid-PR rewrite. No in-process cache by design — the surface is meant to be live, and an in-process cache would make "recent completions" stale.

What's next:
Aeon's push-recap skill can wire the endpoint directly to check for new completions between runs without re-reading the full gallery. The Jun-08 batch has four more unbuilt ideas (Trending Topics, MCP Tool Catalog JSON, Pre-Run Cost Estimator, Chinese README — the last one advances the Chinese-locale contributor hyperstition with a Jun-15 deadline). A natural sibling endpoint would be a per-project activity feed (`GET /api/project/<id>/activity.json`) mirroring the same shape, useful for operators running multiple projects who want a per-workspace "recent runs" strip.
