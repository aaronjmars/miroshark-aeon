*Feature Built — 2026-04-28*

Animated Belief Replay (GIF)
MiroShark simulations now have an animated belief-replay GIF — same 1200×630 canvas as the existing share card, but instead of a static snapshot it plays the per-round belief drift one frame at a time with bullish / neutral / bearish bars sliding to each round's distribution, a round counter, and a progress bar at the bottom. Drop the URL into a Discord or Slack channel and it auto-plays inline; download it from the Embed dialog and post it anywhere a GIF works.

Why this matters:
The share card is static — every published simulation produced one frozen image. Discord and Slack auto-play GIFs from a direct file URL, so a per-round animation lands as motion in places where a screenshot can't compete. The 1.91:1 OG aspect matches the existing card so unfurl shapes stay consistent across the static + animated formats. With the 1K-stars-by-Apr-30 sprint at 853 stars and 2 days left, every share now ships motion alongside the still — the same audience surface, twice the asset.

What was built:
- backend/app/services/replay_gif.py: New 405-line Pillow renderer. Builds one frame per round on a dark canvas (bullish teal / neutral slate / bearish coral, MiroShark orange accent), composes the same font-discovery + text-wrap helpers the share card uses, encodes the GIF with `loop=0` infinite repeat and `disposal=2` so shrinking bars don't ghost between rounds. Final frame holds 3× longer (1800 ms vs 600 ms) so the resting consensus reads as the punch-line. `MAX_FRAMES=60` cap with even subsampling that always preserves the final round — a 200-round sim compresses to ≤60 frames so playback stays under ~36 s.
- backend/app/api/simulation.py: New `GET /<sim_id>/replay.gif` endpoint right next to the existing share-card route. Reuses `_build_embed_summary_payload()` for the data pipeline (so trajectory parsing stays single-sourced), enforces the same `is_public` publish gate (private sims get 403), and caches generated GIFs on disk at `<sim_dir>/replay-gifs/<hash>.gif` with `Cache-Control: public, max-age=3600`.
- frontend/src/components/EmbedDialog.vue: New "Belief replay (animated)" section under the share-card row. Dark themed to differentiate from the static-card section above, with a paused thumbnail + tap-to-play overlay so opening the dialog doesn't pull GIF bytes for every viewer (the `<img>` mounts only on click). Copyable replay URL + Download GIF button match the existing share-card actions.
- backend/tests/test_unit_replay_gif.py: 14 offline unit tests — GIF signature + dimension via raw byte parse, frame count via `PIL.ImageSequence`, `MAX_FRAMES` subsampling preserves the final round, empty-trajectory poster fallback, cache-key stability under float jitter, route-decorator presence as a guard against silent removal.
- backend/openapi.yaml + README + docs/FEATURES.md: Spec entry under the Publish & Embed tag (drift-detection test will fail if the route is renamed without updating the spec), README Features row, and a dedicated section in FEATURES.md.

How it works:
The endpoint pulls the same per-round belief percentages the share card uses (`belief.rounds`, `belief.bullish/neutral/bearish` from `_build_embed_summary_payload`), then `extract_frames_from_summary()` either passes them through (≤60 rounds) or evenly subsamples down to 60 frames while keeping the last round in place. Each frame is a 1200×630 RGB Pillow image: header band with brand wordmark, scenario headline (auto-wrapped to 2 lines), three labelled stacked horizontal bars with rounded pill ends and percentage values, a thin orange progress bar, and a footer band with the GitHub URL. `Image.save(format="GIF", save_all=True, append_images=frames[1:], duration=[600,...,1800], loop=0, disposal=2)` encodes the lot in a single Pillow call — no FFmpeg, no extra deps. Cache key hashes the trajectory rounded to 1 decimal place so float jitter in `json.load` doesn't bust the disk cache.
