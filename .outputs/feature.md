*Feature Built — 2026-04-26*

Completion Webhook (Slack / Discord / Zapier / n8n / custom)

MiroShark simulations now POST a JSON summary to a URL of your choosing the moment a run finishes (or fails) — one URL field in Settings → Integrations · Webhook wires up Slack channels, Discord channels, Zapier/Make/n8n workflows, IFTTT, or any custom listener. The payload includes the scenario, final bull/neutral/bear consensus split, quality assessment, polymarket resolution outcome, and the public share-card URL so Slack and Discord auto-unfurl with the 1200×630 OG card from PR #42.

Why this matters:
Researchers who run long simulations (50+ agents, 20+ rounds) close the tab and come back later — they miss the moment of completion. Browser push notification reaches one device; the operator's Slack channel reaches the whole team plus their automation stack. This was the #3 idea in the Apr 24 repo-actions batch and is a strict superset of the standalone "Post to Discord/Slack" button (#4) — pointing the same URL at a Zapier/Make/n8n trigger fans out to email, Notion, Airtable, Sheets, custom dashboards, or any of 5,000+ integrations they offer. It's also the production-quality checklist item the Paradigm-adjacent ML/dev audience scans for when evaluating a tool as infrastructure rather than demo — a direct lever on the 1K-stars-by-Apr-30 sprint (currently 829, needs ~43/day with 4 days left).

What was built:
- backend/app/services/webhook_service.py: new service. `build_payload()` reads the same on-disk artifacts as the share card and gallery card (config/quality/trajectory/resolution/state) and applies the same ±0.2 stance threshold so the consensus the webhook reports matches what users see. `fire_webhook_for_simulation()` runs on a daemon thread (caller returns instantly), deduped per (sim_id, status). `send_test_webhook()` for the Settings test button. `validate_url`/`mask_url` accept http(s) only and mask saved URLs to scheme://host/*** so the secret portion of a Slack/Discord URL never echoes through GET /api/settings. Stdlib `urllib.request`, zero new dependencies.
- backend/app/services/simulation_runner.py: hook calls added at both terminal paths — the exit-code path (RunnerStatus.COMPLETED/FAILED) and the per-platform `simulation_end` event path. Both wrapped in try/except so a failing webhook never takes down the runner.
- backend/app/api/settings.py: GET /api/settings exposes `integrations.webhook = {configured, url_masked, public_base_url}`; POST /api/settings accepts the new fields with validation; new POST /api/settings/test-webhook returns {success, message, latency_ms, url_masked}.
- frontend/src/components/SettingsPanel.vue: new "Integrations · Webhook" section above the AI Integration · MCP block — status badge, URL input that pre-renders the masked saved value as the placeholder, optional Public base URL, and a "Send test event" button that surfaces ✓ Delivered (Nms) or the failure reason inline.
- docs/WEBHOOKS.md, README, .env.example: full payload schema, headers, delivery semantics (fire-and-forget, no retries, dedup, 5s timeout, 2xx = success), and Slack/Discord/Zapier/n8n/custom-listener recipes. README Features + Documentation tables both link the new doc.
- backend/tests/test_unit_webhook.py: 18 offline tests covering payload shape (full/minimal/corrupt/long-scenario/failure-with-error), URL validation+masking, fire-no-op when unconfigured, async dispatch (caller returns before slow POST), per-(sim_id,status) dedup including completed+failed both firing, exception swallowing, unknown-status guard, send_test_webhook end-to-end. Mocks `_post_json` so no real HTTP fires.

How it works:
At completion, the simulation runner's monitor thread calls `fire_webhook_for_simulation(sim_id, "completed", state=state)` alongside the existing `send_push_notification`. The webhook service late-reads `Config.WEBHOOK_URL` (so runtime Settings updates take effect immediately), no-ops if it's empty, applies (sim_id, status) dedup, builds the JSON payload from the sim_dir artifacts, and fires the POST on its own daemon thread with a 5-second timeout. The runner is never blocked. When `Config.PUBLIC_BASE_URL` is set, the payload includes absolute share_url + share_card_url so Slack and Discord auto-unfurl with the OG card; without it, only relative paths are sent (consumers can build absolute URLs themselves). All errors are logged + swallowed — the webhook is a notification, not a critical path.
