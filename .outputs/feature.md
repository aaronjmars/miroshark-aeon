*Feature Built — 2026-04-30*

Public Gallery RSS / Atom Feeds
MiroShark's public simulation gallery now has RSS and Atom feeds. Researchers, DeFi analysts, and AI tooling operators who already follow content via Feedly, Readwise, Inoreader, NetNewsWire, or Obsidian RSS can subscribe in their existing reader, and every newly published MiroShark simulation lands in their feed automatically — no curation, no login, no MiroShark account required.

Why this matters:
The /explore gallery (PR #43) is a pull surface — you visit it. The research audience MiroShark is courting reads AI/DeFi content via RSS subscriptions, the same way they follow Anthropic's blog or Decrypt's DeFi feed. Until today, there was no way to "follow MiroShark" without manually checking the gallery. This was repo-actions Apr 28 idea #3, and it converts the pull surface into a push channel — a passive distribution multiplier that compounds with every published simulation. Pairs with the share card (preview, Apr 22), replay GIF (motion, Apr 28), and transcript export (text, Apr 29) as the fourth orthogonal share / discovery surface, all projections of the same on-disk sim folder.

What was built:
- backend/app/services/feed.py: New 584-line stdlib renderer (xml.etree.ElementTree + html — zero new deps). Produces both Atom 1.0 and RSS 2.0 from the same payload. Per-entry XML carries scenario as title (truncated to 100 chars with ellipsis), bullish/neutral/bearish consensus split as the plain summary, share-card PNG as <media:thumbnail> + <media:content> (so River-view aggregators like Feedly surface a preview), animated replay GIF as a second <media:content> (so motion-capable surfaces auto-play), outcome and quality as <category> elements (so subscribers can filter on them in their reader). RSS variant adds <enclosure> for podcatcher-style readers and <guid isPermaLink="false"> for stable identification across deployments.
- backend/app/api/feed.py: New 144-line Flask blueprint mounted at /api with GET /api/feed.atom and GET /api/feed.rss. Both accept ?verified=1 to restrict to simulations with a recorded outcome — the syndication mirror of /verified. Cache-Control: public, max-age=300 (5-minute CDN-friendly cache). Honors Config.PUBLIC_BASE_URL + X-Forwarded-Proto/Host so feeds served behind a reverse proxy point at the canonical public URL. Fail-soft against SimulationManager errors — an empty feed is the right interim state if anything goes wrong.
- backend/openapi.yaml + tests/test_unit_openapi.py: Both endpoints documented under Publish & Embed; feed_bp: /api added to the drift-detection blueprint map. Spec stays in lockstep with code.
- frontend/index.html: <link rel="alternate" type="application/atom+xml"> + RSS variant for browser auto-discovery — readers expose a globe icon in the address bar.
- frontend/src/views/ExploreView.vue: New "📡 Subscribe via RSS" chip in the /explore header that mirrors the active filter (verified-only when on).
- frontend/src/components/EmbedDialog.vue: "Follow the gallery via RSS" callout with one-click Atom / RSS / Verified-only subscribe buttons — appears once a simulation is public so the operator can join the stream they're now part of.
- backend/tests/test_unit_feed.py: New 566-line test file with 17 offline unit tests — Atom validity, RSS validity, share-link + media enclosures, stance-split summary, truncation, outcome+quality categories, empty-feed handling, self-link query-string preservation, RSS guid stability, dispatcher MIME, verified-only branch, selection helper filter + sort + limit + graceful degradation. CI green on first run (29s).

How it works:
The feed module reuses the exact gallery card builder (_build_gallery_card_payload) the live /api/simulation/public endpoint returns — selection stays single-sourced so a feed and the gallery never drift on what counts as "the newest 20 public simulations." A select_public_cards() helper does the filter + sort + cap + graceful-degradation logic in one place, with the route layer plugging in the real card builder and outcome reader; this keeps the renderer fully unit-testable without booting Flask. The Atom variant uses the standard Clark-notation namespace registration (xmlns="http://www.w3.org/2005/Atom" + media RSS for the previews). The same ±0.2 stance threshold every other surface uses — gallery, share card, replay GIF, transcript, webhook — keeps the "62% bullish" string byte-identical across surfaces.
